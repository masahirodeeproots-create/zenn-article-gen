8体のAIエージェントが分業してKOL向けの投稿指示書を自動生成するシステムを作っています。商品調査、KOLリサーチ、ビジュアル素材の生成、スライド組み立て、レビューまで全自動。手作業でKOL8人分を丸1日かけていた仕事が、今は30分で終わる。15人案件でも1時間弱。

ただ、ここに至るまでに踏んだ地雷の数がすごかった。

この記事で伝えたいのは1つだけ。マルチエージェント設計で一番大事なのは「各エージェントに何をさせるか」じゃない。**「誰が何を知るべきか」** — 情報の境界の設計がすべてを決めます。

## まず全部壊した

最初は1つの大きなエージェントに全工程を任せました。コンテキスト（AIが参照できる情報の範囲）が途切れないからシンプルだろうと。

3人目で壊れた。

LLMはコンテキストが長くなると後半を軽視します。8人分のリサーチと素材生成を1つのコンテキストに詰め込んだら、5人目以降のKOLが1人目の焼き直しになった。

じゃあエージェントを分けよう。調査担当、デザイン担当、組み立て担当、レビュー担当。マイクロサービスと同じ発想で。

ここで最初の地雷を踏みます。**分けた瞬間に、「渡す情報」の設計が必要になる。** 分けること自体は簡単。問題は分けた後。

全エージェントにプロジェクトの全情報を渡したら、エージェントは何が重要か判断できなくなって的外れな出力を連発。逆に必要最小限に絞ったら、キャンペーンの方向性を知らないデザイン担当がブランドと噛み合わない素材を量産。マイクロサービスで全サービスが同じDBを見るアンチパターンと同じ構造です。

落ち着いた設計がこれ。

```
全体方針（全員が知るべきこと）
  ├── ブランドのトーン
  ├── キャンペーンの方向性
  └── 品質基準

個別情報（担当者だけ）
  ├── 特定KOLのリサーチ結果
  ├── プラットフォーム要件
  └── ビジュアル素材
```

まあここまでは教科書的で、誰でも思いつく。問題はこの先。

## 8人の料理人がバラバラのスープを作る

全体と個別を分離したのに、実際に動かすとまだ壊れる。

8人のKOLに個別の素材を作るとして、キャンペーン全体のトーンは統一したい。各KOL担当のエージェントに「キャンペーンの方向性」を渡して個別に作らせたら、バラバラ。あるKOLはポップで、別のKOLはシック。

8人の料理人に「和食を作って」と言って、全員がベースのスープからバラバラに作り始めたようなもの。当然、8種類の違う味になります。ベースのスープを先に作ってから個別のトッピングを変えるべきだった。

```
[素材取得] → [全体スタイル確定] → [個別KOL素材 ×N] → [統合チェック]
                    ↑
            ここが通らないと先に進めない
```

全体スタイルが確定する前に個別制作を始めてはいけない。並列にしていいのは互いに独立したタスクだけ。「全部同時に走らせちゃえ」の誘惑は強いんですけど、AIエージェントだと「全員に全情報渡してるし大丈夫でしょ」という錯覚が生まれやすくて、ここで事故る。依存関係のあるタスクの並列実行は、どの技術領域でも古典的な地雷です。

## 司令塔をマイクロマネージャーにしてしまった

このゲート（関所）を管理するのがオーケストレーター。各エージェントの順序を決めて、前工程の成果物が揃ったかチェックしてから次に進ませる司令塔。

最初、オーケストレーターにリサーチの中身までチェックさせようとしました。ファイルの存在確認だけだとスカスカの成果物を見逃すから。フォロワー分析が500文字以上あるか、商品との相性スコアが含まれているか。

ダメでした。

KOL20人分のリサーチ内容を全部読んで品質判断させたら、オーケストレーター自体のコンテキストが溢れる。元の問題に戻る。組織で考えれば当然のことで、マネージャーが部下の仕事の中身まで口を出し始めたら全体が回らなくなる。

```
オーケストレーターの責務:
  ✅ 成果物の存在チェック（8人分全員あるか？）
  ✅ 前提条件の検証（全体スタイルが確定してるか？）
  ✅ 次工程への遷移判断
  ❌ 内容の品質判断（レビューエージェントの仕事）
  ❌ 工程内の手順（各エージェントの仕事）
```

結局のところ、**「やらないこと」を決める**のが一番大事だった。やることは放っておいても増える。増えすぎるとオーケストレーター自体がモノリスになって壊れる。CIパイプラインと同じで、CIサーバーはテストが通ったかのゲートだけ。テスト自体の責務はテストコードにある。

ゲートで「揃ってない」と判定されたとき、どこまでやり直すか。8人中7人成功して1人失敗なら、その1人だけ再実行。全体の方向性の工程が失敗したら？ 全部やり直し。再実行の粒度は工程の独立性に依存する。情報の境界を引く → 工程の順序が決まる → ゲートの位置が決まる → 再実行の粒度が決まる。全部つながっている。

## 前の案件の亡霊

実運用で一番腹が立ったバグ。

案件を2回流したとき、前の案件で作ったカバー画像がディスクに残っていて、新しい案件のエージェントがそれを「最新の成果物」として使った。ファイルの場所も名前も同じ。中身だけが違う。

人間なら「あ、これ古いやつだ」と気づく。AIエージェントには記憶がない。ファイルがあれば使う。エージェントは悪くない。「どの実行に属する成果物か」の情報がどこにもなかったのが悪い。

各成果物に「実行ID」を埋め込んで、読む側で「今回の実行のものか」をチェックする仕組みにしました。全成果物に漏れなく刻印するのは大変なので、ゲートの検査に「スコープ一致チェック」を追加。刻印がなければ止める。通過させるのではなく、**止める方向に倒す**。フェイルセーフ。

この問題は「誰が何を知るべきか」の裏返しで、**「何を知ってはいけないか」**の設計です。エージェントは前の案件の成果物を知ってはいけない。でも放っておくとファイルが残ってるから知ってしまう。明示的に遮断する仕組みが要る。

## 同じ問題が入れ子で再発する

3人のKOLで動いてた設計を15人案件に流したらタイムアウト。規模に応じてモードを切り替えるしかない。少数モードはオーケストレーターが処理も兼務、多数モードは管理に専念して全部委任。どちらのモードでも成果物の形式は統一。後工程はモードの違いを知らない。

で、モード切替も入れて安心してたら、素材生成工程で嫌なことに気づきました。

工程内部の処理が重すぎる。全部1つのエージェントにやらせたら、失敗箇所の特定ができないし、後半の品質が落ちる。

嘘でしょ。

大きなパイプラインの中に小さなパイプラインがある。どっちの層にも同じ設計原則 — 情報の境界、ゲート、スコープの刻印 — が必要。正直うんざりしました。でも同時に「同じ解法が使えるのか」という安心感もあった。パイプライン全体で学んだことが、そのまま各工程の内部設計に転用できる。

:::details レビューループについて
PPTXの最終レビューも設計が必要。レビューエージェントが「データの問題」「ビジュアルの問題」とラベルを付け、オーケストレーターがラベルに従ってルーティング。ループ上限10回、超えたら人間にエスカレーション。ビルド→レビューの閉ループは1つのチームとして囲い込み、内部は自律的に回す設計。
:::

:::details 全体の設計原則
```
┌─ 情報の境界 ──────── 誰が何を知るか
├─ 全体→個別の順序 ── グローバルを先に確定
├─ 成果物ゲート ────── 完全性チェックで遷移判断
├─ スコープの刻印 ─── どの実行に属するか
├─ 品質の閉ループ ─── レビュー→修正→再検証
├─ 責務の絞り込み ─── オーケストレーターは管理のみ
└─ モード切替 ──────── 規模に応じた実行戦略
```
:::

## 暗黙知を設計に落とす

マルチエージェントの設計は「エージェントの能力」じゃなくて「エージェント間の関係性」の設計でした。何ができるかより、情報をどう渡し、どこでチェックし、どう修正サイクルを回すか。

人間のチームでは暗黙知として機能していること — 境界、品質チェック、委譲 — を全部形式化する必要がある。「あうんの呼吸」はAIに通じない。ただ、逆に設計が全部見えるのは面白い。人間のチームだと「なんとなくうまく回ってる」の内実を、データ構造とプロトコルに落とし込む作業。正直、組織設計の教科書を読むよりAIエージェント8体を協調させるほうが組織論への理解が深まると思っています。半分本気で。

50人案件でスケールするか。まだ確信はない。ただ今のところ8人案件で30分、15人で1時間弱。手作業の10分の1以下。

まだ山ほど壊れるだろうけど、壊れ方のパターンは見えてきました。
