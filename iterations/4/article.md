3人目のKOLのキャプション方向性が、1人目のコピーになっていました。5人目のビジュアル指示は2人目とほぼ同じ。

8人のKOLにカスタマイズした投稿指示書をPPTXで自動生成するシステムを作っていて、最初は1つの大きなエージェントに全部任せていました。コンテキストウィンドウに全情報を詰め込めば受け渡しの設計がいらない。合理的だったはずが、LLMは長いコンテキストの中で情報を平均化していくんですよね。後半のKOLが前半の繰り返しになる。

エージェント分ければ解決するだろう、と。調査、デザイン、ビルド、レビュー。

**分けた瞬間に本当の問題が始まりました。**

エージェントの分割は難しくないんです。難しいのは分けた後の「境界」の設計で、しかもこの境界が複数の方向に分岐して、最終的に全部同じ仕組みに合流していく。この記事はその合流の話です。

## 全体のトーンを先に決めろ

8人のKOL担当エージェントにそれぞれ「キャンペーンの方向性」をテキストで渡して、個別にビジュアル素材を作らせました。

結果。あるKOLはポップ。別のKOLはシック。同じキャンペーンに見えない。

ラーメンで考えてみてください。ベースのスープを先に作ってからトッピングを変えるのが正解で、8人に「豚骨っぽい味で」って言葉だけ渡して各自にスープから作らせたら、そりゃ8種類の別のラーメンができますよね。

```
[素材取得] → [全体スタイル確定] → [個別KOL制作 ×N] → [統合チェック]
```

全体スタイルが確定する前に個別制作を始めてはいけない。8人のKOLは互いに独立しているから並列化したくなります。ただ「全体スタイル → 個別KOL」の依存は直列で守る必要がある。ここをサボって全部同時に走らせたら不整合が出ました。

順序の制約を守るために工程間にゲートを置きます。前の工程の出力が基準を満たしているか確認してから次に進む。

ゲートで何を検査するか。最初は中身の品質を見ようとしたんですけど、無理でした。調査エージェントの出力が正しいかどうかなんて次の工程には判断できない。ゲートが検査すべきは**成果物の存在と完全性**です。8人分のリサーチ結果が全部あるか。必須項目が埋まっているか。3人分しかないのに次に進もうとしていないか。

オーケストレーターはマネージャーと同じです。中身は判定しない。「必要なものが揃ったか」だけ見る。

ここまでは素直な設計。DDDの境界づけられたコンテキストをエージェントに適用して、工程間にゲートを置く。

問題は次に起きました。

## 前の案件のカバー画像が新しい案件に混入した

案件を2回流したら、前の案件の成果物が残っていて、新しい案件のエージェントがそれを「最新の成果物」として使いました。

前の案件で作ったカバー画像を、新しい案件でそのまま採用。ファイルの場所は同じ、名前も同じ、中身だけが古い。

人間なら「あ、これ前のやつだ」とわかります。エージェントにはわからない。

最初の対策は「前の案件のファイルを消してから走らせる」。消し忘れました。2回目も消し忘れました。3回目は消したのに一部の中間成果物が残っていてまた混入。

もう嫌になりました。

ここで笑えなかったのは、この問題がさっき作った「時間の境界」のゲートをすり抜けるからです。ゲートは「成果物が存在するか」をチェックする。前の案件の成果物は存在する。名前も合っている。ゲートは素通りさせてしまう。

**存在だけでは足りない。「どの実行に属するか」の情報が必要です。**

データベースにMVCCという仕組みがあります。各行がどのトランザクションで書き込まれたかを持っていて、読み取り時にそのバージョンをチェックする。古いトランザクションの行は見えなくなる。この発想で、各成果物に実行IDを刻印しました。

ゲートの検査項目が2つに統合されます:

1. **成果物が存在するか**（順序の境界のチェック）
2. **成果物のスコープが現在の実行と一致するか**（所有の境界のチェック）

ここが合流点です。「工程の順序を守る」問題と「古い成果物が混入する」問題を、最初は全然違う課題だと思っていました。ところが解決策のレイヤーでは同じゲートの検査項目として統合される。別々に悩んでいた時間がもったいなかった。

「全部の成果物にスコープを付けるなんて、忘れたら終わりでしょ」。

はい、だからゲートをフェイルセーフにしています。スコープ情報がない成果物はゲートが止める。通過させるんじゃなくて止める方向に倒す。「忘れた」が「事故になる」のではなく「止まる」になる設計です。

:::details 余談：なぜ「全部消す」じゃダメだったか
「毎回全ファイル消してクリーンにすればいいのでは」は最初に試しました。問題は中間成果物のキャッシュ。素材取得のステップで大きなファイルをダウンロードするので、毎回ゼロからやると遅すぎる。かといって選択的にキャッシュを残すと「どれが安全にキャッシュできるか」の判定が必要になって、結局スコープ管理と同じ問題に帰着しました。
:::

## 大きすぎる工程の中にも同じ構造が出てくる

素材制作の工程が重すぎました。素材を集めて、全体トーンを決めて、KOL個別の素材を作って、全部揃ってるか確認する。これを1つのエージェントに全部やらせたら、途中で失敗してどこで壊れたかわからない。

最初からやり直し。

...あれ。これ、記事の冒頭で「1人に全部任せたら品質が落ちた」と言ったのとまったく同じ構造じゃないですか。粒度が違うだけ。

```
パイプライン全体:
  [調査] →ゲート→ [計画] →ゲート→ [制作] →ゲート→ [レビュー]

「制作」の内部:
  [素材取得] →ゲート→ [全体トーン] →ゲート→ [KOL個別×N] →ゲート→ [統合]
```

大きなパイプラインの中に小さなパイプラインがある。情報の境界もゲートもスコープの刻印も全部要る。設計原則が再帰的に適用される。

これに気づいたとき、ちょっと拍子抜けしました。もっと新しいルールが必要だと身構えていたのに、同じ原則の適用先が増えるだけだった。設計がスケールするというのはこういう意味なんでしょうね。

ただし、ゲートの検査密度に最適解はまだ見えていません。増やすと安全だけど遅い。減らすと速いけど事故る。今のところ全部入れていますけど、過剰なのかもしれません。

## レビューループとスケール

残り2つ、手短に。

PPTXのレビューは修正→再チェックのループです。データの問題ならビルド担当へ、ビジュアルの問題なら見た目担当へ差し戻す。レビューエージェントが分類ラベルを付けて、オーケストレーターはラベルでルーティングするだけ。中身は見ない。ビルドからレビューまでは1つのチームに囲い込んで裁量を渡し、上限を超えたら人間にエスカレーション。

KOLが3人のときと30人のとき、同じ設計は使えません。3人ならオーケストレーターが全部見られます。30人だとコンテキストが溢れる。入力の規模でモードを切り替えて、少数なら直接処理、多数なら管理に専念。成果物のインターフェースはどちらでも同じにする。

正直に言うと、少数モードはアーキテクチャとして美しくないです。3人のKOLのために大仰な委任構造を組むのはオーバーエンジニアリングだと判断して簡易版にしたんですけど、30人対応のときに結局リファクタが必要になりました。最初から多数モードで統一すべきだったのか。いまだにわかりません。

## 暗黙知をプロトコルにする

```
┌─ 情報の境界 ──── 誰が何を知るか
├─ 時間の境界 ──── 全体→個別の順序 ─┐
├─ 所有の境界 ──── 成果物のスコープ ─┤→ ゲートで合流
├─ 品質ループ ──── レビュー→修正     │
└─ モード切替 ──── 規模に応じた戦略  ┘
```

マルチエージェント設計で一番面白かったのは、別々の問題に見えるものが同じ仕組みに収束していく過程でした。「順序を守る」と「古い成果物を排除する」は全然違う問題に見えて、ゲートの検査項目として合流する。

人間のチームだとこの手の調整は暗黙的に処理されています。「これ前のやつだよね」と誰かが気づく。「先に全体の方向決めようよ」と誰かが言う。AIエージェントではそれを全部、検査可能なプロトコルとして書き下す必要がある。

読者の方がマルチエージェントシステムを作るとき、エージェントを分けた途端に「受け渡し」の問題にぶつかると思います。そのとき、問題の種類を分けるより解決の仕組みがどこで合流するかに注目すると、設計がシンプルになるかもしれません。少なくとも筆者の場合はそうでした。
