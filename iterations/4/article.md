3人目のKOLのキャプション方向性が、1人目のコピーになっていました。

5人目のビジュアル指示は2人目とほぼ同じ。8人目はもう1人目の使い回し。8人のKOLにカスタマイズした投稿指示書をPPTXで自動生成するシステムを作っていて、最初は1つの大きなエージェントに全情報を詰め込んでいたんですけど、LLMは長いコンテキストの中で情報をぼんやり平均化していきます。後半のKOLが前半の繰り返しになる。

エージェント分ければ解決するだろう。調査、デザイン、ビルド、レビュー。

分けた瞬間に本当の問題が始まりました。この記事はその「本当の問題」の話です。

## ベースのスープを先に作れ

8人のKOL担当エージェントにそれぞれ「キャンペーンの方向性」をテキストで渡して、個別にビジュアル素材を作らせました。

あるKOLはポップ。別のKOLはシック。もう1人は謎にレトロ。

同じキャンペーンに見えない。

ラーメンで考えるとわかりやすくて、ベースのスープを先に作ってからトッピングを変えるべきところを、8人の料理人に「豚骨っぽい味で」って言葉だけ渡して各自にスープから作らせたんです。8種類の違うラーメンができました。当然ですよね。

```
❌ やったこと:
  [方向性テキスト] → [KOL担当A] → ポップな素材
                   → [KOL担当B] → シックな素材
                   → [KOL担当C] → なぜかレトロ

✅ こうすべきだった:
  [素材取得] → [全体スタイル確定] → [KOL個別 ×N] → [統合チェック]
```

全体スタイルが確定する**前に**個別制作を始めちゃダメ。

ここで厄介だったのが並列化の誘惑です。8人のKOLは互いに独立しているから全員分を同時に走らせたい。速いし。実際KOL間の並列化は正しいんです。でも「全体スタイル確定 → 個別KOL制作」の上から下への依存は直列で守らないといけない。

「まあ大丈夫でしょ」で全部同時に走らせました。壊れました。知っていて踏んだ。うんざりしました。

この経験から工程間に**ゲート**を置くことにしました。前の工程の出力が条件を満たしているか確認してから後工程に進む。「全体スタイルの成果物が存在する」ことが個別制作開始の前提条件です。

ゲートで何を見るかも迷いました。最初は出力の中身の品質を検査しようとしたんですけど、調査エージェントが「このKOLは20代女性に人気」って出してきて、それが正しいかどうかなんて次の工程にはわからない。

結局ゲートは中身を見ないようにしました。見るのは**存在と完全性**だけ。8人分のリサーチ結果が全部あるか。必須項目が埋まっているか。中身の正しさはレビューエージェントに任せます。

ここまでは教科書的な話。DDDの境界づけられたコンテキストをエージェントに適用して、工程間にゲートを置く。

問題は次に起きました。

## 前の案件のカバー画像がそのまま出てきた

これが一番キツかった話です。

案件を2回流したとき、前の案件の成果物が残っていて、新しい案件のエージェントがそれを「最新の成果物」として使いました。前の案件で作ったカバー画像が新しい案件のPPTXにそのまま載っている。

納品直前に気づいて肝が冷えました。

ファイルの場所は同じ。名前も同じ。中身だけが古い。人間なら「あ、これ前のだ」って気づきますけど、エージェントにはその判断ができません。

最初の対策。「前の案件のファイルを消してから走らせる」。

消し忘れました。

2回目。また消し忘れました。3回目は消したのに一部の中間成果物が残っていてまた混入。もう途方に暮れました。人間の注意力に依存する設計は3回で破綻するんだなということを、3回破綻して学びました。冗談みたいな話ですけど本当の話です。

笑えなかったのは、この問題が前のセクションで作ったゲートを**すり抜ける**からです。ゲートは「成果物が存在するか」をチェックする。前の案件の成果物は存在する。名前も正しい。ゲートは通してしまう。

存在チェックだけじゃ足りない。「この成果物がどの実行に属するか」の情報が必要なんです。

各成果物に実行IDを刻印することにしました。データベースにMVCCという仕組みがあって、各行がどのトランザクションで書き込まれたかを持っていて読み取り時にチェックするんですけど、同じ発想です。ゲートの検査項目が2つになります。

1. 成果物が存在するか
2. 成果物の実行IDが今回の実行と一致するか

2番目がないと前の案件の亡霊がゲートをすり抜けます。

ここで面白かったのが、最初は「工程の順序を守る問題」と「古い成果物の混入問題」を全然別の課題だと思っていたことです。でも解決策のレイヤーでは同じゲートの検査項目に合流する。「存在する？」と「今回のもの？」。順序の問題も所有の問題もこの2つで解ける。別々に悩んでいた時間を返してほしい。

「全成果物にIDを付けるなんて忘れたら？」。もっともな懸念で、だからゲートをフェイルセーフにしています。実行IDがない成果物はゲートが止める。通過させるんじゃなくて止める。忘れた側が罰せられる設計です。

:::details なぜ「毎回全消し」ではダメだったか
「毎回全ファイル消してクリーンにすれば」は最初に試しました。素材取得ステップで数百MBのファイルをダウンロードするので毎回30分以上かかるんですよね。キャッシュを残したい。でも選択的にキャッシュを残すと「どれがキャッシュ安全か」の判定が必要になって、結局スコープ管理と同じ問題に帰着します。
:::

## 同じ構造が入れ子で出てくる

素材制作の工程が重すぎて1つのエージェントに全部やらせたら、途中で失敗してどこで壊れたかわからない。最初からやり直し。

待ってください。これ記事の冒頭で「1人に全部任せたら品質が落ちた」って言ったのと同じ構造じゃないですか。粒度が違うだけ。

```
パイプライン全体:
  [調査] →ゲート→ [計画] →ゲート→ [制作] →ゲート→ [レビュー]

「制作」の内部にも同じ構造:
  [素材取得] →ゲート→ [全体トーン] →ゲート→ [KOL個別×N] →ゲート→ [統合]
```

大きなパイプラインの中に小さなパイプラインがある。情報の境界もゲートもスコープの刻印も全部再帰的に必要になる。

正直、拍子抜けしました。もっと新しいルールが必要だと身構えていたのに、同じ原則の適用先が増えるだけだった。新しいことを覚えなくていいのは嬉しいんですけど、「じゃあ最初からこの原則で全部設計できたのでは」と思うと、遠回りした感覚もあります。

ゲートの密度に最適解はまだ見えていません。全サブステップにゲートを入れると安全ですけど遅い。省略して2回事故りました。今は全部入れています。過剰かもしれない。でも事故るよりマシです。

:::details レビューループとモード切替

残り2つの設計判断を手短に。

**レビューの閉ループ**: PPTXのレビューは修正→再チェックのループです。データの問題ならビルド担当へ、ビジュアルの問題なら見た目担当へ差し戻します。レビューエージェントが分類ラベルを付けて、オーケストレーターはラベルでルーティングするだけ。ビルドからレビューまでを1チームに囲い込んで裁量を渡し、上限超えで人間にエスカレーション。

**モード切替**: KOL3人と30人で同じ設計は使えません。少数ならオーケストレーターが処理の一部を直接担当、多数なら管理に専念してサマリーだけ読む。成果物のインターフェースは統一。少数モードはアーキテクチャとして美しくないです。最初から多数モードで統一しておけばよかったのかもしれませんが、3人のKOLのために大仰な委任構造を組むのはやりすぎに感じて。この判断が正しかったかは今でもわかりません。
:::

## まだわかっていないこと

マルチエージェント設計で一番面白かったのは、別々の問題に見えるものが同じ仕組みに収束していく過程でした。「順序を守る」と「古い成果物を排除する」は全然違う問題に見えて、ゲートの検査項目として合流する。

```
┌─ 情報の境界 ──── 誰が何を知るか
├─ 時間の境界 ──── 全体→個別の順序 ─┐
├─ 所有の境界 ──── 成果物のスコープ ─┤→ ゲートで合流
├─ 品質ループ ──── レビュー→修正     │
└─ モード切替 ──── 規模に応じた戦略  ┘
```

人間のチームだとこの手の調整は暗黙的に処理されています。「これ前のやつだよね」と誰かが気づく。「先に全体決めよう」と誰かが言う。AIエージェントではそれを全部プロトコルとして書き下す必要がある。面倒ですけど、やってみると人間のチームが無意識にやっていた調整の複雑さに驚きます。

ゲートの検査密度はまだ手探りのままです。全部入れると安全だけど遅い、減らすと速いけど事故る。工程が20、30と増えたときにオーケストレーター自体がボトルネックにならないかも気になっています。人間のチームなら「もうチームリーダー置いて任せよう」と階層を足しますけど、エージェントで同じことをやったときに品質が維持できるのかは、正直まだ答えが出ていません。
