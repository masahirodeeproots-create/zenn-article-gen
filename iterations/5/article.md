3人目のKOLのキャプション方向性が、1人目のコピーになっていました。5人目のビジュアル指示は2人目とほぼ同じ。8人分のKOL向け投稿指示書をPPTXで自動生成するシステムを作っていて、最初は1つの大きなエージェントに全工程を任せていました。コンテキストウィンドウに全情報を詰め込めば受け渡しの設計がいらない。合理的だったはずが、LLMは長いコンテキストの中で情報を平均化していくんですよね。後半のKOLが前半の繰り返しになる。

エージェントを分ければ解決するだろう、と。調査、デザイン、ビルド、レビュー。

**分けた瞬間に本当の問題が始まりました。**

エージェントを分割すること自体は難しくないんです。難しいのは分けた後の「境界」の設計。しかもこの境界が複数の方向に分岐して、最終的に全部同じ仕組みに合流していく。この記事はその合流の話です。

## 全体のトーンを先に決めろ

8人のKOL担当エージェントにそれぞれ「キャンペーンの方向性」をテキストで渡して、個別にビジュアル素材を作らせました。

結果。あるKOLはポップ。別のKOLはシック。同じキャンペーンに見えない。

ラーメンで考えてみてください。ベースのスープを先に作ってからトッピングを変えるのが正解で、8人に「豚骨っぽい味で」って言葉だけ渡して各自にスープから作らせたら、8種類の別のラーメンができる。当然です。

```
[素材取得] → [全体スタイル確定] → [個別KOL制作 ×N] → [統合チェック]
```

全体スタイルが確定する前に個別制作を始めてはいけない。8人のKOLは互いに独立しているから並列化したくなります。ただ「全体スタイル → 個別KOL」の依存は直列で守る必要がある。ここをサボって全部同時に走らせたら不整合が出ました。

順序の制約を守るために工程間にゲートを置きます。前の工程の出力が基準を満たしているか確認してから次に進む。

ゲートで何を検査するか。最初は中身の品質を見ようとしたんですけど、無理でした。調査エージェントの出力が正しいかどうかなんて次の工程には判断できない。ゲートが検査すべきは**成果物の存在と完全性**。8人分のリサーチ結果が全部あるか。必須項目が埋まっているか。3人分しかないのに次に進もうとしていないか。

オーケストレーターはマネージャーと同じです。中身は判定しない。「必要なものが揃ったか」だけ見る。

ここまではDDDの境界づけられたコンテキストをエージェントに適用して、工程間にゲートを置く、という素直な設計です。

問題は次に起きました。

## 前の案件のカバー画像が新しい案件に混入した

案件を2回流したんですよ。そしたら前の案件の成果物が残っていて、新しい案件のエージェントがそれを「最新の成果物」として使いました。

前の案件のカバー画像を、新しい案件でそのまま採用。ファイルの場所は同じ、名前も同じ、中身だけが古い。

人間なら気づきます。エージェントは気づかない。

最初の対策。「前の案件のファイルを消してから走らせる」。消し忘れました。2回目も消し忘れました。3回目は消したのに一部の中間成果物が残っていてまた混入。

もう嫌になりました。

ただ、ここで笑えなかったのはもっと根深い理由があって、この問題がさっき作った「順序の境界」のゲートをすり抜けるんです。ゲートは「成果物が存在するか」をチェックする。前の案件の成果物は存在する。名前も合ってる。ゲートは素通りさせてしまう。

**存在だけでは足りない。「どの実行に属するか」の情報が必要です。**

データベースにMVCCという仕組みがあります。各行がどのトランザクションで書き込まれたかを持っていて、読み取り時にバージョンをチェックする。古いトランザクションの行は見えなくなる。この発想を成果物に適用して、各成果物に実行IDを刻印しました。

ゲートの検査項目が2つに統合されます:

1. **成果物が存在するか**（順序の境界）
2. **成果物のスコープが現在の実行と一致するか**（所有の境界）

ここが合流点です。

「工程の順序を守る」問題と「古い成果物を排除する」問題は、見た目がまったく違います。片方はワークフローの話で、もう片方はデータの鮮度の話。ところが解決策のレイヤーでは同じゲートの検査項目として統合される。別々に途方に暮れていた2週間は何だったのかと思いました。

「全部の成果物にスコープを付けるなんて忘れたら終わりでしょ」。はい。だからゲートをフェイルセーフにしています。スコープ情報がない成果物はゲートが止める。通過させるんじゃなくて止める方向に倒す。「忘れた」が「事故」になるんじゃなくて「停止」になる。人間の注意力に依存する設計は3回で破綻するという教訓は、ここで回収されます。

:::details 余談：なぜ「全部消す」じゃダメだったか
「毎回全ファイル消してクリーンにすればいいのでは」は最初に試しました。素材取得のステップで大きなファイルをダウンロードするので毎回ゼロからやると遅すぎる。選択的にキャッシュを残すと「どれが安全にキャッシュできるか」の判定が必要になって、結局スコープ管理と同じ問題に帰着しました。
:::

## 粒度が変わっても構造は同じ

素材制作の工程が重すぎました。素材を集めて、全体トーンを決めて、KOL個別の素材を作って、全部揃ってるか確認する。1つのエージェントに全部やらせたら途中で失敗して、どこで壊れたかわからない。

最初からやり直し。

...これ、冒頭で「1人に全部任せたら品質が落ちた」と言ったのとまったく同じ話ですよね。粒度が違うだけです。

```
パイプライン全体:
  [調査] →ゲート→ [計画] →ゲート→ [制作] →ゲート→ [レビュー]

「制作」の内部:
  [素材取得] →ゲート→ [全体トーン] →ゲート→ [KOL個別×N] →ゲート→ [統合]
```

大きなパイプラインの中に小さなパイプラインがある。情報の境界もゲートもスコープの刻印も全部要る。設計原則が再帰的に適用される。

拍子抜けしました。もっと新しい設計ルールが必要だと身構えていたのに、同じ原則の適用先が増えるだけだった。設計がスケールするというのはこういうことなんでしょうね。

ゲートの検査密度に最適解はまだ見えていません。増やすと安全だけど遅い。減らすと速いけど事故る。今のところ全部入れていますけど、過剰かもしれません。

## レビューループとスケール

残り2つ、短く。

PPTXのレビューは修正→再チェックのループです。データの問題ならビルド担当へ、ビジュアルの問題なら見た目担当へ差し戻す。レビューエージェントが分類ラベルを付け、オーケストレーターはラベルでルーティング。中身は見ない。ビルドからレビューまでを1つのチームに囲い込み、上限を超えたら人間にエスカレーション。

KOL3人と30人で同じ設計は使えません。少数ならオーケストレーターが直接処理、多数なら管理に専念。成果物のインターフェースはどちらでも統一。正直、少数モードは美しくないです。3人のために大仰な委任構造を組むのはやりすぎだと判断して簡易版にしたんですけど、30人対応のときに結局リファクタが必要になりました。最初から多数モードで統一すべきだったのか。いまだにわかりません。

## 暗黙知をプロトコルにする

```
┌─ 情報の境界 ──── 誰が何を知るか
├─ 時間の境界 ──── 全体→個別の順序 ─┐
├─ 所有の境界 ──── 成果物のスコープ ─┤→ ゲートで合流
├─ 品質ループ ──── レビュー→修正     │
└─ モード切替 ──── 規模に応じた戦略  ┘
```

マルチエージェント設計で一番面白かったのは、別々の問題に見えるものが同じ仕組みに収束していく過程でした。「順序を守る」と「古い成果物を排除する」は全然違う問題に見えて、ゲートの検査項目として合流する。

人間のチームだとこういうことは暗黙的に処理されています。「これ前のやつだよね」と気づく人がいる。「先に方向決めようよ」と言う人がいる。AIエージェントではそれを全部、検査可能なプロトコルとして書き下す必要がある。

エージェントを分けた途端に「受け渡し」の問題にぶつかったとき、問題の種類を分類するより、解決の仕組みがどこで合流するかに注目すると設計がシンプルになるかもしれません。少なくとも筆者の場合はそうでした。
