## Changes to style_guide.md

### 1. Rule 5 強化: テーゼ繰り返し禁止（🔴 CRITICAL）

**背景:** テーゼ「コンテキスト管理の話」が本文中に6回以上登場。人間のベンチマーク記事（`immutable-immer.md` 等）はテーゼを1回述べて後は実証で語る。

**変更内容:**
- Rule 5 に「テーゼは最大2回（冒頭1回、結論1回）」の制約を追記
- 本文中での繰り返し例を ❌ として追加
- チェック手順を「テーゼを述べている文が3箇所以上なら2箇所削除」と具体化

### 2. Rule 9 新設: 観察可能な失敗の具体例（🔴 CRITICAL）

**背景:** 「品質が落ちた」「指示が粗くなった」という問題記述が抽象的で、読者が症状を具体的にイメージできない。`immutable-immer.md` のベンチマーク数値テーブルのような「見える証拠」が欠けていた。

**変更内容:**
- 新ルール追加: 問題セクションごとに実際の出力失敗を1文で示すことを必須化
- ❌ 例: 「品質が粗くなる」（抽象的）
- ✅ 例: 「3人目のKOLのキャプション方向性が1人目とほぼ同一になった（"潤い"と"自然由来"を繰り返すだけになった）」

### 3. Rule 7 強化: 未実装コードの明示義務（🔴 CRITICAL）

**背景:** `assert_state_plan_invariant` 関数が「まだ実装できていません」と本文で述べながら、コードブロックの体裁は動作コードと同一だった。読者を誤解させるリスクがある。

**変更内容:**
- 未実装コードには `# 未実装 — 設計案` コメントを必須化
- ❌/✅ 例を追加

### 4. Rule 16 新設: クロージングは1文で着地（🟡 IMPORTANT、旧Rule 15 まとめに統合）

**背景:** クロージング段落が全問題の再列挙 + 哲学的結論 + 不確実性表明の3役を担い、散漫になっていた。`react-use-rfc.md` ベンチマークは単一の評価文で終わる。

**変更内容:**
- Rule 15「まとめ」を Rule 16「まとめ・クロージング」に改名・拡張
- 「クロージングは1つの鋭い文で終わる」ルールを追加
- ❌: 全体要約段落 / ✅: 「次に何かが壊れるとしたら、まだ境界を引いていないどこかです。」

### 5. Rule 17 強化: 「少し脱線します」シグナル禁止（🟢 POLISH、旧 Rule 16）

**背景:** 「少し脱線します」という見出しは著者自身が構造的欠陥を認めているシグナル。人間の書き手は脱線をシームレスに織り込むか、`:::details` に移すか、カットする。

**変更内容:**
- 「脱線シグナルを本文に残してはならない」ルールを追加
- ❌: 「少し脱線します。〜」/ ✅: 主流に統合またはブロック移動

### 6. ANTI-PATTERNS テーブルに5項目追加

新規追加パターン:
| パターン | 理由 |
|---|---|
| テーゼの3回以上の繰り返し | 読者への不信 |
| 問題の実際の出力を示さない | 診断の説得力がゼロ |
| 「少し脱線します」シグナル | 構造欠陥の自己申告 |
| クロージングが全体要約段落 | 要約は読者の仕事 |
| 未実装コードを動作コードと同じ体裁で掲載 | 読者を誤解させる |

### 追加変更1: Rule 10a 強化（Honest Failure Trace）

**背景**: Iteration 2記事でvisualizerのフォント色問題が「ルールを追加したら直った」の1文で終わっていた。迷走の記録が抜けると「解決策カタログ」になるという問題。

**変更**: 「ルールを入れたら直った」という1文解決パターンを❌例として追加し、チェックガイダンスを強化。

### 追加変更2: Rule 11 強化（ボールドカウント手順）

**背景**: Iteration 2記事のボールドが8〜9箇所（上限5〜6）。生成後カウントする手順が明示されていなかった。

**変更**: 「`**`の開始タグをカウントして7個以上なら削減」という具体的手順を追加。

### 追加変更3: Rule 10 強化（Fix Loop・サブシステムセクションの接続）

**背景**: Iteration 2記事でFix Loopセクション（Phase 5-6-7）が前のセクション（stale artifact問題）と接続していない孤立セクションになっていた。

**変更**: 「Phase NからPhase N+1への接続を飛ばさない」という具体的ガイダンスを追記。
