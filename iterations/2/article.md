3人目のKOLのキャプション方向性が、1人目のコピーになっていました。

8人のKOLにそれぞれカスタマイズした投稿指示書をPPTXで自動生成するシステムを作っていて、最初のアプローチは「1つの大きなエージェントに全部任せる」でした。コンテキストが途切れないし受け渡しの設計もいらない。で、出力を見たら後半のKOLが前半の繰り返しになっている。LLMは長いコンテキストの中で情報がぼんやり平均化されていくんですよね。5人目のビジュアル指示が2人目と酷似していて、さすがにこれは使えないなと。

「じゃあエージェント分ければいいでしょ」と思って分けました。調査、デザイン、ビルド、レビュー。マイクロサービスの発想です。

分けた瞬間に地獄が始まりました。

## 境界の話が全部つながっている

エージェントを分割すること自体は簡単です。マイクロサービスでも同じで、サービスを切り出すのは難しくない。難しいのはサービス間のインターフェース。DDDで言う「境界づけられたコンテキスト」をどこに引くか。

で、エージェントでもまったく同じ問題が起きるんですけど、面白いのは「境界」という言葉の裏に複数の設計課題が全部くっついてくることでした。ここが人間のチーム設計との最大の違いだと思っていて、この記事ではそこを掘ります。

人間のチームだと「誰が何を知るべきか」って暗黙的に機能しています。「あ、これ前の案件のファイルだ」とか「先に全体の方向性決めてからやろう」とか、無意識にやっている。AIエージェントではこの暗黙知を全部、データ構造とプロトコルとして明示的に設計しないといけない。

具体的には3つの境界を設計することになりました。

1. **情報の境界** — 誰が何を知るか
2. **時間の境界** — どの順序で確定させるか
3. **所有の境界** — この成果物はどの実行に属するか

見た目は別々の問題なんですけど、全部つながっています。1つ欠けると他も壊れる。以下、それぞれがどう壊れたかの話です。

## 情報の境界 — 全員に全部渡すとどうなるか

最初に試したのが、全エージェントにプロジェクト全体の情報を渡すアプローチです。DDDでいうと「全サービスが同じDBを見る」状態で、まあアンチパターンですよね。

情報が多すぎるとエージェントは何が重要かを判断できなくなります。逆に情報を絞りすぎると文脈を持てなくて的外れなアウトプットを出す。

結論としては「全体方針」と「個別情報」を分離する方針に落ち着きました。ブランドのトーン、キャンペーンの方向性、品質基準は全員が知る。特定KOLのリサーチ結果やプラットフォーム要件は担当者だけが知ればいい。

ここまでは教科書通りです。2文で終わる話。

## 時間の境界 — ベースのスープを先に作れ

本当に痛い目を見たのはここからで、これが記事のメインです。

8人のKOL担当エージェントにそれぞれ「キャンペーンの方向性」をテキストで渡して、個別にビジュアル素材を作らせました。結果、あるKOLはポップで別のKOLはシック。まるで別のキャンペーン。

ラーメンに例えると、ベースのスープを先に作ってからトッピングを変えるべきところを、「豚骨っぽいやつで」って言葉だけ渡して8人に各自スープから作らせた。そりゃバラバラになりますよ。

```
[素材取得] → [全体スタイル確定] → [個別KOL制作 ×N] → [統合チェック]
```

全体スタイルが確定する前に個別制作を始めちゃダメなんです。で、この「ダメ」が本当に厄介なのは、並列処理の誘惑があるから。

8人のKOLは互いに独立しているので、直感的には全員分を同時に走らせたくなります。実際、KOL間の並列化は正しい。でも「全体スタイル → 個別KOL」の依存関係は直列で守らないといけない。何が並列可能で何が直列必須かを明示的に宣言していなくて、「まあ大丈夫でしょ」で全部同時に走らせたら不整合が出ました。依存関係のあるタスクの並列実行は分散システムの古典的なバグですけど、まさかエージェントシステムで踏むとは思わなかった。

んで、この「全体を先に確定させてから個別に入る」という順序制約を守るために、工程間に**ゲート**を置くことにしました。前の工程の出力が基準を満たしているか確認してから次に進む。製造業の工程間検査と同じ。

ゲートで何を検査するかもけっこう悩んで。最初は中身の品質を見ようとしたんですけど、調査エージェントが「このKOLは20代女性に人気」って出してきたとして、正しいかどうかなんて次のエージェントにはわかりません。

組織で考えたら整理できました。マネージャーの仕事は部下の仕事の中身を判定することじゃなくて、「必要なものが全部揃ったか」「次に進めていいか」の判断です。オーケストレーターもそうあるべきで、成果物の存在と完全性をチェックする。8人分のリサーチ結果が全部あるか。必須項目が埋まっているか。3人分しかないのに次に進もうとしていないか。CIパイプラインのゲートサーバーと同じ構造ですね。

で、このゲートは最初「パイプライン全体の工程間」にだけ置いてたんですけど、素材制作工程が重すぎて1エージェントに任せたら途中で失敗してどこで壊れたか切り分けられない。最初からやり直し。えっ、これ最初に「1人に全部任せたら品質落ちた」のとまったく同じ問題じゃないですか。粒度が違うだけで構造が同じ。

```
パイプライン全体:
  [調査] →ゲート→ [計画] →ゲート→ [制作] →ゲート→ [レビュー]

「制作」の内部:
  [素材取得] →ゲート→ [全体トーン] →ゲート→ [KOL個別×N] →ゲート→ [統合]
```

大きなパイプラインの中に小さなパイプラインがある。設計原則が再帰的に適用される。フラクタルです。同じ原則の適用先が増えるだけで新しいルールが増えるわけじゃないんですよね。これは設計していて気持ちよかった部分で、「スケールする設計原則」ってこういうことかと。

ただし、ゲートの検査密度には最適解がまだ見えていません。ゲートを増やすと安全だけど遅くなる。減らすと速いけど事故る。「制作」内部のサブステップにもゲートを全部入れるべきかどうか、規模によって変えるべきなんだろうと思いつつ、今のところ全部入れています。過剰設計かもしれません。

## 所有の境界 — 一番キレた設計ミス

これが一番腹立った話です。笑い話にできるまで3案件かかりました。

案件を2回流したとき、前の案件の成果物が残っていて、次の案件のエージェントがそれを「最新の成果物」として使った。前の案件で作ったカバー画像が新しい案件にそのまま混入。ファイルの場所も名前も同じで中身だけが古い。

人間なら「あ、これ前のやつだ」ってわかる。エージェントにはわからない。

最初の対策は「前の案件のファイルを消してから走らせる」。消し忘れで2回目もやらかしました。人間の注意力に依存する設計は必ず破綻しますよね。当たり前のことなんですけど、当たり前のことに3案件かかって到達した。

で、本質的な問題は「どの実行に属する成果物か」の情報がどこにもないこと。分散システムの「キャッシュの無効化」や「古いレプリカの読み込み」と同じ問題です。解決策はデータベースのMVCCと同じ発想にしました。MVCCっていうのは各行がどのトランザクションで書かれたかを持っていて読み取り時にチェックする仕組みで、これを成果物にも適用する。各成果物に実行IDを刻印して、ゲートはスコープの一致もチェックする。

「全部の成果物にスコープ付けるの大変じゃない？ 忘れたら？」というのは当然の疑問で、だからゲートをフェイルセーフにしました。スコープ情報がない成果物はゲートが止める。通過させるんじゃなくて止める方向に倒す。

ここで面白いのは、この「所有の境界」が前の「時間の境界」のゲートと合流する点です。ゲートの検査項目が「成果物の存在チェック」と「スコープの一致チェック」の2つに統合される。別々に考えていた問題が同じ仕組みで解決される。境界の問題が全部つながっている、というのはこういう意味です。

## 残りの設計判断は「閉ループ」と「モード切替」

ここまでが記事の核心です。残り2つの設計判断は簡潔に。

**レビューの閉ループ**: PPTXのレビューは「直して」→修正→再チェックのループ。データの問題ならビルド工程に、ビジュアルの問題なら見た目担当に差し戻す。レビューエージェントが分類ラベルを付けて、オーケストレーターはラベルでルーティングするだけ。郵便局の仕分け担当と同じで中身は読まない。ビルドからレビューまでを1チームに囲い込んで裁量を渡し、ループ上限を決めて超えたらエスカレーション。

**規模に応じたモード切替**: KOL3人なら1つのオーケストレーターが全部見られますけど、30人だとコンテキストが溢れます。少数モードではオーケストレーターが処理の一部を直接担当、多数モードでは管理に専念。成果物のインターフェースはどちらのモードでも統一します。ポリモーフィズム。正直、多数モードのほうが設計は綺麗で、3人のために大仰な委任構造を組むのはオーバーエンジニアリングな気もするんですけど、この判断が正しかったかは今でもわかりません。

## エージェント設計は暗黙知との戦い

```
┌─ 情報の境界 ──── 誰が何を知るか
├─ 時間の境界 ──── 全体→個別の順序。ゲートで遷移管理
├─ 所有の境界 ──── 成果物にスコープを刻印
├─ 品質ループ ──── レビュー→修正→再検証の閉じたサイクル
└─ モード切替 ──── 規模に応じた実行戦略の切替
```

マルチエージェント設計の本質は「各エージェントが何をできるか」じゃなくて「エージェント同士がどう情報を渡し、どこでチェックし、どう修正サイクルを回すか」の設計です。人間のチームワークの暗黙知を、プロトコルとして形式化する作業。

読者の方のプロジェクトでも、エージェントを分けた途端に「受け渡し」の問題にぶつかると思います。そのとき、境界の種類を分けて考えると整理しやすいかもしれません。情報、時間、所有。この3つが独立しているように見えて実は全部ゲートで合流する、というのが筆者にとっての最大の学びでした。
