3人目のKOLへの指示書が、1人目のコピーになっていました。

キャプションの方向性も、ビジュアルの雰囲気も、ほぼ同じ。KOLごとにフォロワー層もプラットフォームも違うのに。AIに全部任せたのに、人間が疲れたときと同じ劣化パターンが出てる。

これはインフルエンサー向けの投稿指示書をAIエージェントで自動生成しようとしたときの話です。KOL（Key Opinion Leader）に渡す「こう投稿してください」というスライド資料を、8体のAIエージェントが分業して作る。商品調査からビジュアル素材の生成、スライド組み立て、レビューまで全自動。KOL3人なら手作業で半日、8人で丸1日、20人の大型案件だと2-3日かかる仕事を、AIにやらせたかったんです。

最初の設計は単純で、エージェントを役割ごとに分ければスケールするだろうと。マイクロサービスと同じ発想。

**甘かった。**

この記事では、そこから先で踏んだ地雷の話をします。結論を先に言うと、マルチエージェントの設計で一番大事だったのは「各エージェントに何をさせるか」ではなく「誰が何を知るべきか」 — 情報の境界設計でした。

## 「誰が何を知るべきか」が全部を決める

1つの大きなエージェントに全工程を任せると、3人目のKOLあたりから品質が崩れる。LLMはコンテキスト（入力情報）が長くなると後半を軽視する傾向があって、8人分のリサーチと素材生成を1つのコンテキストに詰め込むと、後半が前半のコピーになります。

分けるしかない。でも分けた瞬間に「エージェント間で何の情報を渡すか」の問題が出てきます。

全エージェントにプロジェクト全体の情報を渡す？ やってみたらダメでした。情報が多すぎると、エージェントは何が重要か判断できなくなる。関係ない情報に引っ張られて、的外れなアウトプットを出す。

逆に絞りすぎると？ キャンペーンの方向性を知らないデザイン担当が、ブランドと噛み合わない素材を量産する。

これ、マイクロサービスで全サービスが同じDBを見るのがアンチパターンなのと同じ構造なんですよね。ドメイン駆動設計（DDD）で言う「境界づけられたコンテキスト」 — 各サービスが自分の領域だけのデータモデルを持つ原則が、AIエージェントにもそのまま当てはまった。

落ち着いた設計はこうです。

```
全体方針（全員が知るべきこと）
  ├── ブランドのトーン
  ├── キャンペーンの方向性
  └── 品質基準

個別情報（担当者だけが知ればいいこと）
  ├── 特定のKOLのリサーチ結果
  ├── 特定のプラットフォーム要件
  └── 特定のビジュアル素材
```

ここまではまあ、教科書的。「全体と個別を分離しましょう」なんて誰でも思いつく。問題は、この分離を**実際に機能させる**ときに何が起きるか。

### 全体スタイルが確定する前に個別制作を始めてはいけない

8人のKOLに個別の素材を作るとして、キャンペーン全体のトーンは統一したいわけです。ロゴの使い方、色味、雰囲気。最初はそれぞれのKOL担当エージェントに「キャンペーンの方向性」を渡して個別に作らせてたんですけど、バラバラになる。あるKOLはポップで、別のKOLはシックで。

料理で考えてみてください。ベースのスープを先に作ってから個別のトッピングを変えるべきなのに、8人の料理人がそれぞれ勝手にスープから作り始めたらどうなるか。当然、全然違う味になる。

```
[素材取得] → [全体スタイル確定] → [個別KOL素材生成 ×N] → [統合チェック]
                    ↑
              ここにゲートが必要
```

「全体スタイルの成果物が存在する」ことが、個別制作開始の前提条件。並列にしていいのは「互いに独立したタスク」だけ。KOL同士の個別制作は独立してるから並列OK。でも「全体スタイル → 個別KOL」は直列でないといけない。

「全部同時に走らせちゃえ」の誘惑は強いんですよ。並列にすれば速いから。でも依存関係のあるタスクを並列にすると不整合が起きる。これは並行処理の古典的な問題で、別に新しくもなんともないんですけど、AIエージェントの文脈だと「各エージェントに全情報を渡してるから大丈夫でしょ」という錯覚が生まれやすい。

### ゲートの検査で一番ハマった話

で、この「全体スタイルが確定してから個別制作」みたいなゲート（関所）を、オーケストレーターという司令塔プログラムが管理します。各エージェントの実行順序を決めて、前工程の成果物が揃ったかチェックしてから次に進ませる。

最初、オーケストレーターにリサーチの中身までチェックさせようとしたんです。「ファイルはあるけど中身がスカスカ」を見逃したくなくて。フォロワー分析が500文字以上あるか、とか。

これがマイクロマネジメントでした。

KOLが20人になったら？ 20人分のリサーチ内容を全部読んで品質判断するの？ オーケストレーター自体のコンテキストが溢れて、最初の「1つのエージェントに全部やらせる」問題に戻ります。

組織論で考えるとわかりやすくて、マネージャーの仕事は「中身の正しさの判定」じゃなくて「必要な成果物が全部揃ったか」と「次に進めていいか」の判断です。中身の品質は専門家（レビューエージェント）に任せる。CIパイプラインと同じで、CIサーバーがコードの品質を判定するんじゃなくて、テストが通ったかどうかのゲートだけを管理する。

```
オーケストレーターの責務:
  ✅ 成果物の存在チェック（8人分全員あるか？）
  ✅ 前提条件の検証（全体スタイルが先に確定してるか？）
  ✅ 次工程への遷移判断
  ❌ 内容の品質判断（レビューエージェントの仕事）
  ❌ 工程内の具体的な手順（各エージェントの仕事）
```

**オーケストレーターに「やらないこと」を定義する**のが、実は「やること」を定義するより大事でした。やることはどんどん増やせるけど、増やしすぎるとオーケストレーター自体がモノリスになって壊れる。

### 再実行の粒度も境界設計の一部

ゲートで「揃ってない」と判定されたとき、どこまでやり直すか。8人中7人のリサーチが成功して1人だけ失敗したなら、その1人だけ再実行すればいい。ただし、各エージェントの出力が独立していることが前提です。

「全体の方向性」を決める工程で失敗したら？ 後続の全工程に影響するので、全部やり直し。「個別KOLのリサーチ」で1人だけ失敗したら？ その1人だけ再実行。

再実行の粒度を設計するのも、情報の境界設計の一部だったんですよね。独立している工程は部分再実行できるし、依存関係がある工程は連鎖的にやり直す必要がある。

...ここまで全部、「誰が何を知るべきか」の設計から派生してます。情報の境界を引く → 工程の順序が決まる → ゲートの位置が決まる → 再実行の粒度が決まる。全部つながっている。

## 前の案件の亡霊 — 一番腹が立ったバグ

実運用で最悪だったのがこれです。

案件を2回流したとき、前の案件で作ったカバー画像が残っていて、次の案件のエージェントがそれを「最新の成果物」だと思って使ってしまった。ファイルの場所は同じ、名前も同じ。でも中身は前の案件のもの。

人間だったら「あ、これ古いやつだ」ってわかります。日付を見る、記憶で判断する、「あれ、なんかこの画像見覚えあるぞ？」って気づく。でもAIエージェントには「記憶」がない。ファイルがあれば使うし、古いかどうかは判断できない。

解決策自体は単純で、各成果物に「実行ID」を埋め込みます。読む側は「今回の実行で作ったものか？」をチェックする。古いIDが付いてたら、前の案件の残骸だとわかる。

ただ、全成果物に忘れずスコープを刻印するのは大変です。1つでも漏れたら古い成果物が混入する。だからゲートの検査項目に「スコープの一致チェック」を含めました。刻印を忘れたとしても、ゲートが「この成果物にスコープ情報がない」と検知して止める。

忘れた場合に**通過させるのではなく、止める方向に倒す。** フェイルセーフです。

これも「誰が何を知るべきか」の変種で、正確には「何を知ってはいけないか」の設計。エージェントは「前の案件の成果物」を知ってはいけない。でも黙ってると知ってしまう（ファイルが残ってるから）。だから明示的に遮断する仕組みが要る。

## 15人案件でタイムアウトした、その先

3人のKOLで動いてた設計をそのまま15人案件に流したら、オーケストレーターがタイムアウトしました。15人分のリサーチ結果を全部読もうとして、コンテキストに収まらなくなって。

規模に応じて実行モードを切り替えるしかない。少数モード（3人以下）ならオーケストレーターが処理の一部も直接担当。多数モード（4人以上）なら管理に専念して処理は全部委任、成果物のサマリーだけ読む。どちらのモードでも成果物の形式は統一する。後工程はモードの違いを意識しない。

で、もう1つ嫌な発見があって。

素材を作る工程が重すぎて、全部1つのエージェントにやらせたら、途中で失敗したときどこで失敗したかわからない。しかもコンテキストが長すぎて後半の品質がまた落ちる。

最初の問題と同じ。

大きなパイプラインの中に小さなパイプラインがある。入れ子構造。で、どっちの層にも同じ原則 — 情報の境界、ゲート、スコープの刻印。同じ問題が、違うスケールで再発する。

:::details 全体の設計原則
```
┌─ 情報の境界 ──────── 誰が何を知るか
├─ 全体→個別の順序 ── グローバルを先に確定
├─ 成果物ゲート ────── 完全性チェックで遷移判断
├─ スコープの刻印 ─── どの実行に属するか
├─ 品質の閉ループ ─── レビュー→修正→再検証
├─ 責務の絞り込み ─── オーケストレーターは管理のみ
└─ モード切替 ──────── 規模に応じた実行戦略
```
:::

## 暗黙知を形式化する作業

マルチエージェントシステムの設計は「エージェントの能力」の設計じゃなくて「エージェント間の関係性」の設計です。個々のエージェントが何をできるかより、情報をどう渡し、どこでチェックし、どう修正サイクルを回すかのほうが重要でした。

人間のチームワークでは暗黙知として機能していること — 情報の境界、品質チェック、責任の委譲 — を、全部明示的に設計しないといけない。「あうんの呼吸」はAIには通じません。だから大変。でも逆に言うと、設計が全部見えるのが面白いんですよね。人間のチームだと「なんとなくうまくいってる」の内実を、全部データ構造とプロトコルに落とし込む作業。

正直、この設計で50人のKOL案件が来たら本当にスケールするのかはまだ確信がありません。ゲートが多すぎて処理時間が爆発しないか。エージェント間の受け渡しでデータが劣化していかないか。

やってみないとわからない部分がまだ山ほどあります。
