8体のAIエージェントが協調して1枚のPPTXデッキを生成するシステムを作りました。KOL向けの投稿指示書を自動で作るやつです。で、作ってみてわかったのは、**マルチエージェントシステムの核心はタスクの分割ではなく、「誰が何を知るべきか」の境界設計にある**ということでした。

## 「全員が全部知っている」が壊れる瞬間

3人のKOLでは動いていました。1つのエージェントに全部やらせる素朴な直列パイプラインで。

6人のKOLで回したら壊れました。5人目と6人目のキャプション方向性が1人目とほぼ同一になっていて、「潤い」と「自然由来」をひたすら繰り返してるだけ。コンテキストウィンドウ（LLMが一度に読める情報量の上限）が長くなるとattentionが前半の情報に偏って、後半は前半のコピーを生成し始める。LLMの構造的な限界です。

皆さんの組織でも同じこと起きてませんか？ 5人のスタートアップでは全員が全部知っていても回るけど、50人になったら情報の流通経路を設計しないと崩壊する。AIエージェントもまったく同じでした。

```
全体方針（全員が知るべきこと）
  ├── ブランドのトーン
  ├── キャンペーンの方向性
  └── 品質基準

個別情報（担当者だけが知ればいいこと）
  ├── 特定KOLのリサーチ結果
  ├── プラットフォーム固有の要件
  └── 個別のビジュアル素材
```

DDDの「境界づけられたコンテキスト」がそのまま当てはまります。で、この分離だけだと片手落ちで、「全体方針を誰がいつ確定するか」の順序設計が要る。全体方針そのものがエージェントの出力なので、その品質が低ければ後工程全員が間違った方向に進みます。工場の組み立てラインと同じです。

## ベースのスープと4つの鍋 — Phase 4を分割した話

ここが一番設計判断が込み入ったところで、一番語りたいところです。

画像素材の生成で2つの問題が同時に起きました。新しい案件なのに3人目のKOLのシーン画像だけ前の案件のものが残っている。しかも1つのエージェントで全画像を生成しているので、途中で失敗すると後続のKOLの画像がないまま次のフェーズに進んでしまう。

結論から言うと、Asset生成を4つのステップに分割しました。

```
[素材取得] → [全体スタイル確定] → [KOL個別 ×N] → [統合チェック]
               ↑ ここにゲート       ↑ 並列OK
```

この分割の核心は「全体スタイル確定」と「KOL個別」の依存関係にあります。

全体スタイル確定（4b）が生成するのはキャンペーン全体のビジュアルトーンです。カバー画像、タイトルカバー、商品メイン画像。デッキ全体の色味と雰囲気を決める画像群。KOL個別（4c）はそのトーンに合わせて各KOLのシーン画像を生成します。

**4bが完了する前に4cを起動してはいけません。**

ラーメンで考えたら一発で腹落ちしたんですけど、ベースのスープを先に作らないとトッピングだけ変えても統一感は出ないですよね。スープなしにチャーシューだけ載せても店の味にはならない。4bのグローバルスタイルがベースのスープで、4cのKOL個別画像がトッピングです。スープなしにトッピングだけ作ったら、あるKOLはパステル調で別のKOLはビビッドで、1つの案件なのにデッキに統一感がない。

で、4cではKOL 1名につき1つのエージェントを起動します。1人がコケても他に影響しない。6人のKOLで3人目だけ画像生成に失敗しても、残り5人は正常。失敗した3人目だけ再実行すればいい。これを1つのエージェントに全部やらせていたときは、3人目でコケたら4人目以降も全滅でした。

ここの設計で一番悩んだのは、実は素材取得（4a）と全体スタイル確定（4b）を分ける必要があるかどうかでした。ロゴと商品参照画像の取得と、カバー画像の生成は、1つにまとめてもよさそうに見える。でもやってみると障害モードが全く違う。素材取得はネットワークアクセス主体で失敗パターンが「画像が見つからない」「解像度が低い」。カバー画像生成は画像生成AIの呼び出しで、失敗パターンが「プロンプトの品質」「スタイルの一貫性」。一緒にすると一方の失敗が他方を巻き込んで、どこで壊れたか切り分けできない。分けました。

正直、4分割は過剰設計じゃないかと最初は思っていました。でもやってみると、分割したことで各ステップの再実行粒度が細かくなって、失敗からの復旧が劇的に速くなった。3時間かかっていたデバッグが10分になった、みたいな話です。フラクタル構造というか、パイプライン全体の設計原則（ゲートで区切って、失敗時は部分再実行）がそのまま各工程の内部にも再帰的に適用される。オブジェクト指向のコンポジションパターンと同じで、ツリーの各ノードが同じインターフェースを持つ。

### stale artifact — 前の案件の残骸に騙される問題

前の案件の画像が残る問題。3時間くらい「なんで3人目だけ画像が古いんだ」って追いかけ回して、原因が「前回の失敗で残ったファイルを今回のだと思い込んでた」だったときは、いやもう笑えなかった。マジで腹が立ちました。というか自分の設計が悪かったんですけど。

解決策はデータベースのMVCC（マルチバージョン同時実行制御 — 各レコードが「どのトランザクションで書かれたか」を持つ仕組み）の再発明みたいなもので、各実行にタイムスタンプのIDを振って、成果物にもそのIDを含めて、ゲートでIDの一致を検証する。不一致ならstale（古い実行の残骸）として無視して再実行する。仕組みとしては何も面白くないんですけど、これがないと毎回前の残骸と戦うことになります。

ゲートの話をもう1つ。統合チェック（4d）が出力する最終マニフェストに「status: ready」と書いてあっても安心できません。「status: partial」のままBuildに進んでPPTXの表紙が真っ白になった案件がありました。statusだけでなくcampaign_coverの存在もチェックするゲートを追加しました。ゲートが甘くてゴミが後工程に流れるコストは、ゲートが厳しくて止まるコストの10倍です。

## その他の設計と最終的な全体像

Phase 4以外の話は手短にまとめます。

KOLリサーチから次のフェーズに進む前に、6項目の完全性チェック（Readiness Check）を入れています。5人中3人分しかリサーチが完了していないのに次に進んで、3人目のプランが「フォロワーに響くコンテンツ」みたいな空虚なものになったのがきっかけです。

Build→Reviewの閉ループはAgentTeams（チーム内でエージェント同士が協調するClaude Codeの仕組み）でBuilder・Visualizer・Reviewerを1チームにし、REJECTされたら原因に応じて再実行します。最大10回ループで安全弁。

```
Phase 0:   Initialize
Phase 1:   Intake
Phase 1.5: Product Research
Phase 2:   KOL Research
             → Readiness Check（6項目全件）
Phase 3:   Creative Planning
Phase 4a:  素材取得
Phase 4b:  全体スタイル確定
Phase 4c:  KOL個別 × KOL人数
Phase 4d:  統合チェック
             → 遷移ゲート（status=ready + campaign_cover存在）
Phase 5-7: Build / Visualize / Review（AgentTeams fix loop）
Phase 8:   Export
```

Orchestratorの責務はモード選択、フェーズ順序、ゲート判定に絞りました。各フェーズの具体手順はsubordinate skills（下位定義）に分離してあります。オーケストレーターのファイルが500行を超えたあたりで「これモノリスだろ」と気づいて。CIパイプラインのサーバーがテストの中身を知らないのと同じで、オーケストレーターは「何が入って何が出るか」だけ知っていればいい。各フェーズの内部手順に踏み込むと、マイクロマネジメントの反パターンそのものです。

この設計をやって一番思うのは、マルチエージェントを「タスクの分割」として設計すると情報の境界が曖昧なまま残るということです。「このエージェントに何を見せて何を見せないか」を先に決めたほうが、タスクの分割は自然に決まる。15人のKOLでPhase 4cを15並列spawnしたときにどうなるかはまだ試していません。
