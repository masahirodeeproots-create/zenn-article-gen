3人目のKOLのキャプションが、1人目のほぼコピーになっていました。

8人のKOLにカスタマイズした投稿指示書をPPTXで自動生成するシステムを作っていて、最初は1つの大きなAIエージェントに全部やらせていたんですけど、3人目あたりからクオリティが露骨に落ちるんですよね。5人目のビジュアル指示が2人目と酷似してるし、8人目はもう1人目の使い回し。LLMは長いコンテキストの中で情報がぼんやり混ざっていって、後半が前半の繰り返しになります。

「エージェント分ければスケールするでしょ」。

...分けた瞬間に別の地獄が始まりました。この記事はその地獄の話です。分けた後に起きた問題と、そこから見えてきた「エージェントを分けるより大事なこと」について書きます。

## エージェントを分けたら「ベースのスープ」が消えた

調査担当、デザイン担当、ビルド担当、レビュー担当...と責務ごとにエージェントを分けて、まず最初に困ったのがビジュアルの統一感でした。

8人のKOLに個別の素材を作るとき、キャンペーン全体のトーンは揃えたいわけですよ。ロゴの使い方、色味、全体の雰囲気。で、最初はKOL担当エージェントそれぞれに「キャンペーンの方向性」のテキストを渡して各自に作らせました。

バラバラ。

あるKOLはポップで、別のKOLはシックで、もう1人は謎にレトロ。同じキャンペーンに見えないんですよね。なんでこうなるかっていうと、「豚骨っぽいやつで」って言葉だけ渡して8人の料理人に各自スープから作らせたのと同じだからです。ベースのスープを先に作ってからトッピングを変えるべきだった。

```
❌ やってしまったこと:
  [方向性テキスト] → [KOL担当A] → 素材A（ポップ）
                   → [KOL担当B] → 素材B（シック）
                   → [KOL担当C] → 素材C（レトロ）

✅ こうすべきだった:
  [素材取得] → [全体スタイル確定] → [個別KOL ×N] → [統合チェック]
                  ↑ ここをスキップした
```

全体スタイルが確定する**前に**個別制作を始めちゃダメなんです。

で、ここからが厄介な話なんですけど、8人のKOLは互いに独立しているから、直感的には全員分を同時に走らせたくなるんですよ。速いし。実際、KOL間の並列化自体は正しいんです。でも「全体スタイル確定 → 個別KOL制作」という依存関係は直列で守らないといけない。「何が並列可能で何が直列必須か」を明示的に宣言しないまま「まあ大丈夫でしょ」で全部同時に走らせて、案の定壊れました。

分散システムの古典的なバグですよね。依存関係のあるタスクの並列実行。知識としては知っていたのに踏んだ。

この経験から、工程間に**ゲート**を置くことにしました。前の工程の出力が所定の条件を満たしているか確認してから、次の工程に進む。「全体スタイルの成果物が存在する」ことが個別制作開始の前提条件。ゲートがOKを出すまで後工程は動かない。

ゲートで何を見るかもけっこう悩みました。最初は出力の中身の品質を検査しようとしたんですけど、たとえば調査エージェントが「このKOLは20代女性に人気」って出してきて、それが正しいかどうかなんて次のエージェントにはわからない。オーケストレーターにもわからない。

結局、ゲートは**中身を見ない**という結論に至りました。ゲートがやるのは「成果物の存在と完全性のチェック」だけ。8人分のリサーチ結果が全部あるか。必須フィールドが埋まっているか。3人分しかないのに次に進もうとしていないか。中身の正しさはレビューエージェントの仕事で、ゲートの仕事じゃない。

ここで面白かったのが、このゲートの考え方がパイプラインの中で**再帰的に**必要になったことです。素材制作工程が重すぎて1エージェントに全部やらせたら、途中で失敗したときにどこで壊れたか切り分けられない。最初からやり直しになって、えっ、これ「1つのエージェントに全部やらせたら品質が落ちた」のとまったく同じ問題じゃないですか。粒度が違うだけで構造が同じ。

```
パイプライン全体:
  [調査] →ゲート→ [計画] →ゲート→ [制作] →ゲート→ [レビュー]

「制作」の内部にも同じ構造:
  [素材取得] →ゲート→ [全体トーン] →ゲート→ [KOL個別×N] →ゲート→ [統合]
```

大きなパイプラインの中に小さなパイプラインがある。ゲートの設計原則が再帰的に適用される。同じ原則の適用先が増えるだけで新しいルールが増えるわけじゃないんですよね。

ゲートの密度にはまだ最適解が見えていなくて。全サブステップにゲートを入れると安全ですけど遅い。「ここはゲートいらないでしょ」と省略して2回事故りました。今は全部入れてます。過剰設計かもしれません。でも事故るよりはマシです。

## 前の案件の画像がそのまま出てきた

これが一番腹が立った話です。

案件を2回流したとき、前の案件の成果物が残っていて、新しい案件のエージェントがそれを「最新の成果物」として使ってしまいました。前の案件で作ったカバー画像が、新しい案件のPPTXにそのまま載っている。納品直前に気づいて肝が冷えました。

ファイルの場所も名前も同じ。中身だけが古い。人間なら「あ、これ前のやつだ」って気づきますけど、エージェントにはその判断ができないんですよ。

嘘でしょ。

最初の対策は「前の案件のファイルを手動で消してから走らせる」。消し忘れて2回目もやらかしました。3案件目でようやく「人間の注意力に依存する設計は必ず破綻する」という当たり前のことに到達しました。3案件かかった。

本質的な問題は「この成果物はどの実行に属するか」という情報がどこにもないことでした。成果物に実行IDを刻印して、ゲートが「この成果物のスコープが今回の実行と一致するか」をチェックするようにしました。スコープ情報がない成果物はゲートが止める。通過させるんじゃなくて止める。フェイルセーフです。

ここで前のセクションの話とつながるんですけど、ゲートの検査項目が「成果物の存在チェック」と「スコープの一致チェック」に統合されるんですよね。全体スタイルが存在するか確認するのも、その全体スタイルが今回の実行のものか確認するのも、同じゲート機構でやれる。別々に考えていた「順序の問題」と「所有の問題」が、同じ仕組みの上で合流する。

これが見えたときはちょっと嬉しかったです。設計がちゃんと一貫してるんだな、と。

:::details レビューループとモード切替の話

残り2つの設計判断を簡潔に。

**レビューの閉ループ**: PPTXのレビューは「直して」→修正→再チェックのループになります。データの問題ならビルド工程に、ビジュアルの問題なら見た目担当に差し戻す。レビューエージェントが分類ラベルを付けて、オーケストレーターはラベルでルーティングするだけ。ビルドからレビューまでを1チームに囲い込んで裁量を渡し、ループ上限を決めて超えたら人間にエスカレーション。

**規模に応じたモード切替**: KOL3人と30人で同じ設計は使えません。3人なら1つのオーケストレーターが全部見られますけど、30人だとリサーチ結果を読むだけでコンテキストが溢れます。少数モードではオーケストレーターが処理の一部を直接担当、多数モードでは管理に専念してサマリーだけ読む。成果物のインターフェースはどちらのモードでも統一。正直、多数モード側のほうが設計は綺麗で、3人用に大仰な委任構造を組むのはやりすぎな気もしますけど、この判断が正しかったかは今でもわかりません。

:::

## エージェント設計で一番大事だったこと

マルチエージェント設計をやってみて一番の学びは、「各エージェントが何をできるか」より「エージェント同士がどう情報を渡すか」のほうがはるかに重要だったということです。

人間のチームワークでは暗黙知として機能していること — 「これ前の案件のだよ」「先に全体の方向性決めようよ」「必要なもの全部揃った？」 — を、AIエージェントでは全部プロトコルとして形式化しないといけない。面倒ですけど、やってみると「人間のチームって実はこんなに複雑なことを無意識にやってたのか」と逆に感心します。

あなたのプロジェクトでエージェントを分割するとき、「分け方」より先に「ベースのスープを誰がいつ作るか」を考えてみてください。たぶんそこが最初の落とし穴です。
