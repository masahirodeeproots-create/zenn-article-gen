マルチエージェントシステムの設計で一番難しいのは、エージェントの能力ではなくて「誰が何を知るべきか」の境界設計です。

KOL向けの投稿指示書を8体のAIエージェントで自動生成するシステムを作りました。で、わかったのは、壊れるのは常にエージェントの「間」だったということです。個々のエージェントが賢いかどうかはほとんど関係なかった。この記事ではその設計原則の話をします。

## 分割した瞬間に始まる本当の設計

1つの巨大なエージェントに全部やらせてみたら、3人目のKOLあたりからキャプションの方向性が1人目のコピーみたいになりました。コンテキストウィンドウに全情報を詰め込むと、LLMは後半で前半のパターンを繰り返すようになります。分割は不可避でした。

ここからが長い話になります。

分けた瞬間に「何を渡すか」の設計が必要になる。マイクロサービスではサービス間のAPIを明示的に定義しますけど、エージェントでも同じです。「なんとなく伝わる」は通用しません。

DDDの「境界づけられたコンテキスト」って概念がありますよね。ドメインごとに情報の境界を引いて、外部とは明示的なインターフェースでやり取りする。マイクロサービスで「全サービスが同じDBを見る」のがアンチパターンなのと同じで、全エージェントに全情報を渡すのはダメでした。LLMは長いコンテキストの中で関係ない情報に引っ張られるんですよね。絞りすぎると今度は必要な文脈を持てなくて的外れなアウトプットを出すし、加減がめちゃくちゃ難しい。

で、試行錯誤の結果たどり着いたのが「全体方針」と「個別情報」の分離です。

8人のKOLに個別の素材を作りつつキャンペーン全体のトーンを統一したい。最初はKOL担当エージェントそれぞれに「キャンペーンの方向性」をテキストで渡して個別に作らせました。バラバラになりました。あるKOLはポップで、別のKOLはシック。同じキャンペーンに見えない。

料理で言うとベースのスープを先に作ってからトッピングを変えるべきだったんです。スープなしにトッピングだけ変えても統一感は出ない。

```
[素材取得] → [全体スタイル確定] → [個別KOL素材 ×N] → [統合チェック]
                    ↑
               ここにゲート
         全体スタイルが確定するまで
           個別制作を始めちゃダメ
```

ここで並列処理の誘惑に負けて「全部同時に走らせちゃえ」ってやると不整合が起きます。並列にしていいのは互いに独立したタスクだけ — KOL間は独立してるから並列OK、でも「全体スタイル → 個別KOL」は直列でないといけない。

情報の境界設計は「誰が何を知るか」だけの話じゃなくて、**「どの順序で情報を確定させるか」の設計でもある**ということに、ここでようやく気づきました。上流の品質が下流を規定するので、工場の組み立てラインと同じく工程間にゲートが必要になります。

で、このゲートが何を検査すべきかがまた悩ましくて。調査エージェントが「このKOLは20代女性に人気」って出してきたとして、それが正しいかどうかなんて次のエージェントには判定できません。もっと地味な問題として、8人分のKOLリサーチを並列で走らせたとき全員分揃ったかどうかの確認。これがないと後工程が中途半端な入力で動き出す。

組織論で考えたら腹落ちしました。マネージャーの仕事って中身の正しさを判定することじゃなくて、「必要な成果物が全部揃ったか」と「次に進めていいか」の判断なんですよね。CIパイプラインとまったく同じ構造です。CIサーバーがコードの品質を判定するんじゃなくて、テストが通ったかどうかのゲートを管理する。

```
オーケストレーターの責務:
  ✅ 成果物の存在チェック（8人分あるか？）
  ✅ 前提条件の検証（必須フィールドが埋まっているか？）
  ✅ 次工程への遷移判断
  ❌ 内容の品質判断（それは専門エージェントの仕事）
  ❌ 工程内の具体的な手順（各エージェントに任せる）
```

情報の境界、確定の順序、ゲートの設計。この3つが全部「誰が何を知るべきか」から派生している。分けた瞬間にこの設計空間が一気に開くんですけど、逆に言えばここさえ固めれば、個々のエージェントの実装はわりとどうにでもなります。

あなたがマルチエージェントシステムを設計するとき、最初に考えるべきは「どんなエージェントを作るか」じゃなくて「エージェント間で何を渡して何を渡さないか」です。これだけは断言できます。

## 古い成果物の亡霊

ここからは短く。

案件を2回流したとき、前の案件の成果物が残っていて、次の案件のエージェントがそれを「最新」だと思って使いました。前のカバー画像がそのまま新しい案件に紛れ込む。ファイルの場所も名前も同じだから。マジで腹が立ったんですけど、んで、怒りが収まってから考えたらエージェントは悪くなくて、「どの実行に属する成果物か」の情報がどこにもない設計が悪い。

分散システムの「古いレプリカを読む」問題と同じです。データベースのMVCCと同じ発想で、各成果物に実行IDを刻印して、ゲートでスコープの一致をチェックするようにしました。刻印漏れ対策として、スコープ情報がない成果物はゲートが止める。フェイルセーフです。最初は「前のファイル消してから走らせればいい」と思ってたんですけど、消し忘れで2回やらかして、人間の注意力に依存する設計は破綻するという当たり前の教訓を得ました。3案件かかった。

## レビューの閉ループとモード切替

PPTXのレビューは「ここ直して」→ 修正 → 再チェックのループです。データの問題なのか見た目の問題なのかで直すべきエージェントが違うので、レビューエージェントが分類ラベルを付けてオーケストレーターがルーティングします。郵便局の仕分けと同じで中身は読まない、宛先だけ見て振り分ける。ビルドからレビューまでを1つのチームとして囲い込んで、内部ループはチームが自律的に回す設計にしました。ループ上限を超えたら人間にエスカレーション。

```
[ビルド] → [ビジュアル] → [レビュー]
                              ↓ REJECT
                        ラベルで振り分け
                      data → ビルドへ
                      visual → ビジュアルへ
                              ↓ APPROVE
                           出力へ
```

スケール時のモード切替もあります。KOLが3人ならオーケストレーターが直接処理の一部も担当して、30人ならオーケストレーターは管理に専念して全部委任する。成果物のインターフェースはどちらのモードでも統一。ポリモーフィズムですね。正直、多数モードのほうが設計は綺麗だし、少数モードでオーケストレーターに処理を持たせてるのは美しくないです。でも3人のKOLのために大仰な委任構造を組むのもどうかと思っていて、この判断が正しかったのかは今でもわかりません。

:::details 設計原則が再帰する話
素材生成の工程が重すぎて分割したんですけど、分割した小さなステップにもパイプライン全体と同じ設計原則が必要でした。情報の境界、ゲート、スコープ刻印...全部。大きなパイプラインの中に小さなパイプラインがある。フラクタル構造です。「各工程の内部まで同じ設計にするのは過剰設計では？」と最初は思ったんですけど、内部にゲートなしで走らせたら途中の失敗を切り分けられなくて結局最初からやり直す羽目になりました。
:::

## まとめない

最初は「エージェント分ければスケールする」くらいに思ってました。

```
┌─ 情報の境界 ──────── 誰が何を知るか
├─ 確定の順序 ──────── 全体→個別は直列
├─ 成果物ゲート ────── 完全性チェックで遷移判断
├─ スコープの刻印 ─── どの実行に属するか
├─ 品質の閉ループ ─── レビュー→修正→再検証
└─ モード切替 ──────── 規模に応じた実行戦略
```

人間のチームワークでは暗黙知として機能していることを、AIエージェントシステムでは全部プロトコルとして形式化しないといけない。そこが大変だけど、そこにしか面白さはないという気がしています。
