深夜、ターミナルのログを眺めていて手が止まりました。修正ループが12回回っています。APIコストを確認したら、この1実行だけで¥4,200が飛んでいました。「止めるのを忘れた」という話ではなく、止め方を知らなかったんです。

別の実行では、キャプション本文に「医師が認めた」「皮膚科推薦」という薬機法アウトの表現が3箇所混入していました。5フェーズ前に明示的にNGワードとして記録したはずのものが、何事もなかったかのように生成物の中に出てきていた。

SNSインフルエンサー（KOL）への投稿指示書スライドを、受注書・KOLリスト・商品情報から全自動生成するシステムを10体のClaudeエージェントで作っています。KOL10人分で45分かかっていた手作業が最終的に13分になりましたが、その過程でこういう事故をいくつか踏みました。失敗を辿っていくと、毎回同じ場所に戻ってきます。**「誰が何を書き、誰が何を読むか」の設計が曖昧だった**。それがこの記事の一本の軸です。

---

## 最初の失敗：1体に全部やらせた

「大きな1つのAIに全部やらせれば済む」という仮説から始めました。

KOL10人分を一括で投げたとき、8人目あたりでKOL1の「ピンクトーン×ナチュラル系」という方向性が消えて、KOL1のキャプションに「クールホワイト系スタジオ演出」という真逆のディレクションが混入してきました。出力を見たとき、正直途方に暮れました。KOL1のファン層は「等身大ナチュラル系」のはずなのに。

コンテキストウィンドウの物理的な制約の前では、エージェントの性能は関係ありません。1体のエージェントに10個の仕事をさせれば、後半で前半を忘れます。これはLLMの「賢さ」の問題ではなく、情報を「渡す設計」の問題だと気づいたのはここからでした。

辿り着いた原則が「1エージェント1成果物」です。

```
[入力] → [調査Agent] → [設計Agent] → [生成Agent] → [組版Agent] → [出力]
           ↓              ↓              ↓              ↓
       research.json  plan.json    content.json   binding.json
```

各エージェントは1つのファイルを出力し、次のエージェントはそのファイルだけを読んで動く。エージェント間の情報伝達はコンテキスト継承ではなくファイル渡しで行います。後半エージェントが「前半で決めたこと」を記憶するのではなく、ファイルを通じて「取りに行く」設計です。

「誰が読む」のかを決めることで、「何を書くか」の粒度も自然に決まってきます。

## スタイルアンカー：一貫性と個別化を時間的に分離する

次の壁は「キャンペーンとしての統一感」と「KOLごとの個性」の矛盾でした。全員同じ指示書にすればKOLのファンには刺さらない。KOLごとに完全独立で設計すれば同一キャンペーンに見えなくなる。

最初の試みは「KOL3人分を一括プランニングエージェントに渡す」でした。トークン使用量が52,000に達したあたりで、KOL-Aのキャプションに書いたはずの「NG表現: 完全に焼けない・100%防御」という制約が跡形もなくなっていた。前半の情報が静かに消えていく。

解決策は同一フェーズで解こうとするのをやめて、時間的に分離することでした。

```
Phase A: キャンペーン共通スタイル確定
   ↓（ブランドカラー・トーン・NGワードを1ファイルに固定）

Phase B: KOL別カスタマイズ（並列可）
   KOL-A: 共通スタイル + KOL-A個性 → A専用設計
   KOL-B: 共通スタイル + KOL-B個性 → B専用設計
   KOL-C: 共通スタイル + KOL-C個性 → C専用設計
```

「スタイルアンカー」と呼んでいます。キャンペーン全体のビジュアルアイデンティティを先に1ファイルに固定し、すべての後続エージェントがそれを「制約」として参照する。KOL個別エージェントは「このアンカーの範囲内でKOLらしさを最大化せよ」という命令になります。

あなたのワークフローに「グローバルな制約」と「個別の裁量」の両方が必要なとき、同じフェーズで解こうとしていませんか？先にアンカーを確定してから個別化する、という時間的分離は、コンテキストウィンドウの問題だけでなく品質の一貫性問題も同時に解きます。

もう一つ気づいたこと。KOL調査（何者か）とKOLクリエイティブ設計（どう表現するか）を同じエージェントに混ぜると品質が落ちます。前者は事実収集、後者は解釈と戦略。「誰が何を書くか」の分解粒度として、この2つは別エージェントに分けるべきでした。

## ループを止めるタイミング

¥4,200が飛んだ夜の話に戻ります。

ログを見ていて血の気が引きました。修正ループが12回回っている。9回目も10回目も、スコアは7.8のまま動かない。11回も12回も同じ。APIコストを確認したら、この1実行で普段の何日分かが消えていました。うんざりしました。

止めすぎても問題がありました。スコア5.0で打ち切ったら、キャプションに必須ワードが半分しか含まれていなかった。

「どのスコアなら合格か」という絶対基準を定義しようとしていたのが間違いで、重要なのは**変化の有無**でした。

```
停止条件の組み合わせ:

① スコア ≥ 9.0 が 2連続 → 良い水準で安定した。STOP
② スコア変化 ±0.5 以内が 3連続 → 改善が止まった。STOP
③ 最大10回 → どんな場合もここで打ち切り
```

①②③のどれかが先に発火したら止めます。¥4,200のケースは②が6回目で発火すべきでした。そうすれば¥1,700前後で済んでいた計算になります。

もう一つ。素材の品質と成果物の品質は別サイクルにする価値があります。素材（調査・設計）の品質が低いまま生成ループに入ると、生成をどれだけ改善しても上限が低い。最初は生成ループだけに注力して頭打ちになって、素材フェーズに戻ったら品質が一気に跳ね上がったことがありました。「原因は下流ではなく上流にある」——これは後から気づくと手遅れなので、先に2段構造（素材PDCA → 生成PDCA）を設計に組み込んでおくのが合理的です。

## ファイルを「契約書」として設計する

ここが核心です。

エージェントが10体になると「前フェーズで決めたことを後フェーズが知らない」という問題が繰り返されます。KOL-Aのキャプション生成フェーズで、5フェーズ前にNGワードとして記録した「医師が認めた」「皮膚科推薦」という薬機法アウトの表現が普通に出てきました。しかも3箇所。

最初は「プロンプトに全情報を詰め込む」で対処しようとしました。でも10体分の累積情報を全プロンプトに入れるとトークン爆発します。前述の積み上げ方式の欠陥と同じ問題です。

転換点は「情報の所有者と参照者」を設計する、という発想に辿り着いたときでした。

```
❌ 積み上げ方式（情報劣化しやすい）
   PhaseA の全出力 + PhaseB の全出力 + ... → PhaseN のプロンプトに全部入れる
   → トークン爆発 + 後半エージェントが前半情報を無視し始める

✅ 引き出し方式（情報劣化しにくい）
   各フェーズが「必要なファイルだけを読む」
   → Captionエージェントは per_kol_packages.json の NGワードだけを読む
   → BuildエージェントはAssetsファイルだけを読む
```

ファイルを「情報の置き場所」として捉えるのをやめました。**「このファイルのこのフィールドに、このエージェントが依存する」という依存関係を、設計段階で明示的に決める**。NGワードは調査エージェントが `per_kol_packages.json` の `caption_direction` フィールドに記録し、キャプションエージェントが必ずそのフィールドを読む——という「契約」が存在することが重要です。この設計にしてから、NGワード混入件数はゼロになりました。

フェーズゲートはこの契約設計の具体的な実装です。

```
Phase 4a: Asset Acquisition → source_assets_manifest.json
          ↓ Gate: ファイルが存在するか？
Phase 4b: Global Style Generator → campaign_cover.png, global_assets_manifest.json
          ↓ Gate: 3画像すべて存在するか？
Phase 4c: KOL Creative Worker × KOL数
          ↓ Gate: 全KOLのassets_plan存在 + run_id 一致？
Phase 4d: Asset Resolver → assets_resolved.json (status=ready)
          ↓ Gate: status=ready か？
```

各フェーズは「必須成果物ファイルが存在するか」を確認してから次に進みます。存在しなければ当該フェーズを再実行する。これで冪等性を保ちながら部分的な再実行が可能な設計になります。

`run_id` はここで重要な役割を果たします。`run_state.json` に実行IDを持たせ、各フェーズ出力の `run_id` と照合することで「古いアーティファクトを掴んだまま次フェーズが動く」問題を防げます。あるKOLのアセットが今回の実行で生成されたものか前回の残骸かは、ファイルの存在だけでは判断できない。`run_id` の一致が「今の契約か」を確認するゲートになります。

正直に白状すると、フィールドの所有権の設計が遅れたことで「このフィールドは誰が書くの？」という問いが後から頻発しました。`per_kol_packages.json` の `render_medium` フィールドはPlannerが書くのかKOL Creative Workerが書くのか、一時期曖昧になっていた。んで、両者が書こうとして後者が前者を上書きするバグが出た。「あとで決めればいい」という先送りが、後から手痛い形で返ってきます。所有権は設計の最初に決めるべきでした——というか、決めないと設計と呼べないかもしれません。

「誰が書き、誰が読むか」という問いに答えないまま実装を進めると、後から必ずここに戻ってくることになります。

## 並列化の境界は依存グラフにある

KOL-Bのシーン画像だけ、ブランドカラーが正しいオレンジ系（`#F28C38`）ではなく紺系（`#1A2B3C`）で生成されていました。そのままクライアントに送付してしまい、30分後に「カラーが全然違う」という連絡が来ました。嘘でしょ。

原因は、グローバルスタイル生成とKOL個別クリエイティブ設計を「並列にできると思って」同時起動したことでした。KOL-Bのエージェントが起動したとき、`global_assets_manifest.json`（ブランドカラーが記録されているファイル）がまだ存在していなくて、エージェントがデフォルト値で補完してしまった。ゲートが機能していなかった。

依存グラフで考えると、並列化の境界は自然に見えてきます。

```
[入力]
  ├─→ [KOL調査] ──────────────────────────┐
  └─→ [商品調査] ──→ [統合プランニング] ─→ [グローバルスタイル生成]
                                                ↓ (完了を確認してから)
                     ┌── [KOL-A クリエイティブ] ─→ [A画像生成]
                     ├── [KOL-B クリエイティブ] ─→ [B画像生成]
                     └── [KOL-C クリエイティブ] ─→ [C画像生成]
                                                        ↓
                                                  [アセット統合]
                                                        ↓
                                           [スライド組版 → レビュー]
```

並列にできる箇所は2ヶ所だけです。KOL調査と商品調査（どちらも入力だけに依存）。そして全KOLのクリエイティブ（グローバルスタイル完了後に全員並列スタート）。グローバルスタイルはその前段として直列でなければならない。

「前のフェーズの結果が必要か？」だけで並列化を判断しようとすると見落としが出ます。正しい問いは「後段エージェントが参照するファイルが、起動する時点で確実に存在するか？」です。フェーズゲートがあれば、これを仕組みとして保証できます。

「1エージェント1成果物」という設計は、並列化競合も同時に解決します。KOL-A/B/Cがそれぞれ異なるファイルを出力する設計になっているので、同一ファイルへの競合書き込みが発生しない。「誰が書くか」を1体に限定することが、並列化の安全性を担保します。

KOL10人の処理が45分から13分になった恩恵の大部分は、この並列化にあります。KOL-A/B/C/Dの4体が同時にクリエイティブ設計を走らせることで、直列なら20分かかるフェーズが5分で終わります。依存グラフの設計が、速度とコストの両方を決めていました。

## 残った問い

5ラウンドの設計を経て、「1エージェント1成果物」という原則が今回のほとんどの問題を解きました。コンテキスト爆発、情報劣化、並列化競合——これらはすべて「誰が何を書き、誰が何を読むか」の設計が曖昧なことから起きていました。

ただ、一つだけ設計で回避しただけで根本から解いていない問いが残っています。

**並列エージェントが同一のファイルを書こうとした場合、どうなるか。**

今回はKOL-A/B/Cがそれぞれ異なるファイルを出力することで競合を回避しました。しかし複数エージェントが共有サマリーファイルに結果を追記しようとするケースは、設計で禁止しているだけで仕組みとして防いでいない。ファイルロック、書き込みキューイング、マージ戦略——いずれかが本来は必要なはずです。

今は「1エージェント1成果物という設計なのだから、共有ファイルへの複数書き込みはしない」という暗黙の合意でシステムが成立しています。この合意は、誰かが外しにかかったときには止められません。

あなたのマルチエージェントシステムでは、この問題をどう扱っていますか。コメントで聞かせてほしいです。
