# Technical Article Style Guide

日本語技術記事を人間が書いたものと区別できないレベルで生成するためのガイド。

---

## 🔴 CRITICAL REQUIREMENTS（出版ブロッカー）

---

### ⚠️ 文体チェック：生成前・生成後の両方で実行すること

**生成した記事の最初の5文を読め。全て「〜です」「〜ます」「〜ました」で終わっているか？**
1つでも「〜だ」「〜である」「〜した。」（体言止め含む）があれば、書き直せ。これは他のすべての要件より先に確認する。

✅ 合格: 「Providerタワーがよく見られます。」「次のRFCです。」「問題になりました。」
❌ 不合格: 「整理すると:」「得たもの: X。」「失ったもの: Y。」「〜が問題だ。」

---

### 1. です・ます体（丁寧体）を全体で統一

**CRITICAL: 「だ/である」は出版ブロッカー。体言止めも同様。**

- 文末は必ず「〜です」「〜ます」「〜ました」「〜でしょう」のいずれかで終える
- カジュアル表現を自然に混ぜる: 「〜ですね」「〜なんです」「〜でしょう」

❌ 「結論から言うと、これProxyとWeakMapだけで実装できる。」（だ体）
❌ 「得たもの: コンテキストサイズの予測可能性。」（体言止め）
✅ 「結論から言うと、これはProxyとWeakMapだけで実装できます。」（です体）
✅ 「得たのはコンテキストサイズの予測可能性でした。」（です体）

### 2. フロントマター

```yaml
---
title: "記事のタイトル"
emoji: "🎯"
type: "tech"
topics: ["claudecode", "ai", "architecture"]
published: true
---
```

### 3. 技術的正確性

- ソースマテリアルの事実に基づく（捏造禁止）
- コード例は正確に動作するものを
- 具体的な数字はソースマテリアルから引用

### 4. 自然な日本語

- 翻訳調ではなく、最初から日本語で考えて書いたように
- 技術用語は日本の技術コミュニティで一般的に使われる英語をそのまま使用

### 5. 🔴 概念スレッド（Conceptual Thread）

**記事全体を貫く「1つの概念的フレーム」を冒頭か第1セクションで明示する。ただし明示するのは最初の1回だけ。**

人間の優れた技術記事は複数の話題を並べるのではなく、1つの鋭い切り口で全体を解釈し直す。`react-server-components-multi-stage.md` が「RSC = 多段階計算」という1概念で全を統一したように、この記事は何を軸に読むべきかを冒頭で提示する。

**テーゼは最大2回（冒頭1回、結論で1回）。それ以上は読者への不信である。**

❌ 「複数の問題を時系列で並べ、最後に『これらは全部同じ根っこでした』と述べる」（概念フレームが末尾に出てくる）
❌ 「これもコンテキスト管理の話に戻ってきます。」「結局これもコンテキストの問題でした。」（本文中での繰り返し）
✅ 「冒頭で概念軸を提示し（例: 『これはコンテキストと状態管理の問題だ』）、各セクションがその軸の別の側面を演じさせ、著者はそれを指摘しない」

**チェック**: 記事冒頭を読んで「この記事は何を軸に読むか」が1文で言えるか？テーゼを述べている文が3箇所以上あれば2箇所削除する。

### 6. 🔴 深さ優先の構造（Depth over Breadth）

**1記事 = 2〜3個の概念を徹底的に掘る。6個以上の概念を浅く並べるのは禁止。**

人間の技術記事は1つのアイデアに対して「そこまで深掘りするのか」という密度で書く。概念の数を減らし、1概念あたりの行数を増やす。

❌ 6つの設計原則を各20-30行で紹介する「機能パレード」
✅ 1つの設計原則を120行かけて、仕組み→問題→試行錯誤→コード→限界まで掘る

**チェック**: 記事内の主要概念を数えて、4個以上あれば統合か削除を検討する。

### 7. 🔴 コード例の必須化

**主要概念ごとに最低1つの実質的なコード例を含める。JSONスニペット1つだけでは不足。**

「実質的」とは: 読者がそこから学べる、動作原理が見えるコード。設定ファイルの出力例ではなく、ロジックを示すコード。

未実装・設計案のコードを掲載する場合は、コメントで明示すること（`# 未実装 — 設計案`）。動作するコードとして提示すると読者を誤解させる。

❌ 記事全体でJSON出力例が1つだけ
❌ 未実装コードを機能するコードと同じ体裁で掲載する
✅ 各主要セクションにPython/TypeScript/YAMLなどのコードがあり、設計判断の根拠がコードで見える
✅ 未実装の場合: ブロック冒頭に `# 未実装 — 設計案` を入れ、なぜまだ実装できていないかを本文で説明する

```python
# ❌ 出力フォーマットの例示だけ
{"status": "success", "output_files": ["..."]}

# ✅ 設計判断が見えるコード
class SkillRouter:
    def select(self, query: str) -> list[Skill]:
        # フロントマターだけを読んで候補を絞る（詳細ロードを避ける）
        candidates = [s for s in self.skills if s.matches_trigger(query)]
        # embedding距離でランク付け
        ranked = sorted(candidates, key=lambda s: self.embed_distance(query, s.summary))
        return ranked[:3]  # 上位3つだけ返す
```

### 8. 🔴 分析的権威（Authority from Analysis）

**「私が作った」ではなく「私が分析した/調べた」という権威の源泉で書く。**

読者を感心させるのではなく、読者に考えさせる。ソースコードを読んだ、RFCを調べた、ベンチマークを取った、という知的作業から権威が生まれる。

❌ 「103個のSkill、22個のエージェント、7,610行の状態マシンという規模になりました」（実績の誇示）
✅ 「Skillが100個を超えたあたりで、起動時のコンテキスト消費が問題になりました。なぜ100個が境界なのか調べてみると...」（分析への導入）

❌ 数字を印象付けとして提示する
✅ 数字を分析の出発点として提示する（「103個」→「なぜ103個で、50でも200でもないのか」）

### 9. 🔴 観察可能な失敗の具体例（Concrete Failure Artifacts）

**問題を述べるセクションには、実際に何が起きたかを1文で示す。「品質が落ちた」は説明ではない。**

人間のベンチマーク記事（`immutable-immer.md` の実測値テーブル、`typescript-intrinsic.md` のコンパイラ出力）は、抽象的な「問題」を具体的な観察で裏付ける。問題の診断が説得力を持つのは、症状が見えるからだ。

❌ 「後半のKOLへの指示品質が前半と比べて明らかに落ちる」（何が落ちたのかが見えない）
❌ 「モデルの品質が粗くなる」（読者は実際の出力を想像できない）
✅ 「3人目のKOLのキャプション方向性が1人目とほぼ同一になった（"潤い"と"自然由来"を繰り返すだけになった）」
✅ 「Phase 3で4人目のKOLのクリエイティブスタイルが、担当モデルの個性でなくデフォルト調になった」

**チェック**: 各「問題セクション」の冒頭に「実際の出力はこうなった」という1文があるか？なければ追加する。

## 🟡 IMPORTANT: 人間らしい文体パターン

### 10. 知的探索としての構造（Intellectual Progression）

記事の構造は「問いの連鎖」で組む。各セクションが前のセクションの発見から生まれる新しい問いで始まり、論理的緊張感が最後まで続く。

❌ 機能パレード: 設計思想 → 3層構造 → LEADパターン → Skill → Feedback → まとめ（各セクションが独立）
❌ 解決カタログ: 「問題Aを解いたら→問題Bが出てきた→問題Bを解いたら...」という線形列挙
✅ 知的連鎖: 問い → 調査 → 発見 → **その解決が見えてきた新しい問い** → さらに深く（「解決できた。ではなぜこれで解決できたのか？次の崩れはどこか？」という流れ）

**構造チェック**: 各H2の冒頭に「前のセクションの発見からこの問いが生まれた」と説明できるか？できなければ構造がフラット。セクションの冒頭1〜2文を書き直す。各H2末尾または次のH2冒頭に「前セクションの解決が生んだ問い」を1〜2文で明示すること。

**Fix Loopやサブシステムのセクション**: 「Phase N → Phase N+1への接続」を飛ばさない。Phase 4のstale artifact問題を解決したら「ではPhase 5のループでは何が問題になったか」という問いで次のセクションに入る。セクションが孤立しているなら接続文を冒頭に足す。

### 10a. 🟡 失敗の誠実な再現（Honest Failure Trace）

**解決策の前に「誤った仮説」または「迷走の記録」を入れる。問題特定のプロセスが抜けると、解説ではなく「解決策カタログ」になる。**

技術記事の読者は「同じ問題に自分が直面したときどう考えるか」を求めている。正しい解決策だけを書いても、その正しさを体感できない。誤った仮説→失敗→観察→正しい仮説の順で書くことで、読者は思考の歩き方を学べる。

❌ 「再現条件の特定に数時間かかりました」で片付ける（迷走を言及したことにしているが内容がない）
❌ 「この失敗の原因を追うと、〜の問題でした」（原因特定が1文で済んでいる）
❌ 「ルールを追加したらREJECT率が下がりました」（何を試して何が効かなかったかが省略されている）
✅ 「最初はXが原因だと思ってYを試した。直らなかった。次にZを疑ってWを確認したとき、ようやく実際の原因が見えた」
✅ 「ログに失敗が残らないタイプのバグで、成功しているように見えるのに出力が違う。どこを疑えばいいか全くわからなかった」

**チェック**: 問題を提示したセクションに「最初に疑ったこと」または「試して失敗したアプローチ」が1段落以上あるか？なければ追加候補。特に「ルールを入れたら直った」「設定を変えたら解決した」という1文解決は迷走の記録がない疑いが強い。

### 11. トーン・声

- 「筆者」は控えめに（3-5回まで）
- **ボールド予算 = 1記事5-6箇所まで。** それ以上はスライドデッキ化する
- 説明の深さは興味で変える（教育的重要性ではなく）

❌ 15箇所以上を太字にして強調する（プレゼンスライドのようになる）
✅ 本当に記事の核心となる概念だけ5-6箇所を太字にする

**ボールドカウント手順**: 生成後に `**` をカウントして開始タグ数を数える。7個以上あれば記事の核心でない箇所から削除する。「解決策名」「手順の番号」「コードの説明」は通常ボールド不要。

### 12. 教育的スキャフォールディングの完全禁止

読者に「ここが重要ですよ」「次にこれを見ますよ」と誘導するフレーズは全て禁止。内容の力で読者を引き込む。

❌ 「ここが一番大事なルールなんですが」「もう少し掘ります」「詳しく話します」「では〜を見ていきましょう」
✅ 誘導フレーズなしで、いきなり内容に入る。重要性は内容自体が示す。

**判定基準**: そのフレーズを削除しても文意が変わらないなら、それはスキャフォールディング。削除する。

### 13. 主張の強度と不確実性

- 確信あること: 断定「〜です」
- 強い意見: 「これは間違いです」
- 推測: 「〜かもしれない」
- 正直な無知: 「実装を確認していないので推測ですが」

**🔴 不確実性は分析的ギャップから生まれなければならない。**

❌ パフォーマティブな不確実性: 「103個が最適な数なのかは全然わかりません」（考えていないだけ）
✅ 分析的な不確実性: 「RFCにはこの挙動の仕様が明記されていないので、私の解釈では...」（調べた上で不明な点）

パフォーマティブな不確実性の見分け方: 「なぜわからないのか」を説明できなければパフォーマティブ。分析的不確実性は「ここまで調べたが、ここから先がわからない」という境界が明確。

### 14. 自己宣伝的フレーミングの禁止

印象的な数字や成果を「すごいでしょ」と見せるのではなく、その数字が生んだ「問題」から入る。

❌ 「orchestrator.pyは7,610行の状態マシンです。『ちょっとしたツール』の域はとっくに超えています」
✅ 「orchestrator.pyが7,000行を超えたあたりで、状態遷移のテストが手に負えなくなりました」

❌ 大袈裟な修辞: 「全ての設計判断の根底にあります」「異常にこだわっています」
✅ 技術的根拠に基づく主張: 「LEADのコンテキストが50トークン増えるごとに、全タスクのコストが約N%上がります」

### 15. 文構造の多様性と口語的なゆるみ

宣言文（XはYです。ZはWです。）の連続を避ける。疑問文、条件分岐、歴史的文脈、読者への問いかけを混ぜる。さらに、各H2セクションに少なくとも1か所、著者の主観的コメント・感情表現・読者への問いかけを入れる。これが欠如すると、論理展開がいかに正確でも「AIが書いたような整理されすぎた文章」に見える。

❌ 「Aはこうです。Bはこうです。つまりCです。」の連続
❌ 全セクションが「〜の問題がありました」「〜を採用しました」という過去形報告のみで終始する
✅ 「Aはこう動きます。ではなぜBではなくAなのか？実は以前Bを試したんですが...」
✅ 「〜なんですよね」「これ、割と厄介で」「正直そこまで深く考えていなかった」のような一人称的コメント
✅ 「すなはち」「なかなか」のような口語的な語感を意図的に混ぜる

**チェック**: 各H2に疑問文または著者の主観的コメントが1か所以上あるか？全セクションが「〜しました」「〜です」のみなら、1か所を問いかけ文または感情コメントに書き直す。

### 16. まとめ・クロージング

- 番号付きの綺麗な要約は禁止
- 残る疑問を提示してもよい
- **クロージングは1つの鋭い文で終わる。** 複数の問題を再列挙する総括段落ではなく、前向きまたは評価的な1文に絞る
- **「開かれた結論」を意識する**: `async-is-not-syntaxsugar.md` の「A. さあ……」のように、記事を綺麗に閉じないことで読者の思考が残る。すべての問題に解決策を提示すると、「完璧に整理されたAI出力」になる。「まだわかっていないこと」「次に何が壊れるか」への誠実な言及を末尾に持ってくる

❌ 「KOLが4人を超えたとき壊れたところから始まって、Phase 4の4分割、stale artifact検出、ドリフト問題と続いてきましたが、どの対処も〜」（全体要約）
❌ すべての問題に解決策が提示され、未解決の問いが何も残らない（綺麗に整理されすぎ）
✅ 「次に何かが壊れるとしたら、まだ境界を引いていないどこかです。」（一文で着地、問いを開いたまま）
✅ 「〜はまだわかっていない。これが次に何かを壊すとしたら〜のタイミングだろう。」

**チェック**: 最終段落を削除したとき、記事が弱くなるか？ならないなら削除する。クロージングに「未解決の問い」が1つ以上あるか？

## 🟢 POLISH

### 17. 本物の脱線（Genuine Tangents）

脱線は新しい情報を導入しなければならない。既に述べた内容を「そういえば」で再訪するのは脱線ではなく構造的欠陥。

**「少し脱線します」と書いた瞬間、そのセクションはカットか統合の候補になる。** 脱線はシームレスに織り込むか、`:::details 余談` ブロックに移すか、削除する。"脱線します" というシグナルを本文に残してはならない。

❌ 擬似脱線: 「そういえばコンテキスト経済の話に戻ると...」（前のセクションの繰り返し）
❌ シグナル付き脱線: 「少し脱線します。〜」（構造的欠陥を宣言している）
✅ 本物の脱線: 別ドメインとの意外な接続、予想外の失敗談、学術論文や先行事例への言及
✅ 脱線が本筋に関係するなら: 主流に統合してシグナルなしで書く

### 18. 不完全さ

- ランダムに分布（均等配置は禁止）
- あるセクションは完璧、別のセクションに3-4個の雑さ
- 未解決のまま残す脱線

### 19. 脚注

- 本文の流れを壊さない技術的補足は脚注に
- 余談ブロック: `:::details 余談` も活用

## ⚠️ ANTI-PATTERNS

| パターン | なぜダメか |
|---|---|
| だ体・体言止めの文末 | 出版ブロッカー。最初の5文を確認すること |
| 概念フレームが末尾にしか出てこない | 軸が冒頭にないと各セクションがバラバラに見える |
| テーゼの3回以上の繰り返し | 読者への不信。冒頭1回+結論1回が上限 |
| 「問題が起きた」だけで実際の出力を示さない | 診断の説得力がゼロになる。症状を1文で見せる |
| 「少し脱線します」シグナル | カットか統合か :::details に移す。シグナルを残さない |
| クロージングが全体要約段落 | 1つの鋭い文で終わる。要約は読者の仕事 |
| 未実装コードを動作コードと同じ体裁で掲載 | 読者を誤解させる。# 未実装 コメントを入れる |
| セクション間の知的連鎖が切れている | 「機能パレード」になる |
| 解決策だけ書いて迷走の過程がない | 「解決策カタログ」になり読者が思考の歩き方を学べない |
| 全セクションが「〜しました」報告のみ | 著者の声調がなくAIらしさが出る。各H2に問いかけ/主観コメントを1か所 |
| すべての問題が綺麗に解決している | 未解決の問いを残さないと「完璧に整理されたAI出力」になる |
| 機能パレード（6+概念を浅く列挙） | 人間は2-3個を深く掘る |
| 印象的な数字の羅列 | 数字は分析の入口であって成果の証明ではない |
| コード例がJSON出力1つだけ | 人間の記事はコードが豊富 |
| パフォーマティブな不確実性 | 「わかりません」だけでは知的誠実さにならない |
| 教育的スキャフォールディング | 「見ていきましょう」は著者の存在を露呈する |
| ボールド10箇所以上 | スライドデッキになる。5-6箇所が上限 |
| 均等に配置された不完全さ | 計算された不完全さは完璧さより不自然 |
| すべてのエピソードに解決策 | 未解決の問いを残す勇気 |
| 均一なヘッジング（全部「〜気がします」） | 確信度は文ごとに変わるべき |
| 綺麗すぎる構成（全スレッド解決、脱線なし） | 知的探索は散らかる |
| 既出内容の擬似脱線 | 脱線は新情報を導入する |
| 大袈裟な修辞（「全ての根底」「異常にこだわる」） | 技術記事は根拠で語る |
| 宣言文の連続（XはY。ZはW。つまりA。） | 文型を混ぜる |
