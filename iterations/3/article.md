KOL向けの投稿指示書をPPTXで自動生成するシステムを作りました。受注書とKOLリストを渡すと、リサーチからデッキ生成まで全自動で回ります。KOLというのはインフルエンサーのことで、案件ごとにキャプション方向性やシーン構成が全部違うので、手作業だと3人分で半日かかっていました。

最終的にこのシステムは8体のAIエージェントが分業する構成になったんですけど、この設計で一番考えたのは「タスクをどう割るか」ではなく**「各エージェントに何を見せるか」**でした。情報の境界を先に設計すると、タスクの分割は勝手に決まります。

## 6人で壊れた

最初は素朴な直列パイプラインでした。Intake、KOL Research、Creative Planning、Asset、Build、Review。各フェーズをsubagent — 親エージェントが起動する子プロセスみたいなもの — として実行して、成果物をファイルで渡していく。3人のKOLなら問題なく動きます。

6人のKOLで回したら5人目と6人目のキャプション方向性が1人目のコピーになりました。「潤い」と「自然由来」をひたすら繰り返しているだけ。

最初はコンテキストウィンドウの問題かなと思って要約ステップを挟んでみたんですけど直らなくて。結局これはattentionの問題でした。6人分のリサーチ結果を全部1つのコンテキストに載せると、前半の情報が支配的になって後半はそのコピーを生成し始めます。

ここで見えてきた構造がこれです。

```
┌────────────────────────────────────────────────────────┐
│ リサーチ:    各KOLの調査は独立       → 並列化できる    │
│ プランニング: 全KOL横断の判断が必要   → ボトルネック   │
│ アセット生成: KOLごとに独立          → 並列化できる    │
└────────────────────────────────────────────────────────┘
```

プランニングだけは全KOLを横断的に見る必要がある。KOL AとKOL Bが似たフォロワー層を持っている場合、意図的にキャプション方向性を変えないといけないから。でもプランニングの出力を `per_kol_packages.json` というKOL別パッケージに分割しておけば、後続フェーズは各KOLを独立して処理できます。

この体験で確信したのは、エージェントへの情報の渡し方を設計しないかぎり、デフォルトでは「全員が全部知っている」状態になるということです。少人数なら動く。スケールした瞬間に壊れる。皆さんの組織でも同じこと起きてませんか？ 5人のスタートアップなら全員が全情報を持っていても回るけど、50人になったら情報の流通設計なしには崩壊する。

KOL数で実行モードも分けました。3人以下なら `classic_small`（Leadエージェントが直接処理もする）、4人以上なら `thin_large`（Leadはゲート判定だけに専念して処理は全部subagentに委譲する）。

## Phase 4を4つに割った話 — ここが本丸です

画像素材の生成で起きた問題から話します。

新しい案件を流したら、キャンペーンカバーは新しく生成されているのに3人目のKOLのシーン画像だけ前の案件のものが残っていました。しかも1つのsubagentで全画像を生成しているから、3人目でコケると4人目以降も全滅。画像がないまま次のフェーズに進んでいました。

これ、3時間くらい追いかけ回しました。なんで3人目だけ画像が古いんだと。ログを見ても「生成成功」って書いてある。んで結局わかったのは、3人目の画像生成が実は失敗していて、前回の実行で残っていたファイルを今回の成果物だと思い込んでいたということです。はー。本当に腹が立ちました。

この経験からAsset生成（Phase 4）を4つに分割することにしました。

```
Phase 4a: Asset Acquisition
  └─ ロゴ・商品参照画像を取得
  └─ → source_assets_manifest.json

Phase 4b: Global Style Gen
  └─ campaign_cover (16:9) + title_cover (21:9) + product_main (9:16)
  └─ → global_assets_manifest.json
  └─ ★ デッキ全体のビジュアルトーンをここで確立

Phase 4c: KOL Creative  ×KOL人数分
  └─ KOL 1名ずつ独立したsubagentをspawn
  └─ 各KOLのシーン画像4枚 + reference image
  └─ → assets_plan_{slug}.json
  └─ ⚠️ 4bの完了が前提条件

Phase 4d: Asset Resolve
  └─ source + global + per-KOL のアセットを統合
  └─ → assets_resolved.json (status=ready)
```

### 4bと4cの依存関係 — ここの設計が一番面白い

この分割で核心になっているのは4bと4cの関係です。

4bが生成するのはキャンペーン全体のビジュアルトーンです。カバー画像、タイトルカバー、商品メイン画像。デッキを開いたときの第一印象を決める画像群。色味、テクスチャ、空気感。4cはそのトーンに合わせて各KOLのシーン画像を生成します。

**4bが完了する前に4cを起動してはいけません。** global_assets_manifest.jsonが存在しない状態でKOLごとのシーン画像を生成するとどうなるか。あるKOLのシーン画像はパステル調で、別のKOLはビビッドで、また別のKOLはモノクロ寄り。1つの案件のデッキなのに、スライドをめくるたびに別の世界観が出てくる。全体の視覚的統一感は4bが確立し、4cはその制約の中で各KOLの個性を表現する。この方向の依存を逆にすると、デッキが破綻します。

4cでKOL 1名ごとに独立したsubagentをspawnするのもポイントです。1人がコケても他に影響しない。6人のKOLで3人目だけ失敗しても、残り5人は正常に生成されています。3人目だけ再実行すればいい。前の構成では、3人目でコケたら4人目以降も全部ない状態でした。

### 4aと4bを分けるかどうか — 正直迷いました

4aと4bを1つのsubagentにまとめてもよさそうに見えたんですよね。ロゴの取得もカバー画像の生成も、どっちもAssetフェーズの話だし。でもやってみて気づいたのは、障害モードが全然違うということです。4aの典型的な失敗は「ロゴが見つからない」「解像度が低い」で、ネットワークアクセスの問題です。4bの典型的な失敗は「プロンプトの品質が悪い」「生成画像のスタイルが意図と違う」で、画像生成AIの問題です。

一緒にすると、ロゴ取得に失敗したときにカバー画像の生成まで巻き添えになります。で、ロゴを修正して再実行するとカバー画像もまた生成し直しになる。分けておけば、失敗したフェーズだけピンポイントで再実行できます。

...とはいえ、2人とか3人のKOLの小さな案件で4a/4b/4c/4dの4回spawnするのはオーバーヘッドではあります。classic_smallモードでは4aと4bをまとめるオプションを入れてもいいのかもしれない。まだやっていないですけど。

### stale artifactとrun_id

前の案件の画像が残る問題は `run_id` で解決しました。各実行にUTCタイムスタンプを振って、Phase 4の各subagentが出力するファイルにも同じrun_idを含める。ゲート判定でrun_idの一致を検証し、不一致ならstale artifactとして無視して再実行する。

```
checkpoint/run_state.json  →  {"run_id": "20260409T120000Z", ...}
各 *_phase_result.json     →  {"run_id": "20260409T120000Z", ...}
Gate判定: run_id不一致 → stale → 該当Phase再実行
```

仕組みとしては何も難しくないです。でもマルチエージェントシステムでは複数のsubagentが非同期にファイルを書き出すから、「このファイルは今回の実行の成果物か前回の残りか」が自明ではありません。人間なら「さっき消したはず」と記憶に頼れますが、エージェントにそんな記憶はないので、ファイル自体に出自を刻印しておく必要があります。

### 遷移ゲート — `status: partial` のまま進んでしまった話

4dが出力する `assets_resolved.json` に `status: ready` と書いてあればBuildに進んでいいんですけど、一度 `status: partial` のまま通過してPPTXの表紙が真っ白になった案件がありました。assets_resolved.jsonの存在だけチェックしていて、中身を見ていなかった。status が ready かつ campaign_cover エントリが global_assets に存在する、この両方を通過条件にしています。

ゲートを厳しくしすぎると進まなくなるんじゃないかと心配になるかもしれません。経験上、ゲートが甘くてゴミが後段に流れるほうが遥かにデバッグコストが高いです。3フェーズ前の入力欠落が原因で最終出力がおかしい、みたいな遡及調査をするくらいなら、ゲートで即座に「campaign_coverがありません」と止まってくれたほうがずっとマシです。

## その他の設計判断

**Readiness Check**: Phase 2→3の遷移で6項目を全件チェックします。intake_packet、kol_targets、product research summary、KOL manifest、KOL research summary、全KOL分のresearchファイル数の一致。1件でも欠落があれば先に進みません。きっかけは、5人中3人分しかリサーチが完了していないのにPhase 3に進んでしまったこと。

**Build→Review fix loop**: Phase 5-7をAgentTeamsで1チームにまとめ、ReviewerがREJECTしたら修正対象に応じてBuilderかVisualizerから再実行する閉ループです。最大10回。通常2-3回で収束します。

```
┌──────────────────────────────────────────┐
│         AgentTeams (Phase 5-6-7)          │
│                                           │
│  Builder ──→ Visualizer ──→ Reviewer      │
│     ↑                          │          │
│     └──── REJECT(data) ────────┘          │
│              REJECT(pptx) ──→ Visualizer  │
│              APPROVE ──→ Export            │
└──────────────────────────────────────────┘
```

:::details Visualizerの「見えないテキスト」問題とcaption-binding-rules
PPTXを開いたらテキストが全部白で見えなかったり、KOL Aのスライドに KOL Bの開示タグが入っていたり。テーマカラー継承の問題は全テキストrunに`030303`をsolidFillで設定することで、caption bindingの問題は`caption-binding-rules.md`をsingle source of truthとしてbuilder spawnプロンプトに含めることで解決しました。
:::

## 全体像

```
Phase 0:   Initialize
Phase 1:   Intake
Phase 1.5: Product Research
Phase 2:   KOL Research
             → Readiness Check（6項目）
Phase 3:   Creative Planning
Phase 4a:  Asset Acquisition
Phase 4b:  Global Style Gen
Phase 4c:  KOL Creative (×KOL人数)
Phase 4d:  Asset Resolve
             → 遷移ゲート（status=ready + campaign_cover）
Phase 5-7: Build / Visualize / Review (AgentTeams, max 10 loop)
Phase 8:   Export
```

Orchestrator自体の責務はモード選択、フェーズ順序、artifact gateの判定だけに絞りました。各Phaseの手順は下位のskill定義と参照ドキュメントに分離してあります。500行を超えたあたりで「このファイルを読むのは人間じゃなくてエージェントなんだから、エージェントにとって読みやすい構造にすべきだ」と気づいて分離しました。

マルチエージェントを「タスクの分割」として設計すると、情報の境界が曖昧なまま残ります。「このエージェントに何を見せるか」を先に決めたほうが結果としてタスク分割も決まる。15人のKOLでPhase 4cを15並列spawnしたらどうなるかはまだ試していません。
