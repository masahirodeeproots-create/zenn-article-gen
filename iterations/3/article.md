3人目のKOLへの指示書が、1人目のコピーになっていました。

キャプションの方向性も、ビジュアルの雰囲気も、ほぼ同じ。KOLごとにフォロワー層もプラットフォームも違うのに。AIに全部任せたのに、人間が疲れたときと同じ劣化パターンが出てる。

これはインフルエンサー向けの投稿指示書をAIエージェントで自動生成しようとしたときの話です。KOL（Key Opinion Leader）に渡す「こう投稿してください」というスライド資料を、8体のAIエージェントが分業して作る仕組み。商品調査からビジュアル素材の生成、スライド組み立て、レビューまで全自動で、手作業だとKOL8人分で丸1日かかっていた仕事が、設計がハマってからは30分で回るようになりました。

ただ、「設計がハマるまで」が本当に長かった。

**甘かった。**

この記事では、「エージェント分ければスケールするでしょ」から始めて踏んだ地雷の話をします。結論を先に言うと、マルチエージェント設計で一番大事だったのは「各エージェントに何をさせるか」ではなく**「誰が何を知るべきか」** — 情報の境界設計でした。

## 「誰が何を知るべきか」が全部を決める

1つの大きなエージェントに全工程を任せると、3人目のKOLあたりから品質が崩れます。LLMはコンテキスト（入力情報）が長くなると後半を軽視する傾向があって、8人分のリサーチと素材生成を詰め込むと後半が前半のコピーになる。

分けるしかない。でも分けた瞬間に「エージェント間で何の情報を渡すか」の問題が出てくる。

全エージェントにプロジェクト全体の情報を渡す？ やってみたらダメでした。情報が多すぎるとエージェントは何が重要か判断できなくなって、関係ない情報に引っ張られて的外れなアウトプットを出す。逆に絞りすぎると、キャンペーンの方向性を知らないデザイン担当がブランドと噛み合わない素材を量産する。

マイクロサービスで全サービスが同じDBを見るのがアンチパターンなのと同じ構造です。ドメイン駆動設計（DDD）で言う「境界づけられたコンテキスト」がAIエージェントにもそのまま当てはまった。

```
全体方針（全員が知るべきこと）
  ├── ブランドのトーン
  ├── キャンペーンの方向性
  └── 品質基準

個別情報（担当者だけが知ればいいこと）
  ├── 特定のKOLのリサーチ結果
  ├── 特定のプラットフォーム要件
  └── 特定のビジュアル素材
```

ここまでは教科書的。問題は、この分離を**実際に機能させる**ときに何が起きるか。

### ベースのスープを先に作れ

8人のKOLに個別の素材を作るとして、キャンペーン全体のトーンは統一したい。最初はそれぞれのKOL担当エージェントに「キャンペーンの方向性」を渡して個別に作らせていました。結果、バラバラ。あるKOLはポップで、別のKOLはシックで。

料理で考えてみてください。8人の料理人にそれぞれ「和食を作ってね」と言って、全員が別々にベースのスープから作り始めたらどうなるか。当然、8種類の違う味になる。

```
[素材取得] → [全体スタイル確定] → [個別KOL素材生成 ×N] → [統合チェック]
                    ↑
              ここにゲートが必要
```

全体スタイルが確定する前に個別制作を始めてはいけない。並列にしていいのは「互いに独立したタスク」だけ。KOL同士の個別制作は並列OK。でも「全体スタイル → 個別KOL」は直列必須。

「全部同時に走らせちゃえ」の誘惑は強い。並列のほうが速いから。でもAIエージェントの文脈だと「全員に全情報渡してるから大丈夫でしょ」という錯覚が生まれやすくて、ここで事故る。

### オーケストレーターの「やらないこと」を決める

ゲート（関所）を管理するのがオーケストレーター。各エージェントの実行順序を決めて、前工程の成果物が揃ったかチェックしてから次に進ませる司令塔です。

最初、オーケストレーターにリサーチの中身までチェックさせようとしました。フォロワー分析が500文字以上あるか、商品との相性スコアが含まれているか。

マイクロマネジメントでした。

KOLが20人になったら20人分のリサーチ内容を全部読んで品質判断する？ オーケストレーター自体のコンテキストが溢れる。最初の「1つのエージェントに全部やらせる」問題の再来です。

```
オーケストレーターの責務:
  ✅ 成果物の存在チェック（8人分全員あるか？）
  ✅ 前提条件の検証（全体スタイルが先に確定してるか？）
  ✅ 次工程への遷移判断
  ❌ 内容の品質判断（レビューエージェントの仕事）
  ❌ 工程内の手順指示（各エージェントの仕事）
```

**「やらないこと」を定義する**のが「やること」を定義するより大事でした。組織論と同じで、マネージャーが部下の仕事のやり方まで口を出し始めると全体が回らなくなる。CIパイプラインも同じ構造。CIサーバーはコードの品質を判定しない、テストが通ったかどうかのゲートだけ管理する。

ゲートで「揃ってない」と判定されたとき、どこまでやり直すか。8人中7人のリサーチが成功して1人だけ失敗したなら、その1人だけ再実行すればいい。「全体の方向性」の工程が失敗したら？ 全部やり直し。再実行の粒度は工程の独立性に依存します。これも情報の境界設計の一部。

情報の境界を引く → 工程の順序が決まる → ゲートの位置が決まる → 再実行の粒度が決まる。全部つながっている。

## 前の案件の亡霊

実運用で一番腹が立ったバグの話。

案件を2回流したとき、前の案件で作ったカバー画像が残っていて、次の案件のエージェントがそれを「最新の成果物」だと思って使った。ファイルの場所は同じ、名前も同じ。でも中身は前の案件のもの。

人間だったら「あ、これ古いやつだ」ってわかる。日付を見る、記憶で判断する、「この画像見覚えあるぞ」って気づく。AIエージェントには「記憶」がない。ファイルがあれば使う。

冷静になってみると、エージェントは悪くない。「どの実行に属する成果物か」の情報がどこにもないのが悪い。

解決策は各成果物に「実行ID」を埋め込むこと。読む側が「今回の実行で作ったものか？」をチェックして、古いIDなら前の案件の残骸だと判断する。

全成果物に忘れず刻印するのは大変なので、ゲートの検査項目に「スコープの一致チェック」を含めました。刻印を忘れたら通過させるのではなく、止める。フェイルセーフ。

これ、「誰が何を知るべきか」の裏返しで、正確には**「何を知ってはいけないか」**の設計。エージェントは前の案件の成果物を「知ってはいけない」。でも黙ってると知ってしまう。ファイルが残ってるから。だから明示的に遮断する仕組みが要る。

## 同じ問題が、違うスケールで再発する

3人のKOLで動いてた設計をそのまま15人案件に流したら、オーケストレーターがタイムアウトしました。

規模に応じて実行モードを切り替えるしかなかった。少数モード（3人以下）はオーケストレーターが処理の一部も直接担当してシンプルに。多数モード（4人以上）は管理に専念して処理は全部委任。どちらのモードでも成果物の形式は統一。後工程はモードの違いを知らない。

で、ここまで作ったとき、素材生成の工程で嫌なことに気づきました。

素材を作る工程が重すぎて、全部1つのエージェントにやらせたら、途中で失敗してもどこで失敗したかわからない。コンテキストが長すぎて後半の品質が落ちる。

嘘でしょ。

最初の問題と同じです。大きなパイプラインの中に小さなパイプラインがある。入れ子構造。どっちの層にも同じ設計原則 — 情報の境界、ゲート、スコープの刻印 — が必要になる。

これに気づいたとき、正直うんざりしました。でも同時に「ああ、同じ解法が使えるのか」という安心感もあった。パイプライン全体で学んだことが、各工程の内部設計にそのまま転用できる。逆に言えば、小さいスケールでうまくいった設計を大きいスケールにも適用できるし、大きいスケールで見つけた問題は小さいスケールでも潜んでいる。

:::details レビューループについて
PPTXのレビュー→修正→再検証のループも設計が必要でした。レビューエージェントが「データの問題」「ビジュアルの問題」とラベルを付け、オーケストレーターがラベルに従ってルーティング。ループ上限は10回、超えたら人間にエスカレーション。ビルド→レビューの閉ループは1つのチームとして囲い込み、内部は自律的に回す設計にしています。
:::

:::details 全体の設計原則
```
┌─ 情報の境界 ──────── 誰が何を知るか
├─ 全体→個別の順序 ── グローバルを先に確定
├─ 成果物ゲート ────── 完全性チェックで遷移判断
├─ スコープの刻印 ─── どの実行に属するか
├─ 品質の閉ループ ─── レビュー→修正→再検証
├─ 責務の絞り込み ─── オーケストレーターは管理のみ
└─ モード切替 ──────── 規模に応じた実行戦略
```
:::

## 暗黙知を設計に落とす

マルチエージェントシステムの設計は「エージェントの能力」の設計じゃなくて「エージェント間の関係性」の設計です。何をできるかより、情報をどう渡し、どこでチェックし、どう修正サイクルを回すかのほうがはるかに重要だった。

人間のチームワークでは暗黙知として機能していること — 情報の境界、品質チェック、責任の委譲 — を、全部明示的に設計しないといけない。「あうんの呼吸」はAIには通じない。だから大変だけど、逆に設計が全部見える。人間のチームだと「なんとなくうまくいってる」の内実を、全部データ構造とプロトコルに落とし込む作業。組織設計の教科書を読むより、AIエージェント8体を協調させるほうが組織論を深く理解できるかもしれない、と半分本気で思っています。

50人のKOL案件が来たらこの設計で本当にスケールするのか。ゲートが多すぎて処理時間が爆発しないか。まだ確信はありません。ただ、今のところ8人案件で30分、15人案件で1時間弱。手作業の10分の1以下の時間で、品質は人間が作ったものと同等かそれ以上。

まだ山ほど壊れるだろうけど、壊れ方のパターンは見えてきました。
