3人目のKOLのキャプション方向性が、1人目のコピーになっていました。

5人目のビジュアル指示は2人目とほぼ同じ。8人目に至ってはもう1人目の使い回し。8人のKOLにカスタマイズした投稿指示書をPPTXで自動生成するシステムを作っていて、最初は1つの大きなエージェントに全情報を詰め込んでいたんですけど、LLMは長いコンテキストの中で情報を平均化していきます。後半のKOLが前半の繰り返しになる。

エージェント分ければ解決するだろう。調査、デザイン、ビルド、レビュー。

分けた瞬間に本当の問題が始まりました。この記事はその「本当の問題」の話です。

## ベースのスープを先に作れ

8人のKOL担当エージェントにそれぞれ「キャンペーンの方向性」をテキストで渡して、個別にビジュアル素材を作らせました。

あるKOLはポップ。別のKOLはシック。もう1人は謎にレトロ。同じキャンペーンに見えない。

ラーメンで言うとですね、ベースのスープを先に作ってからトッピングを変えるべきところを、8人の料理人に「豚骨っぽい味で」って言葉だけ渡して各自にスープから作らせたんです。そりゃバラバラになりますよ。

```
❌ やったこと:
  [方向性テキスト] → [KOL担当A] → ポップな素材
                   → [KOL担当B] → シックな素材
                   → [KOL担当C] → なぜかレトロ

✅ こうすべきだった:
  [素材取得] → [全体スタイル確定] → [KOL個別 ×N] → [統合チェック]
```

全体スタイルが確定する**前に**個別制作を始めちゃダメ。

ここで厄介なのが並列化の誘惑です。8人のKOLは互いに独立しているから「全員分を同時に走らせれば速い」。実際KOL間の並列化は正しいんですけど、「全体スタイル確定 → 個別KOL制作」という上から下への依存関係は直列で守らないといけない。「まあ大丈夫でしょ」で全部同時に走らせたら案の定壊れました。

知識としては知っていたんですよ、依存関係のあるタスクの並列実行がバグの温床だということは。知っていて踏んだ。

この経験から工程間に**ゲート**を置くことにしました。前の工程の出力が条件を満たしているか確認してから後工程に進む。「全体スタイルの成果物が存在する」ことが個別制作開始の前提条件になります。

ゲートで何を見るかも最初は迷いました。出力の中身の品質を検査しようとしたんですけど、たとえば調査エージェントが「このKOLは20代女性に人気」って出してきて、正しいかどうかなんて次の工程にはわからない。

結局、ゲートは中身を見ないようにしました。見るのは**成果物の存在と完全性**だけ。8人分のリサーチ結果が全部あるか。必須項目が埋まっているか。3人分しかないのに次に進もうとしていないか。中身の正しさはレビューエージェントに任せる。

ここまでは、まあ、教科書的な話です。DDDの境界づけられたコンテキストをエージェントに適用して、工程間にゲートを置く。本に書いてある範囲。

問題は次に起きました。

## 前の案件のカバー画像がそのまま出てきた

これが一番キツかった。

案件を2回流したとき、前の案件の成果物が残っていて、新しい案件のエージェントがそれを「最新の成果物」として使いました。前の案件で作ったカバー画像が新しい案件のPPTXにそのまま載っている。

納品直前に気づいて肝が冷えました。

ファイルの場所は同じ。名前も同じ。中身だけが古い。人間なら「あ、これ前のだ」って気づきますけど、エージェントにはその判断ができません。「ここにファイルがあるからこれが最新だろう」としか考えられない。

最初の対策。「前の案件のファイルを消してから走らせる」。消し忘れで2回目もやらかしました。3回目は消したのに一部の中間成果物が残っていてまた混入。

3案件目でようやく気づきました。人間の注意力に依存する設計は3回で破綻するんだなと。当たり前のことなんですけど、当たり前のことに到達するのに3案件かかった。

笑えなかったのは、この問題が前のセクションで作ったゲートを**すり抜ける**からです。ゲートは「成果物が存在するか」をチェックする。前の案件の成果物は存在する。名前も正しい。ゲートは通してしまう。

存在チェックだけじゃ足りない。「この成果物がどの実行に属するか」の情報が必要なんです。

各成果物に実行IDを刻印することにしました。ゲートの検査項目が2つになります。

1. 成果物が存在するか
2. 成果物の実行IDが今回の実行と一致するか

この2番目がないと前の案件の亡霊がゲートをすり抜ける。

ここで面白いのが、最初は「工程の順序を守る問題」と「古い成果物の混入問題」を全然別の課題だと思っていたんですけど、解決策のレイヤーでは同じゲートの検査項目に合流したことです。ゲートが見るのは「存在する？」と「今回のもの？」の2つだけ。順序の問題も所有の問題もこの2つで解ける。

別々に考えていた問題が同じ仕組みの上で合流する。設計がちゃんと一貫してるんだなと、ここで初めて安心しました。

「全部の成果物にIDを付けるなんて忘れたら？」。もっともな懸念で、だからゲートをフェイルセーフにしています。実行IDがない成果物はゲートが止める。通過させるんじゃなくて止める。忘れた側が罰せられる設計です。

:::details なぜ「毎回全消し」ではダメだったか
「毎回全ファイル消してクリーンにすれば」は最初に試しました。素材取得ステップで数百MBのファイルをダウンロードするのに毎回30分以上かかるんですよね。キャッシュを残したい。でも選択的にキャッシュを残すと「どれがキャッシュ安全か」の判定が必要になって、結局スコープ管理と同じ問題に帰着します。
:::

## 同じ構造が入れ子で出てくる

素材制作の工程が重すぎて1つのエージェントに全部やらせたら、途中で失敗したときにどこで壊れたかわからない。最初からやり直しです。

待ってください。これ、記事の冒頭で「1つのエージェントに全部やらせたら品質が落ちた」のと同じ問題じゃないですか。粒度が違うだけで構造が同じ。

```
パイプライン全体:
  [調査] →ゲート→ [計画] →ゲート→ [制作] →ゲート→ [レビュー]

「制作」の内部にも同じ構造:
  [素材取得] →ゲート→ [全体トーン] →ゲート→ [KOL個別×N] →ゲート→ [統合]
```

大きなパイプラインの中に小さなパイプラインがある。情報の境界もゲートもスコープの刻印も、全部再帰的に必要になる。同じ原則の適用先が増えるだけで、新しいルールは増えない。これがわかったときは少し嬉しかったです。

ゲートの密度に最適解はまだ見えていません。全サブステップにゲートを入れると安全ですけど遅い。省略して2回事故りました。今は全部入れています。過剰設計かもしれない。でも事故るよりマシです。

:::details レビューループとモード切替

残り2つの設計判断を手短に。

**レビューの閉ループ**: PPTXのレビューは修正→再チェックのループです。データの問題ならビルド担当へ、ビジュアルの問題なら見た目担当へ差し戻します。レビューエージェントが分類ラベルを付けて、オーケストレーターはラベルでルーティングするだけ。ビルドからレビューまでを1チームに囲い込んで裁量を渡し、上限超えで人間にエスカレーション。

**モード切替**: KOL3人と30人で同じ設計は使えません。少数ならオーケストレーターが処理の一部を直接担当、多数なら管理に専念してサマリーだけ読む。成果物のインターフェースは統一。正直、少数モードはアーキテクチャとして美しくないです。最初から多数モードで統一しておけばよかったのかもしれませんが、3人のKOLのために大仰な委任構造を組むのはやりすぎに感じて。この判断が正しかったかは今でもわかりません。
:::

## エージェントを分けるより大事なこと

マルチエージェントの設計をやってみて一番の学びは、「どう分けるか」より「分けた後にどう繋ぐか」のほうが圧倒的に難しかったということです。

人間のチームだと暗黙的に処理されていること — 「これ前のやつだよ」「先に全体決めよう」「全員分揃った？」 — を全部、検査可能なプロトコルとして書き下す必要がある。面倒ですけど、やってみると人間のチームって実はものすごく複雑な調整を無意識にやってたんだなと逆に感心します。

エージェントシステムでゲートを設計するとき、「存在するか」と「今回のものか」の2つだけ考えてみてください。この2つで驚くほど多くの問題が解けます。少なくとも、前の案件の亡霊に悩まされることはなくなります。
