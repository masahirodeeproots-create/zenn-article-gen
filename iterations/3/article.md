KOL向けの投稿指示書をPPTXで自動生成するシステムを作りました。受注書とKOLリストを渡すと、リサーチからデッキ生成まで全自動で回ります。KOLはインフルエンサーのことで、案件ごとにキャプション方向性やシーン構成が全部違うので手作業だと3人分で半日です。

最終的に8体のAIエージェントが分業する構成になったんですけど、この設計で一番考えたのは「タスクをどう割るか」ではなく**「各エージェントに何を見せるか」**でした。情報の境界を先に設計すると、タスクの分割は勝手に決まります。

## 6人で壊れた

最初は素朴な直列パイプラインでした。Intake、KOL Research、Creative Planning、Asset、Build、Review。各フェーズをsubagentとして実行して成果物をファイルで渡していく。subagentというのは親エージェントが起動する子プロセスみたいなもので、独立したコンテキストで動きます。3人のKOLなら問題なく動きます。

6人のKOLで回したら5人目と6人目のキャプション方向性が1人目のコピーになりました。「潤い」と「自然由来」をひたすら繰り返しているだけ。6人分のリサーチ結果を全部1つのコンテキストに載せると、前半の情報が支配的になって後半はそのコピーを生成し始める。LLMのコンテキストウィンドウが長くなるとattentionが偏るという構造的な限界です。

皆さんの組織でも似たこと起きてませんか？ 5人のスタートアップなら全員が全情報を持っていても回る。50人になったら情報の流通設計なしには崩壊する。AIエージェントでもまったく同じでした。

```
┌────────────────────────────────────────────────────────┐
│ リサーチ:    各KOLの調査は独立       → 並列化できる    │
│ プランニング: 全KOL横断の判断が必要   → ボトルネック   │
│ アセット生成: KOLごとに独立          → 並列化できる    │
└────────────────────────────────────────────────────────┘
```

プランニングだけは全KOLを横断的に見る必要がある。似たフォロワー層を持つKOLがいる場合、意図的にキャプション方向性を変えないといけないから。でもプランニングの出力をKOL別パッケージに分割しておけば、後続フェーズは各KOLを独立して処理できます。KOL数で実行モードも分けて、3人以下ならLeadエージェントが直接処理もする軽量モード、4人以上ならLeadはゲート判定だけに専念して処理は全部subagentに委譲するモード。

## Phase 4を4つに割った話 — ここが本丸です

画像素材の生成で起きた問題から。

新しい案件を流したら、キャンペーンカバーは新しいのに3人目のKOLのシーン画像だけ前の案件のものが残っていました。しかも1つのsubagentで全画像を生成しているから、3人目でコケると4人目以降も全滅。画像がないまま次のフェーズに進んでいました。

これ3時間くらい追いかけ回したんですよ。なんで3人目だけ画像が古いんだと。ログには「生成成功」って書いてある。んで結局わかったのは、3人目の画像生成が失敗していて、前回の実行で残っていたファイルを今回の成果物だと思い込んでたということです。はー。マジで腹が立ちました。というか自分の設計が悪い。

この経験からAsset生成を4つのステップに分割しました。

```
[素材取得] → [全体スタイル確定] → [KOL個別 ×N] → [統合チェック]
               ↑ ゲート            ↑ 並列OK
```

### 4bと4cの依存関係 — ベースのスープがないとトッピングだけでは

この分割の核心は全体スタイル確定（4b）とKOL個別（4c）の関係です。

4bが生成するのはキャンペーン全体のビジュアルトーン。カバー画像、タイトルカバー、商品メイン画像。デッキを開いたときの第一印象を決める画像群です。色味、テクスチャ、空気感。4cはそのトーンの制約の中で各KOLのシーン画像を生成する。

**4bが完了する前に4cを起動してはいけません。**

ラーメンで考えるとわかりやすくて、ベースのスープを先に作らないとトッピングだけ変えても店の味にならないですよね。4bのグローバルスタイルがベースのスープで、4cのKOL個別画像がトッピング。スープなしにトッピングだけ作ったら、あるKOLはパステル調で別のKOLはビビッドで、1つの案件なのにスライドをめくるたびに別の世界観が出てくる。

4cではKOL 1名ごとに独立したsubagentをspawnします。1人がコケても他に影響しない。6人のKOLで3人目だけ失敗しても残り5人は正常に生成されています。3人目だけ再実行すればいい。

### 4aと4bを分けるかどうか

ここは正直迷いました。ロゴの取得もカバー画像の生成も、どっちもAssetフェーズの話だし、1つのsubagentにまとめてもよさそうに見える。

でもやってみて気づいたのは障害モードが全然違うということです。素材取得（4a）の典型的な失敗は「ロゴが見つからない」「解像度が低い」でネットワークアクセスの問題。全体スタイル確定（4b）の典型的な失敗は「プロンプトの品質が悪い」「生成画像のスタイルが意図と違う」で画像生成AIの問題。一緒にするとロゴ取得に失敗したときにカバー画像の生成まで巻き添えになって、ロゴだけ修正して再実行してもカバー画像もまた生成し直しになる。

...とはいえ、2〜3人のKOLの小さな案件で4回subagentをspawnするのはオーバーヘッドかもしれません。正直まとめてもよかったかも。まだ結論が出ていないところです。

### stale artifactとrun_id

前の案件のファイルが残る問題は、各実行にUTCタイムスタンプのIDを振って成果物にも同じIDを含める、ゲートでIDの一致を検証する、不一致ならstale（古い残骸）として無視して再実行する。それだけです。

```
checkpoint/run_state.json  →  {"run_id": "20260409T120000Z", ...}
各 *_phase_result.json     →  {"run_id": "20260409T120000Z", ...}
Gate判定: run_id不一致 → stale → 該当Phase再実行
```

データベースのMVCCと同じ発想ですね。各レコードが「どのトランザクションで書かれたか」を持つ仕組み。仕組みとしては何も難しくないんですけど、マルチエージェントでは複数のsubagentが非同期にファイルを書き出すから、「このファイルは今回の実行の成果物か前回の残りか」が自明ではありません。ファイル自体に出自を刻印しておく必要があります。

### 遷移ゲートの設計思想

4d（統合チェック）の出力マニフェストに `status: ready` と書いてあればBuildに進むんですけど、一度 `status: partial` のまま通過してPPTXの表紙が真っ白になった案件がありました。マニフェストの存在だけチェックして中身を見ていなかった。

statusが ready かつ campaign_coverが存在する。この両方を通過条件にしています。ゲートが甘くてゴミが後段に流れるコストと、ゲートが厳しくて止まるコスト、10倍くらい違います。

## その他の設計判断

Phase 4以外は手短に。

**Readiness Check**: KOLリサーチから次のフェーズに進む前に6項目を全件チェック。intake_packet、kol_targets、product research summary、KOL manifest、KOL research summary、全KOL分のresearchファイル数の一致。1件でも欠落があれば先に進みません。5人中3人分しかリサーチが完了していないのにPhase 3に進んでしまい、3人目のプランが空虚だったのがきっかけです。

**Build→Review fix loop**: Phase 5-7をAgentTeamsで1チームにまとめて閉ループを回します。AgentTeamsはClaude Codeの仕組みでチーム内のエージェント同士が協調できる。ReviewerがREJECTしたら修正対象に応じてBuilderかVisualizerから再実行。最大10回ループの安全弁付き。

```
┌──────────────────────────────────────────┐
│         AgentTeams (Phase 5-6-7)          │
│                                           │
│  Builder ──→ Visualizer ──→ Reviewer      │
│     ↑                          │          │
│     └──── REJECT(data) ────────┘          │
│              REJECT(pptx) ──→ Visualizer  │
│              APPROVE ──→ Export            │
└──────────────────────────────────────────┘
```

:::details Visualizerの色問題とcaption binding
PPTXを開いたらテキストが全部白で見えなかったり、KOL Aのスライドに KOL Bの開示タグが入っていたり。テーマカラー継承の問題は全テキストrunに`030303`をsolidFillで設定して解決。caption bindingの問題は`caption-binding-rules.md`をsingle source of truthとしてbuilder spawnプロンプトに必ず含める運用で解決しました。
:::

## 全体像

```
Phase 0:   Initialize
Phase 1:   Intake
Phase 1.5: Product Research
Phase 2:   KOL Research
             → Readiness Check（6項目）
Phase 3:   Creative Planning
Phase 4a:  素材取得
Phase 4b:  全体スタイル確定
Phase 4c:  KOL個別 (×KOL人数)
Phase 4d:  統合チェック
             → 遷移ゲート（status=ready + campaign_cover）
Phase 5-7: Build / Visualize / Review (AgentTeams, max 10)
Phase 8:   Export
```

Orchestratorの責務はモード選択、フェーズ順序、ゲート判定だけです。各Phaseの手順は下位のskill定義と参照ドキュメントに分離してあります。500行を超えたあたりで「これモノリスだろ」と気づいて。CIサーバーがテストの中身を知らないのと同じで、オーケストレーターは「何が入って何が出るか」だけ知っていればいい。

マルチエージェントを「タスクの分割」として設計すると情報の境界が曖昧なまま残ります。「このエージェントに何を見せるか」を先に決めたほうが、タスク分割は自然に決まる。15人のKOLで4cを15並列spawnしたらどうなるかはまだ試していません。
