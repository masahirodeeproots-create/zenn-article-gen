3人目のKOLのキャプション方向性が、1人目のコピーになっていました。5人目のビジュアル指示は2人目とほぼ同じ。

8人のKOLにカスタマイズした投稿指示書をPPTXで自動生成するシステムを作っていて、最初は1つの大きなエージェントに全部任せていました。コンテキストウィンドウに全情報を詰め込めば受け渡しの設計がいらない。合理的な判断だったはずが、LLMは長いコンテキストの中で情報を平均化していく。後半のKOLが前半の繰り返しになる。

エージェント分ければ解決するだろうと思って分けました。調査、デザイン、ビルド、レビュー。

**分けた瞬間に本当の問題が始まりました。**

エージェントの分割は難しくないんです。難しいのは分けた後の「境界」の設計で、しかもこの境界が複数の種類に分岐して、最終的に全部同じ仕組みに合流する。この記事はその合流の話をします。

## 全体のトーンを先に決めろ

8人のKOL担当エージェントにそれぞれ「キャンペーンの方向性」をテキストで渡して、個別にビジュアル素材を作らせました。

あるKOLはポップ。別のKOLはシック。同じキャンペーンに見えない。

ラーメンに例えると、ベースのスープを先に作ってからトッピングを変えるべきところを、8人に「豚骨っぽい味で」って言葉だけ渡して各自にスープから作らせた状態です。

```
[素材取得] → [全体スタイル確定] → [個別KOL制作 ×N] → [統合チェック]
```

全体スタイルが確定する前に個別制作を始めてはいけない。8人のKOLは互いに独立しているから並列化したくなるんですけど、「全体スタイル → 個別KOL」の依存は直列で守る必要があります。これをサボって全部同時に走らせたら不整合が出ました。

順序の制約を守るために工程間にゲートを置きます。前の工程の出力が基準を満たしているか確認してから次に進む。

ゲートで何を検査するか。最初は中身の品質を見ようとしたんですけど、調査エージェントの出力が正しいかどうかなんて次の工程には判断できません。ゲートが検査すべきは**成果物の存在と完全性**です。8人分のリサーチ結果が全部あるか。必須項目が埋まっているか。3人分しかないのに次に進もうとしていないか。オーケストレーターはマネージャーと同じで、中身の判定はせず「必要なものが揃ったか」だけ見る。

ここまでは割と素直な設計です。DDDの境界づけられたコンテキストをエージェントに適用して、工程間にゲートを置く。教科書に書いてある範囲。

問題は次に起きました。

## 前の案件のカバー画像が混入した

案件を2回流したんですよ。そしたら前の案件の成果物が残っていて、新しい案件のエージェントがそれを「最新の成果物」として使った。

前の案件で作ったカバー画像を、新しい案件でそのまま採用。ファイルの場所は同じ、名前も同じ、中身だけが古い。人間なら「あ、これ前のやつだ」とわかりますけど、エージェントにはわからない。

最初の対策は「前の案件のファイルを消してから走らせる」でした。消し忘れで2回目もやらかしました。3回目は消したのに一部の中間成果物が残っていてまた混入。人間の注意力に依存する設計は3回で破綻するということを身をもって学びました。

冗談みたいな話なんですけど、ここで笑えなかったのは、この問題が「時間の境界」のゲートをすり抜けるからです。ゲートは「成果物が存在するか」をチェックする。前の案件の成果物は存在する。名前も正しい。ゲートは通過させてしまう。

つまり、**成果物の「存在」だけでは足りない。「どの実行に属するか」の情報が必要**なんです。

データベースにMVCCという仕組みがあります。各行がどのトランザクションで書かれたかを持っていて、読み取り時にチェックする。古いトランザクションの行が見えない仕組みです。これと同じ発想で、各成果物に実行IDを刻印しました。

ゲートの検査項目が2つに統合されます:

1. **成果物が存在するか**（時間の境界のチェック）
2. **成果物のスコープが現在の実行と一致するか**（所有の境界のチェック）

これが合流です。最初は「工程の順序を守る」問題と「古い成果物が混入する」問題を別々の課題だと思っていました。でも解決策のレイヤーでは同じゲートの検査項目として統合される。

「全部の成果物にスコープを付けるなんて忘れたら終わりでしょ」という懸念はもっともで、だからゲートをフェイルセーフにしています。スコープ情報がない成果物はゲートが止める。通過させるんじゃなくて止める方向に倒す。忘れた側が罰せられる設計。

:::details 余談：なぜ「消す」じゃダメだったか
「毎回全ファイル消してクリーンにすればいいのでは」という発想は最初に試しました。問題は中間成果物のキャッシュ。素材取得のステップで大きなファイルをダウンロードするんですけど、毎回ゼロからやると時間がかかりすぎる。かといって選択的にキャッシュを残すと「どれが安全にキャッシュできるか」の判定が必要になって、結局スコープ管理と同じ問題に帰着します。
:::

## 大きすぎる工程の内部にも同じ構造が出てくる

素材制作の工程が重すぎました。素材集めて、全体トーンを決めて、KOL個別の素材を作って、全部揃ってるか確認する。これを1つのエージェントにやらせたら途中で失敗してどこで壊れたかわからない。最初からやり直し。

あれ、これ最初に「1人に全部任せたら品質が落ちた」のと同じ問題じゃないですか。粒度が違うだけ。

```
パイプライン全体:
  [調査] →ゲート→ [計画] →ゲート→ [制作] →ゲート→ [レビュー]

「制作」の内部:
  [素材取得] →ゲート→ [全体トーン] →ゲート→ [KOL個別×N] →ゲート→ [統合]
```

大きなパイプラインの中に小さなパイプラインがある。情報の境界もゲートもスコープの刻印も全部要る。設計原則が再帰的に適用される。

ゲートの検査密度に最適解はまだ見えていません。増やすと安全だけど遅い、減らすと速いけど事故る。「制作」内部のサブステップに全部ゲートを入れるのは過剰設計なのかもしれないし、必要なのかもしれない。今のところ全部入れています。

## レビューループとスケール

残り2つの設計判断を手短に。

PPTXのレビューは修正→再チェックのループになります。問題はルーティング。データの問題ならビルド担当へ、ビジュアルの問題なら見た目担当へ差し戻す。レビューエージェントが分類ラベルを付けて、オーケストレーターはそのラベルでルーティングするだけです。中身は見ない。ビルドからレビューまでは1つのチームに囲い込んで内部ループの裁量を渡し、上限を超えたら人間にエスカレーション。

KOLが3人のときと30人のときで同じ設計は使えません。3人ならオーケストレーターが全部見られますけど、30人だとコンテキストが溢れる。入力の規模に応じて実行モードを切り替えます。少数なら直接処理、多数なら管理に専念。成果物のインターフェースはどちらのモードでも同じにする。

正直、少数モードはオーケストレーターに処理を持たせているのでアーキテクチャとしては美しくないです。3人のKOLのために大仰な委任構造を組むのはオーバーエンジニアリングだと判断したんですけど、結局30人対応のときにリファクタが必要になりました。最初から多数モードで統一しておけばよかったのか、いまだにわかりません。

## 暗黙知をプロトコルにする作業

```
┌─ 情報の境界 ──── 誰が何を知るか
├─ 時間の境界 ──── 全体→個別の順序 ─┐
├─ 所有の境界 ──── 成果物のスコープ ─┤→ ゲートで合流
├─ 品質ループ ──── レビュー→修正     │
└─ モード切替 ──── 規模に応じた戦略  ┘
```

マルチエージェント設計で一番面白かったのは、見た目が別々の問題に見えるものが同じ仕組みに収束していく過程です。「順序を守る」と「古い成果物を排除する」は全然違う問題に見えて、ゲートの検査項目として合流する。

人間のチームだとこの手の調整は暗黙的に処理されています。「これ前のやつだよね」と誰かが気づく。「先に全体の方向決めようよ」と誰かが言う。AIエージェントではそれを全部、検査可能なプロトコルとして書き下す必要がある。

読者の方がマルチエージェントシステムを作るとき、エージェントを分けた途端に「受け渡し」の問題にぶつかると思います。そのとき、問題の種類を分けるよりも、解決の仕組みがどこで合流するかに注目すると、設計がシンプルになるかもしれません。少なくとも筆者の場合はそうでした。
