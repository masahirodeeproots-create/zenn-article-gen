深夜0時、AIコストの確認画面を開いたら¥4,200が表示されていました。修正ループが23回まわっていて、スコアは9回目も10回目も7.8のまま動かない。何も変わらないのにループだけが回り続けていました。

SNSインフルエンサー（KOL）向けの投稿指示書を、10体のClaudeエージェントが連携して全自動生成するシステムを作っていました。受注書とKOLリストを渡すと、トーン調査・クリエイティブ設計・画像生成・スライド組版・レビューを人手なしで一括実行する仕組みで、KOL10名の処理が手作業の45分から13分まで短縮できる見込みでした。

その見込みは最終的には達成しましたが、5ラウンドの設計を通じて同じ根っこの問いに何度も戻りました。コンテキスト爆発・NGワード事故・ループ暴走・並列化競合——どの失敗も、**「誰が何を書き、誰が何を読むか」の設計が曖昧なところで起きていました。**

---

## 崩壊は同じ場所から

最初の構成は単純でした。「1つのAIに全部やらせる」。KOL10人分をまとめて渡して、スライドを作ってもらう。

8人目で崩壊しました。KOL1は「ピンクトーン×ナチュラル系」の設定でした。8人目を処理するころに、KOL1のキャプションに「クールホワイト系スタジオ演出」が混入していました。ファン層が「等身大ナチュラル系」なのに、正反対のディレクションです。コンテキストウィンドウの物理的制約の前では、エージェントの性能は関係ありません。1体に10の仕事をさせると、後半で前半を忘れます。

フェーズ分割した後に別の事故が起きました。5フェーズ後のキャプション生成で、調査フェーズで記録したNGワード——「医師が認めた」「皮膚科推薦」という薬機法アウトな表現——が3箇所出てきたんです。内容確認して見つけたのではなく、先方への送付後にクライアントから指摘が来て気づいた形でした。送付ボタンを押した直後に確認メールが来たときの感覚は、よく覚えています。

それが、¥4,200事故です。

どれも「情報がどこかで途切れた」失敗でした。エージェントを増やしても、情報の伝達を設計していなければ劣化します。

## 成果物を契約書にする

フェーズ分割で最初に徹底したのは「1エージェント1成果物」の原則です。各エージェントは1つのファイルを出力し、次のエージェントはそのファイルを読む。コンテキスト継承ではなく、ファイルで繋ぐ。

```
[KOL調査]      → kol_research_summary.md
[プランニング]  → per_kol_packages.json
[画像生成]     → assets_plan_{slug}.json × KOL数
[アセット統合]  → assets_resolved.json
[スライド組版]  → {project}.pptx
```

この構造自体は自然でしたが、問題は「どのファイルのどのフィールドを、誰が書くか」を最初に決めていなかったことです。後から「このフィールドは誰が書くの？」という問いが何度も頻発しました。これが地味に繰り返し出てきて、正直うんざりしました。

NGワード事故の原因を遡ると、NGワードは `per_kol_packages.json` の `caption_direction` フィールドに書かれていたのに、キャプション生成エージェントのプロンプトがそのフィールドを読む指示になっていなかったことでした。情報はファイルの中にあった。読まれなかっただけです。

「ファイルを情報の置き場所として設計する」から「エージェント間の**契約書**として設計する」への転換が効きました。「このフィールドはAエージェントが書き、Bエージェントが読む」という所有関係を先に決める。キャプションエージェントは `caption_direction` を必ず読む契約にする。

契約が機能しているかを確認するため、フェーズゲートを設けました。各フェーズは「必須成果物ファイルが存在し、かつ `status: ready` か」を条件に次フェーズへ進みます。`warnings` に何か入っていても進まない。このゲートを設けてからNGワード混入事故はゼロになりました。前フェーズの成果物が不完全なまま後段が走ることを根絶できます。

---

同じ構造の問題がキャンペーン統一感とKOL個別化の両立でも出ました。

全KOL分を1つのプランニングエージェントに渡したら、KOL3人分で約52,000トークン。途中から前半KOLの `caption_direction` に書いたNGワードが消えていました。1体に10人分を持たせると後半で前半を忘れる——フェーズを分割した後にも、同じ問題が形を変えて出てくるのが興味深いというか、見ていて疲弊しました。

解決策は「先にキャンペーン全体のスタイルアンカーを1ファイルに固定し、KOL個別エージェントはそのアンカーを制約として個別化する」という時間的分離です。

```
Phase A: キャンペーン共通スタイルを global_assets_manifest.json に固定
            ↓（ブランドカラー・NG要素・トーン方針）

Phase B: KOL別クリエイティブ（並列実行可）
   各KOL: 「アンカーの範囲内でKOLの個性を最大化」
```

グローバル一貫性は共有ファイルで保証し、個性はエージェントの裁量に委任する。「誰がアンカーファイルを書き、誰が読むか」を先に決めれば、この構造は自然に機能します。「グローバルな制約と個別の裁量」を同一フェーズで解こうとして詰まった経験はないでしょうか。おそらく、時間的な分離が効きます。

## ループは「変化しているか」で止める

¥4,200事故の停止条件は「スコアが9.0以上か」だけでした。スコアが7.8のまま動かなくても、「まだ9.0に達していない」からループが継続する。単純に見えますが、設計しているときは意外と見えにくい判断ミスです。

「絶対スコアが閾値を超えたか」に加えて「変化しているか」を停止条件に加えることで解決しました。

```
① スコア ≥ 9.0 が 2連続  → 良い水準で安定した → STOP
② スコア変化 ±0.5 以内が 3連続 → 改善が止まった → STOP
③ max 10回              → どんな場合もここで打ち切り
```

②の停滞検知があれば、23回のループが6回目で止まります。コスト試算では¥1,700前後まで下がります。「上がり続けているか」という変化パターンの監視は、コスト最適化でもあります。

品質の上限が素材品質で決まる、というのは後から気づいた設計の失点でした。生成ループを回し続けて7点台で頭打ちになって、素材フェーズ（調査・設計）に戻って品質を上げたら一気に8.5まで跳ね上がりました。下流をいくら最適化しても、上流が弱ければ上限が低い。素材PDCA → 生成PDCAの2段階構造は、この経験から出てきました。正直、最初からそうすべきだったと思います。

## 並列化の前に依存グラフを書く

グローバルスタイル生成とKOL調査を「どちらも前フェーズに依存しない」と判断して並列起動したところ、KOL-Bのシーン画像のブランドカラーが `#1A2B3C`（紺）になりました。正しいカラーは `#F28C38`（オレンジ）です。送付後30分で「カラーが全然違います」と連絡が来ました。

KOL-BエージェントはグローバルスタイルのJSONを読んでからシーン生成する設計でした。そのファイルがまだ存在しない状態で起動して、ブランドカラーをデフォルト値で補完していた。「前フェーズの結果が必要か？」だけで並列化を判断していたのが間違いで、「依存ファイルが完成しているか？」まで確認する必要がありました。

```
[入力]
  ├─→ [KOL調査]  ─────────────────────────────────────┐
  └─→ [商品調査] ─→ [統合プランニング] ─→ [グローバルスタイル生成]
                                                  ↓ ← 完了後に
                       ┌─── [KOL-A クリエイティブ] ─→ [A画像生成]
                       ├─── [KOL-B クリエイティブ] ─→ [B画像生成]
                       └─── [KOL-C クリエイティブ] ─→ [C画像生成]
                                                        ↓
                                               [アセット統合] → [組版] → [出力]
```

並列にできるのは2箇所だけです。KOL調査と商品調査（どちらも入力にのみ依存）と、KOL別クリエイティブ（グローバルスタイル完了後）。グローバルスタイルは間に挟まって、並列にできません。

もう一つ、`run_id` でフェーズ間の成果物整合性を確認するゲートも追加しました。以前の実行で生成された古いアーティファクトを掴んで誤動作するケースが何度かあったためです。各フェーズの出力JSONに `run_id` を刻印し、次フェーズのゲートでIDが一致しているか照合します。ステールアーティファクト問題はこれで消えました。

## まだ答えが出ていないこと

5ラウンドを通じて、ほとんどの問題が「誰が何を書き、誰が何を読むか」の設計から起きていたと気づきました。コンテキスト爆発・情報劣化・ループ暴走・並列化競合——これらは全て、この問いへの答えが曖昧なまま実装を進めたときに起きていました。「1エージェント1成果物」という原則が、実は並列化競合の回避も同時に解いていたことに後から気づいたのは面白かったです。

ただ、一つだけ設計で回避しているだけで根本から解いていない問いが残っています。

**並列エージェントが同一ファイルに書こうとした場合、どうなるか？**

今回はKOL-A/B/Cがそれぞれ異なるファイルを出力することで競合を回避しました。でも「複数エージェントが同一サマリーファイルに追記する」ケースは、設計で禁止しているだけで仕組みとして防いでいません。ファイルロック・書き込みキューイング・マージ戦略のどれかが必要になるはずで、今はそのどれも実装していません。

自分の手元のマルチエージェントシステムではこの問題をどう扱っているか、コメントで教えてもらえたら嬉しいです。
