3人目のKOLのキャプション方向性が、1人目のコピーになっていました。

KOL向けの投稿指示書を自動で作るシステムです。8体のAIエージェントが協調して1枚のPPTXスライドデッキを生成します。調査、デザイン、ビルド、レビュー。で、3人目から品質が崩壊した。コンテキストウィンドウに全情報を詰め込むと、LLMは後半で前半のパターンを繰り返すんですよね。「潤い」と「自然由来」をひたすら繰り返してるだけのキャプション。

結論から言うと、マルチエージェントシステムの設計で一番難しいのはエージェントの能力ではなくて**「誰が何を知るべきか」の境界設計**でした。壊れるのは常にエージェントの「間」です。この記事ではその話をします。

## 境界を引く

1つの巨大なエージェントではダメだったので分割する。ここまでは誰でも思いつきます。責務ごとに分ければマイクロサービスと同じでスケールするだろう、と。

でも分けた瞬間に「何を渡すか」の設計が始まる。マイクロサービスだとサービス間のAPI仕様を明示的に定義しますよね。エージェントでも同じで、「なんとなく伝わる」は通用しない。LLMベースのエージェントは長いコンテキストの中で関係ない情報に引っ張られるので、渡す情報を絞ること自体が設計の本質です。

DDDでいう「境界づけられたコンテキスト」がそのまま当てはまりました。マイクロサービスで全サービスが同じDBを見るのはアンチパターンですけど、エージェントも同じで全員に全情報を渡すと品質が落ちる。でも絞りすぎると必要な文脈を持てなくて的外れなアウトプットを出す。この加減がめちゃくちゃ難しくて、正直いまだに完璧な答えは出てないです。

試行錯誤で辿り着いたのが「全体方針」と「個別情報」の分離でした。

8人のKOLに個別素材を作りつつキャンペーン全体のトーンを統一したい。最初はKOL担当エージェントそれぞれに「キャンペーンの方向性」をテキストで渡して個別に作らせたんですけど、あるKOLはポップで別のKOLはシックで、同じキャンペーンに見えない。料理で言えばベースのスープを先に作ってからトッピングを変えるべきで、スープなしにトッピングだけ変えても統一感は出ないんですよね。

```
[素材取得] → [全体スタイル確定] → [個別KOL素材 ×N] → [統合チェック]
                    ↑
               ここにゲート
         全体スタイルが確定するまで
           個別制作を始めちゃダメ
```

並列処理の誘惑に負けて「全部同時に走らせちゃえ」ってやると不整合が起きます。並列にしていいのは互いに独立したタスクだけで、KOL間は並列OK、でも「全体スタイル → 個別KOL」は直列でないといけない。

ここで気づいたのが、情報の境界設計は「誰が何を知るか」だけじゃなくて**「どの順序で情報を確定させるか」の設計でもある**ということです。上流の品質が下流を規定する。工場の組み立てラインと同じで、工程間に検査ゲートが必要になります。

で、ゲートが何を検査すべきかという話になるんですけど、調査エージェントが「このKOLは20代女性に人気」って出してきたとして、それが正しいかなんて次のエージェントには判定できない。もっと地味な問題として、8人分のKOLリサーチを並列で走らせて全員分揃ったかどうかの確認。これないと後工程が中途半端な入力で動き出します。

組織論で考えたら腹落ちしました。マネージャーの仕事って中身の正しさを判定することじゃなくて「必要な成果物が全部揃ったか」と「次に進めていいか」の判断なんですよね。CIパイプラインとまったく同じ構造です。CIサーバーがコードの品質を判定するんじゃなくて、テストが通ったかどうかのゲートを管理する。

```
オーケストレーターの責務:
  ✅ 成果物の存在チェック（8人分あるか？）
  ✅ 前提条件の検証（必須フィールドが埋まっているか？）
  ✅ 次工程への遷移判断
  ❌ 内容の品質判断（それは専門エージェントの仕事）
  ❌ 工程内の具体的な手順（各エージェントに任せる）
```

8人中7人のリサーチが成功して1人だけ失敗した場合は、その1人だけ再実行すればいい。ただし各エージェントの出力が独立している場合に限ります。全体方針に関わる工程なら1つの失敗が全体に波及するから部分再実行じゃダメで全体やり直し。再実行の粒度も情報の境界設計の一部です。

情報の境界、確定の順序、ゲートの設計。この3つは全部「誰が何を知るべきか」から派生してます。分けた瞬間にこの設計空間が開くんですけど、逆にここさえ固まれば個々のエージェントの実装はわりとどうにでもなる。あなたがマルチエージェントを組むときも、最初に考えるべきは「どんなエージェントを作るか」じゃなくて「エージェント間で何を渡して何を渡さないか」だと思います。

## 古い成果物を掴んだ話

案件を2回流したとき、前の案件の成果物がディスクに残ってて、次の案件のエージェントがそれを「最新」だと思って使いました。

前の案件で作ったカバー画像がそのまま新しい案件に。ファイルの場所も名前も同じだから。これマジで腹が立ったんですけど、んで、怒りが収まってから冷静に考えたらエージェントは何も悪くなくて。「どの実行に属する成果物か」って情報がどこにもない設計が悪かった。最初は「前のファイル消してから走らせればいい」と思ってたんですけど消し忘れで2回やらかして、3案件目でようやく「人間の注意力に依存する設計は破綻する」という当たり前のことに気づきました。当たり前のことに気づくのに3案件かかるの、笑えないですよね。

分散システムの「古いレプリカを読む」問題と本質的に同じでした。データベースのMVCCと同じ発想で解決しています。各行がどのトランザクションで書かれたかを持っていて読み取り時にチェックする、あれと同じことを成果物にやるわけです。各成果物に実行IDを刻印して、ゲートでスコープの一致を確認する。刻印漏れ対策として、スコープ情報がない成果物はゲートが通さない。フェイルセーフです。

## 品質ループとモード切替

PPTXのレビューは「ここ直して」→ 修正 → 再チェックのループです。データの問題なのか見た目の問題なのかで直すべきエージェントが違う。レビューエージェントが分類ラベルを付けて、オーケストレーターはラベルに従ってルーティングするだけ。中身は読まない。郵便局の仕分けと同じです。

ビルドからレビューまでを1つのチームとして囲い込んで、ループはチームが自律的に回す設計にしました。ループ上限を超えたら人間にエスカレーション。

```
[ビルド] → [ビジュアル] → [レビュー]
                              ↓ REJECT
                        ラベルで振り分け
                      data → ビルドへ
                      visual → ビジュアルへ
                              ↓ APPROVE → 出力
```

KOLが3人ならオーケストレーターが直接処理も担当、30人なら管理に専念して全部委任。成果物のインターフェースはどちらでも統一。ポリモーフィズムです。正直、少数モードはオーケストレーターに処理を持たせてるので設計として美しくはないんですけど、3人のために大仰な委任構造を組むのもな...という迷いがあって、今のところこれで回ってます。

:::details 設計原則が再帰する話
素材生成の工程が重すぎて分割したんですけど、分割した小さなステップにもパイプライン全体と同じ原則が必要でした。情報の境界、ゲート、スコープ刻印。大きなパイプラインの中に小さなパイプラインがある。フラクタルです。最初は過剰設計だと思ったんですけど、内部にゲートなしで走らせたら途中の失敗を切り分けられなくて最初からやり直しになりました。
:::

## 設計の全体像

```
┌─ 情報の境界 ──────── 誰が何を知るか
├─ 確定の順序 ──────── 全体→個別は直列
├─ 成果物ゲート ────── 完全性チェックで遷移判断
├─ スコープの刻印 ─── どの実行に属するか
├─ 品質の閉ループ ─── レビュー→修正→再検証
└─ モード切替 ──────── 規模に応じた実行戦略
```

最初は「エージェント分ければスケールする」くらいに思ってました。人間のチームワークでは暗黙知として機能していることを全部プロトコルとして形式化しないといけない。そこが大変だけど、そこにしか面白さはないという気がしています。モード切替の閾値をKOL数で決めてるのは雑すぎるかもしれなくて、コンテキスト量で動的に判定すべきなのかもしれない。でもまあ動いてるうちは触らないことにしてます。
