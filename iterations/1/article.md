3人目のKOLのキャプションが、1人目のコピーになっていました。

KOL向けの投稿指示書を自動生成するシステムを作っていて、最初は1つの大きなAIエージェントに全部やらせていたんですけど、3人目あたりからクオリティが露骨に落ちるんですよね。コンテキストウィンドウの中で情報が混ざって、後半が前半の繰り返しになる。「エージェント分ければスケールするでしょ」と思って分割を始めたところから、想像以上に深い設計の話に入り込みました。

この記事では、8体のAIエージェントが協調して1枚のPPTXを作るシステムを設計した過程で気づいたことを書きます。結論から言うと、マルチエージェントシステムの設計は**「エージェントの能力」の設計じゃなくて「エージェント間の関係性」の設計**でした。

## 分けたら分けたで、別の問題が出てくる

マイクロサービスと同じ発想で、調査担当、デザイン担当、組み立て担当、チェック担当...と責務ごとにエージェントを分けました。

で、すぐに壁にぶつかります。

分けた瞬間、エージェント間で「何を渡すか」を決めなきゃいけない。人間のチームなら「なんとなく伝わる」でいけることが、AIエージェントではコンテキストが共有されないので全部明示的に設計する必要があります。渡す情報が少なすぎると的外れなアウトプットが出るし、全部渡すと今度は情報が多すぎて何が重要かをエージェントが判断できなくなる。

DDDの「境界づけられたコンテキスト」と全く同じ構造です。

じゃあ何を渡して何を渡さないか。ここで「全体方針」と「個別情報」を分離する設計にしました。ブランドのトーンやキャンペーンの方向性は全員が知るべきこと、特定のKOLのリサーチ結果や特定のプラットフォーム要件は担当者だけが知ればいいこと。

ここまでは「まあそうだよね」って話だと思います。

## ベースのスープ問題

本当に困ったのはここからです。

8人のKOLに個別の素材を作るとき、キャンペーン全体のトーンは統一したいわけですよ。ロゴの使い方、色味、全体の雰囲気。で、最初はそれぞれのKOL担当エージェントに「キャンペーンの方向性」を渡して個別に作らせていたんですけど、**バラバラになる**。あるKOLはポップで、別のKOLはシック。同じキャンペーンに見えない。

ラーメンで考えるとわかりやすくて、ベースのスープを先に作ってからトッピングを変えるべきところを、8人の料理人に「豚骨っぽいやつで」って言葉だけ渡して各自にスープから作らせた。そりゃバラバラになりますよね。

```
[素材取得] → [全体スタイル確定] → [個別KOL素材生成 ×N] → [統合チェック]
```

全体スタイルが確定する**前に**個別制作を始めちゃダメなんです。ここで並列処理の誘惑との戦いがありました。「全部同時に走らせれば速いじゃん」。でも依存関係のあるタスクを並列にすると壊れる。KOL間は独立しているから並列OK、でも「全体スタイル → 個別KOL」は直列でないといけない。

この「情報の境界」って、「誰が何を知るか」だけじゃなくて「どの順序で情報を確定させるか」の設計でもあるんです。上流の品質が下流を完全に規定するので、工程間に検査ゲートが必要になる。工場の組み立てラインと同じ構造です。

正直、この依存関係の整理にかなり時間を使いました。速度を捨てて安全を取る判断って、頭ではわかっていても手が動かないんですよね。

## ゲートは「中身」を見ない

で、ゲートで何を検査するかが次の問題です。

調査エージェントが「このKOLは20代女性に人気」って出してきたとして、それが正しいかどうかなんて次のエージェントにはわからない。オーケストレーターにもわからない。

ここで組織論の考え方が役に立ちました。マネージャーの仕事って、部下の成果物の中身が正しいかを判定することじゃなくて、「必要な成果物が全部揃ったか」と「次に進めていいか」の判断なんですよね。

```
オーケストレーターの責務:
  ✅ 成果物の存在チェック（8人分のリサーチが全部あるか）
  ✅ 前提条件の検証（必須フィールドが埋まっているか）
  ✅ 次工程への遷移判断
  ❌ 内容の品質判断（レビューエージェントの仕事）
  ❌ 工程内の手順管理（各エージェントの仕事）
```

CIパイプラインと同じ。CIサーバーがコードの品質を判定するんじゃなくて、テストが通ったかのゲートだけ管理する。

ゲートで「揃ってない」判定が出たときの再実行粒度も設計が要ります。8人中7人のリサーチが成功して1人だけ失敗なら、その1人だけ再実行すればいい。ただしこれは出力が独立している場合だけで、「全体の方向性」に関わる工程だと1つの失敗が全体に波及する。

## 前の案件の画像がそのまま出てきた

これが一番腹が立ちました。

案件を2回流したとき、前の案件の成果物が残っていて、新しい案件のエージェントがそれを「最新の成果物」として使ってしまった。前の案件で作ったカバー画像を新しい案件でそのまま採用。ファイルの場所も名前も同じ、中身だけが古い。

冗談みたいな話なんですけど、エージェントは悪くないんですよ。「どの実行に属する成果物か」という情報がどこにもなかった。

分散システムの「古いレプリカを読んでしまう」問題と本質的に同じです。解決策はデータベースのMVCCと同じ発想にしました。MVCCというのは、各レコードがどのトランザクションで書かれたかを記録していて、読み取り時にバージョンをチェックする仕組みです。これを成果物にも適用して、各成果物に「この実行のものである」という刻印を付ける。ゲートは存在チェックに加えてスコープの一致もチェックする。

で、もちろん刻印を忘れるリスクがある。だからゲートをフェイルセーフ方向に倒しました。スコープ情報がない成果物が見つかったら、通過させるんじゃなくて止める。

最初は「前の案件のファイルを消してから走らせればいいでしょ」って思ってたんですけど、消し忘れで2回やらかしました。人間の注意力に依存する設計は必ず破綻します。当たり前のことに気づくのに3案件かかった。

## レビューループを「チーム」として囲い込む

PPTXができあがった後のレビュープロセスも設計が必要でした。「ここ直して」ってフィードバック返して、修正して、またチェック。このループで特に難しかったのはルーティングです。

データの問題なのか見た目の問題なのかで、直すべきエージェントが違う。レビューエージェントは「何が問題か」だけでなく「どこに問題があるか」を分類ラベルとして出力する。オーケストレーターはそのラベルに従ってルーティングするだけ。郵便局の仕分け担当が封筒の中身を読まずに宛先だけ見て振り分けるのと同じです。

「あれ、これオーケストレーターが中身を判断してない？」って一瞬思ったんですけど、ラベルを見てるだけなので中身は読んでない。

で、このビルド→レビューの閉ループを丸ごと1つのチームとして囲い込む設計にしました。

```
┌─────────────────────────────────────┐
│  ビルド→レビュー チーム             │
│                                     │
│  [ビルド] → [レビュー]             │
│       ↑        │                   │
│       │    FAIL(データ)→ [ビルド]   │
│       │    FAIL(見た目)→ [ビジュアル]│
│       └────────┘                   │
│          PASS → 完了                │
│      上限N回でエスカレーション       │
└─────────────────────────────────────┘
```

オーケストレーターがステップごとに介入するんじゃなくて、チームに裁量を渡して内部ループはチームが自律的に回す。ループ上限だけ決めておいて、超えたら人間にエスカレーション。組織で言えばプロジェクトチームへの権限委譲ですね。

## 同じ原則がフラクタルに再帰する

設計してて思わず笑ったことがあります。

素材制作の工程が重すぎたので分割したんですけど、分割した各ステップにもパイプライン全体と**まったく同じ設計原則**が必要でした。情報の境界、ゲート、スコープ刻印。全部。大きなパイプラインの中に小さなパイプラインがある。

```
パイプライン全体:
  [調査] →ゲート→ [計画] →ゲート→ [制作] →ゲート→ [レビュー]

「制作」の内部:
  [素材取得] →ゲート→ [全体トーン] →ゲート→ [KOL個別×N] →ゲート→ [統合]
```

最初は過剰設計を疑ったんですけど、内部にゲートを入れずに走らせたら途中で失敗したときにどこで壊れたか切り分けられない。「1人に全部任せたら壊れた」のと同じ問題が、粒度を変えて再発する。

「設計がスケールする」ってこういうことなんですね。同じ原則の適用先が増えるだけで、新しいルールが増えるわけじゃない。

ちなみに、KOLが3人のときと30人のときで同じ設計にするかも考えました。3人なら正直オーケストレーターが全部見ても大丈夫。30人だとリサーチ結果を全部読むだけでコンテキストが溢れる。会社組織で言えば5人のスタートアップと50人の組織の違い。入力規模に応じて実行モードを切り替えるけど、成果物のインターフェースは統一する。ポリモーフィズムです。少数モードのほうが設計は汚いんですけど、3人のために大仰な委任構造を組むのはオーバーエンジニアリングかなと。この判断が正しかったかは、正直まだわかりません。

## 暗黙知を形式化する作業

人間のチームだと「あ、これ前の案件のだ」「先に全体の方向決めてから」「これ誰に聞けばいい？」を無意識にやっています。暗黙知として機能していることを、AIエージェントでは全部データ構造とプロトコルにしないといけない。

```
┌─ 情報の境界 ──── 誰が何を知るか
├─ 全体→個別 ──── グローバルを先に確定
├─ 成果物ゲート ── 完全性チェックで遷移判断
├─ スコープ刻印 ── どの実行に属するか
├─ 品質ループ ──── レビュー→修正→再検証
├─ 責務の絞込 ──── オーケストレーターは管理のみ
└─ モード切替 ──── 規模に応じた実行戦略
```

面倒だけど、面白い。ただ、ゲートの検査密度の最適解はまだ見えていません。ゲートを増やすと安全だけど遅い、減らすと速いけど事故る。規模が変わるたびにチューニングが要る気がしていて、この辺りの塩梅は手探りのまま次の案件に進んでいます。
