インフルエンサー向けの投稿指示書を自動生成するシステムを作りました。投稿指示書というのは、SNSのインフルエンサーに「この商品を、こういう世界観で、こんなキャプションで投稿してください」と伝えるためのスライド資料です。商品を調べて、インフルエンサーの過去投稿を分析して、クリエイティブの方向性を考えて、画像素材を用意して、スライドに落とし込む。8体のAIエージェントにこの仕事を分業させたら、5人目のインフルエンサーの指示書が1人目のコピペみたいになる問題は解消されるはずでした。

されるはずだったんですけど、「分ければスケールする」は半分嘘でした。

## 手作業の何がつらかったか

1案件でインフルエンサー5人分の投稿指示書を作ると、丸1日かかります。最初の2人分は丁寧に作れるんですけど、3人目あたりから「さっきも似たようなの書いたな」ってなって、5人目にはキャプションの方向性が1人目とほぼ同じになってたりする。クライアントに出す直前に気づいてヒヤッとしたことが何度かあります。

で、「AIにやらせればいいじゃん」と。調査担当、企画担当、画像担当、スライド組み立て担当、レビュー担当。マイクロサービスみたいに分ければスケールするでしょ、くらいの雑な仮説から始めました。

## 最初の設計：「全員に全部見せる」

最初に考えた設計はシンプルでした。全エージェントに全情報を渡す。商品情報、インフルエンサーのプロフィール、ブランドガイドライン、過去の投稿データ、全部。情報は多い方が判断の質が上がるだろう、と。

```
[全情報] → [調査エージェント]
[全情報] → [企画エージェント]
[全情報] → [画像エージェント]
[全情報] → [スライドエージェント]
```

これ、ダメでした。

画像生成エージェントに商品の競合分析情報まで渡したら、競合ブランドのビジュアルスタイルに引っ張られた画像を作り始めたんですよ。「参考にした」つもりなんでしょうけど、クライアントからしたら最悪です。

## 「誰が何を知るべきか」という問い

ここで気づいたのが、マルチエージェントの設計で一番大事なのは「誰に何をやらせるか」じゃなくて**「誰が何を知るべきか」**だったということです。

組織設計でいう「Need to Know」の原則がそのまま当てはまります。画像生成エージェントには「商品のメイン画像」と「このインフルエンサーの世界観」だけ渡せばいい。競合分析は要らない。むしろ渡すと害がある。

各エージェントについて3つのことを定義しました。

- **何を読んでいいか**: そのエージェントの仕事に必要な情報
- **何を書いていいか**: そのエージェントが生成する成果物
- **何を読んではいけないか**: 渡すと混乱する情報

これ、コードでいうとpublic/privateのアクセス修飾子と同じ発想なんですよね。でもAIエージェントの場合、アクセス制御は言語機能じゃなくて設計者が意識的にやらないといけない。サボると静かに壊れます。

## パイプラインの依存関係

知識の境界を定義したら、次は「どの順番で実行するか」です。

```
[受注資料]
    ↓
[Intake] ── 資料の読み取りと整理
    ↓
┌────────────────────────┐
│ [商品調査]  [KOL調査]  │  ← 並列OK
└────────────────────────┘
    ↓ (合流ゲート)
[クリエイティブ企画] ← KOLごと
    ↓
[素材取得] → [全体素材] → [KOL別素材] → [素材統合]
    ↓
┌──── チーム ─────┐
│ [ビルド]         │
│ [ビジュアライズ]  │
│ [レビュー]       │
│   ↕ 修正ループ   │
└──────────────────┘
    ↓
[エクスポート]
```

商品調査とインフルエンサー調査は並列に走れます。片方の結果がなくてももう片方は進められるから。ただし、クリエイティブ企画は両方の結果が揃わないと始められない。「この商品を、このインフルエンサーなら、どう見せるか」を考えるわけなので。

ここに**ゲート**を置きます。「商品調査と KOL調査の両方が完了したか？」をチェックして、揃ってなければ待つ。CIパイプラインのステージゲートと同じ考え方です。

## ゲートは2層必要だった

最初は「成果物が存在するかどうか」だけチェックしてました。ファイルがあればOK、なければNG。

甘かったです。

ファイルはあるけど中身がスカスカ、というケースが発生しました。形式的には正しいけど、商品調査の結果が「商品名: ○○、特徴: 高品質」の2行だけ、とか。これでクリエイティブ企画に進んでも、まともな企画は出てこない。

ゲートは2層構成にしました。

1. **形式チェック**: 必要なフィールドが存在するか
2. **内容チェック**: そのフィールドの中身が次工程の入力として十分か

ソフトウェアのテストでいえば、コンパイルが通るか（形式）と、テストが通るか（内容）の違いです。

## 「これ最新？」問題

レビューで差し戻しが発生したとき、地味に厄介だったのがこれです。

画像を差し替えたのに、スライド組み立てエージェントが古い画像を参照してた。ファイル名は同じなので、エージェントには「古い」かどうか判断する手段がない。

分散システムでいう「ステール読み取り」そのものです。

解決策は、全成果物に「実行ID」を埋め込むことでした。今回の実行で生成されたものか、前回のやり直し前のものか。IDが一致しなければ古い成果物として無視して、再生成を要求する。

正直、ここまでやるの？ って思いました。投稿指示書を作るだけなのに、分散システムの一貫性制御みたいな話になってる。でもやらないと壊れるんですよ、静かに。エラーは出ない。ただ古い画像が最終スライドに残るだけ。気づかないまま納品しかねない。

## 画像生成を4段階に分けた理由

画像素材の生成は、システム全体で一番複雑な工程でした。種類が多いんです。商品のメイン画像、キャンペーン全体のカバー画像、インフルエンサーごとのシーン画像。

最初は1つのエージェントに全部やらせてました。

で、人間と同じ問題が起きた。後半の画像がだんだん雑になる。AIに集中力の概念はないはずなのに、コンテキストが長くなりすぎて前半の指示を忘れてるんじゃないかと。

分けました。4段階に。

1. 既存素材（ロゴ、商品写真）を集める
2. キャンペーン全体の世界観を決めて、カバー画像を生成する
3. その世界観をベースに、各インフルエンサーのシーン画像を生成する（並列可）
4. 全素材の整合性を確認する

ポイントは「全体→個別」の順序です。全体の世界観が決まってからインフルエンサーごとの画像を作らないと、5人分の画像がバラバラのトーンになります。これ、手作業でも同じ失敗をしてたんですよね。

で、3番目の「各インフルエンサーのシーン画像」は並列に生成できます。全体の世界観が共有データとして確定した後なら、各エージェントが独立して動ける。ただし、2番目が完了する前に3番目を始めてはいけない。ここにもゲート。

:::details 4段階に分けるかは迷った
正直、分けすぎるとエージェント間の受け渡しコストが馬鹿にならないです。4段階にする前に「2段階（全体素材 / 個別素材）」も試しました。ただ、2段階だと差し戻し時に「カバー画像はOKだけどシーン画像がダメ」なとき、全部やり直しになる。粒度は差し戻しの単位で決めた、というのが正直なところです。
:::

## ビルドとレビューのチーム化

最後の工程、スライドの組み立て（ビルド）→ 見た目の調整（ビジュアライズ）→ レビュー。ここで一番悩みました。

最初は逐次呼び出しでやってました。ビルドエージェントがスライドを作る → レビューエージェントがチェックする → ダメなら修正指示を次のビルドに渡す。

伝言ゲームになりました。

レビューが「このスライドの商品画像が小さすぎて目立たない」と指摘する。次のビルドに「画像を大きくして」と伝える。結果、画像だけ巨大になってテキストが潰れた。レビューの文脈（「他の要素とのバランスで目立たない」）が落ちてたんです。

解決策は、ビルド・ビジュアライズ・レビューを「チーム」として管理すること。逐次呼び出しではなく、1つのチームの中で修正ループを回す。チーム内でコンテキストが共有されるので、レビューの指摘がビルドに正確に伝わります。

チーム間はファイルで疎結合。チーム内は密結合。

ループの上限は設けました。最大10回。それを超えたら人間にエスカレーション。嘘でしょ、って思うかもしれませんけど、実際に7回ループしたことがあります。

## 3人と30人で設計を変える

ここまで読んで「めんどくさ」と思った方、正解です。正直、ここまで面倒だと思わなかった。

で、気づいたんですけど、この全部の仕組みが要るのはインフルエンサーが多い案件だけなんですよ。3人くらいの案件なら、オーケストレーターが直接全部処理しても回る。

設計を2モードに分けました。

- **直接処理モード（3人以下）**: オーケストレーターが調査もアセット周りも直接やる
- **委譲モード（4人以上）**: オーケストレーターはハンドオフと品質チェックに専念、各工程は専門エージェントに委譲

どちらのモードでも成果物のフォーマットは同じにしてます。後続工程からは、どちらのモードで処理されたか見分けがつかない。

## 振り返り：分割より結合

最初は「エージェントを分ければスケールする」と思ってました。マイクロサービスと同じノリで。

実際にやってみると、分割は簡単でした。「商品調査」「企画」「画像生成」「スライド組み立て」。5分で決まる。

大変だったのは、分けた後に繋げる部分です。成果物の契約、ゲート、知識の境界、実行IDによる鮮度管理、チーム化、スケールによるモード切り替え。これら全部、「繋ぎ」の設計です。

で、さらに言うと、繋いだ後に「繋がりを維持する」仕組みまで要る。実行IDの話がまさにそれで、一度繋いでも差し戻しが発生すると繋がりが壊れる。壊れたことに気づく仕組みまで作らないと、静かにおかしな成果物が出てくる。

一番の学びは「誰が何を知るべきか」の設計がマルチエージェントシステムの核心だということです。まさかそこがメインディッシュになるとは思ってなかった。全エージェントに全情報を渡せばいい、という最初の設計方針は、恥ずかしいくらい甘かった。

これからマルチエージェントシステムを作る方へ。分けることから始めないでください。「各エージェントが何を知っていて、何を知らなくていいか」から始めると、分け方は自然に決まります。
