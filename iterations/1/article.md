8体のAIエージェントが協調して1枚のPPTXデッキを生成するシステムを作りました。KOL向けの投稿指示書を自動で作るやつです。で、作ってみてわかったのは、**マルチエージェントシステムの設計で一番大事なのは「エージェントの能力」ではなく「エージェント間の関係性」の設計だ**ということです。

もっと具体的に言うと、「誰が何を知るべきか」の境界を先に設計しないと、どんなに賢いエージェントを並べても品質が壊れます。

## 「分ければスケールする」から始まった

KOL向けの投稿指示書って、KOLごとにキャプション方向性もシーン構成も全部変える必要があって、手作業だと3人分で半日かかります。これを自動化したい。

最初に考えたのは素朴な話で、「マイクロサービスみたいにエージェント分ければスケールするでしょ」と。調査する人、デザインする人、組み立てる人、チェックする人。責務ごとに分けて、成果物をファイルで渡していく。

で、まず1体のエージェントに全部やらせてみたんです。分割する前にモノリスの限界を確認しておきたかったので。

3人のKOLで試したら、ちゃんと動きました。よしよし。

6人のKOLで試したら壊れました。5人目と6人目のキャプション方向性が1人目のコピーになってる。「潤い」と「自然由来」をひたすら繰り返してるだけ。コンテキストウィンドウの限界に近づくと、LLMは新しいパターンを出すより既出パターンを繰り返すほうを選ぶみたいなんですよね。

分割は不可避でした。でも問題は「どう分けるか」じゃなかった。

## 境界づけられたコンテキスト、またの名を「何を知らせないか」

分割すると決めたはいいけど、「誰が何を知るべきか」の線引きがまったくわからない。全員にプロジェクト全体の情報を渡す？ それとも最小限？

DDDの「境界づけられたコンテキスト」がそのまま当てはまりました。マイクロサービスで「全サービスが同じDBを見る」のがアンチパターンなのと同じで、全エージェントに全情報を渡すと、関係ない情報に引っ張られて品質が落ちます。でも絞りすぎると必要な文脈を持てなくて的外れなアウトプットを出す。

```
全体方針（全員が知るべきこと）
  ├── ブランドのトーン
  ├── キャンペーンの方向性
  └── 品質基準

個別情報（担当者だけが知ればいいこと）
  ├── 特定KOLのリサーチ結果
  ├── プラットフォーム固有の要件
  └── 個別のビジュアル素材
```

ただ、この分離だけだと片手落ちでした。「全体方針」を誰が決めるのか。全体方針そのものがエージェントの出力なわけで、その品質が低ければ後工程の全員が間違った方向に進みます。

工場の組み立てラインと同じですよね。上流の品質が下流を規定する。だから工程間に品質チェックのゲートが要る。

## ゲートの正体はCIパイプラインだった

で、ゲートが何を検査するかで悩みました。最初は「成果物の内容が正しいか」を検証しようとしたんですけど、それ無理です。調査エージェントが「このKOLは20代女性に人気」って出してきて、それが本当かどうかなんてゲートには判定できない。

組織論で考えると腹落ちしました。マネージャーの仕事って部下の仕事の中身を判定することじゃなくて、「必要な成果物が全部揃ったか」と「次に進めていいか」の判断なんですよね。

```
オーケストレーターの責務:
  ✅ 成果物の存在チェック
  ✅ 前提条件の検証
  ✅ 次工程への遷移判断
  ❌ 内容の品質判断（専門エージェントの仕事）
  ❌ 工程内の具体的な手順（各エージェントに任せる）
```

CIパイプラインとまったく同じ構造です。CIサーバーがコードの品質を判定するんじゃなくて、テストが通ったかどうかのゲートを管理する。テストの中身の責務はテストコードにある。

8人のKOLリサーチを並列で走らせたとき、7人分は成功したけど1人分が失敗してたことがあります。で、そのまま次の工程に進んで、その1人だけプランが薄っぺらくなった。「8人分全部揃ったか？」という検査をゲートに入れたら、欠落を検知して1人分だけ再実行できるようになりました。

ゲートで検査すべきは「中身の正しさ」ではなく「成果物の完全性」です。これ、地味に聞こえるかもしれないですけど、この判断がシステム全体の信頼性を決定的に変えました。

## ベースのスープを先に作る

ビジュアル素材の話で別の壁にぶつかりました。8人のKOLに個別の素材を作るとして、キャンペーン全体のトーンは統一したいわけです。

最初はそれぞれのKOL担当エージェントに「キャンペーンの方向性」をテキストで渡して個別に作らせてたんですけど、バラバラになるんですよ。あるKOLはポップで、別のKOLはシック。同じキャンペーンに見えない。

これ、ラーメンで考えたら一発で腹落ちしました。ベースのスープを先に作って、その上で個別のトッピングを変える。スープなしにトッピングだけ変えても統一感は出ない。

```
[素材取得] → [全体スタイル確定] → [個別KOL素材 ×N] → [統合チェック]
                    ↑
               ここにゲート
         （全体スタイルが確定するまで個別制作を開始しない）
```

重要なのは依存関係の明示です。個別KOL間は独立してるから並列にできる。でも「全体スタイル → 個別KOL」は直列でないといけない。並列化の誘惑に負けて全部同時に走らせると、全体スタイルなしに個別制作が動き出して不整合が起きます。

「何が並列可能で何が直列必須か」を依存関係グラフとして宣言する。ゲート設計の核心はここでした。

## MVCCの再発明 — 成果物にスコープを刻印する

運用で最高に腹が立った問題の話をします。

案件を2回連続で流したとき、前の案件の成果物がディスクに残ってて、次の案件のエージェントがそれを「最新の成果物」だと思って使っちゃったんです。前の案件で作ったカバー画像を、新しい案件でもそのまま使ってる。名前も場所も同じ。でも中身は前のやつ。

人間なら「あ、これ古いわ」ってわかる。でもエージェントには「どの実行に属する成果物か」の情報がない。3案件くらい無駄にして、ようやく気づきました。エージェントが悪いんじゃなくて、「所属」の情報がどこにもない設計が悪い。

解決策はデータベースのMVCC（マルチバージョン同時実行制御）と同じ発想です。データベースでは各行が「どのトランザクションで書かれたか」を持っていて、読み取り時にそのIDをチェックする。同じことを成果物にやります。各成果物に実行IDを刻印して、ゲートは存在チェックだけでなくスコープの一致もチェックする。

刻印を忘れるリスク？ あります。だからゲートをフェイルセーフにする。「スコープ情報がない成果物」が見つかったらゲートが止める。通過させるのではなく、止める方向に倒す。

:::details 余談
最初は「前の案件のファイルを消してから走らせればいい」と思ってました。消し忘れで2回やらかしました。人間の注意力に依存する設計は、必ず破綻しますね。
:::

## レビューの閉ループ — 郵便局の仕分け

ここまでの設計は「前工程 → ゲート → 次工程」の一方通行でした。でも最終成果物の品質保証にはフィードバックループが要ります。

PPTXが出来た後、レビューエージェントが「キャプションのトーンがブリーフと合ってない」と指摘する。なのにそのまま出力されてしまう。修正の仕組みがなかった。

```
[ビルド] → [ビジュアル適用] → [レビュー]
                                  ↓ REJECT
                           どこに戻す？
                         データの問題 → ビルドへ
                         見た目の問題 → ビジュアルへ
                                  ↓ APPROVE
                              出力へ
```

ここで面白い設計判断がありました。レビュー結果に基づいてルーティングするんですけど、「オーケストレーターは中身を判断しない」という原則と矛盾しないのか？

矛盾しません。郵便局の仕分け担当と同じです。中身は読まない。宛先を見て振り分けるだけ。レビューエージェントが「データの問題です」「ビジュアルの問題です」という分類ラベルを付けてくれれば、オーケストレーターはラベルに従ってルーティングするだけです。

さらに、ビルドからレビューまでを1つのチームとして囲い込むようにしました。オーケストレーターがステップごとに介入するんじゃなくて、チームに裁量を渡して内部のループはチームが自律的に回す。ループの上限だけ設定しておいて、超えたら人間にエスカレーション。

## フラクタル構造 — 設計原則は再帰する

設計していて笑ってしまったことがあります。

素材生成の工程が重すぎたから分割したんですけど、分割した小さなステップにもパイプライン全体とまったく同じ設計原則が必要だったんです。素材集め → 全体トーン確定 → KOL個別 → 統合チェック。これ自体が1つの小さなパイプラインで、情報の境界もゲートもスコープの刻印も全部要る。

大きなパイプラインの中に小さなパイプラインがある。フラクタル構造です。オブジェクト指向のコンポジションパターンと同じで、ツリーの各ノードが同じインターフェースを持つ。

最初は「そんなバカな、各工程の内部まで同じ設計にするのは過剰設計では」と思ったんですよね。でも実際に内部にゲートを入れずに走らせたら、工程の途中で失敗したときにどこで失敗したか切り分けられない。結局最初からやり直すしかなくて、非効率の極み。

## オーケストレーターは何を「しないか」

最後に。ゲート管理、モード選択、ルーティング...と責務を足していくと、オーケストレーター自体がモノリスになります。組織論で言うマイクロマネジメントの反パターン。

決めたのはこうです。

やること: 実行モードの選択。工程の順序管理。成果物のゲート判定。

やらないこと: 各工程の具体的手順。成果物の内容評価。情報の加工。

工程固有の手順は各エージェントの中に閉じ込めて、オーケストレーターが知るのは「何が入って何が出るか」だけ。UNIX哲学の「1つのことをうまくやる」。

正直、最初に「エージェント分ければスケールする」って言ったとき、ここまで複雑な設計論になるとは想像もしてなかったです。でもこの複雑さは「偶発的な複雑さ」ではなく「本質的な複雑さ」なんですよね。情報の境界、ゲート、スコープ管理、品質ループ、モード切替。どれか1つ抜けると全体が壊れる。有機的に連携している。

人間のチームワークでは暗黙知として機能していることを、AIエージェントシステムでは全部プロトコルとして形式化しないといけない。まだ設計が正解かはわかっていません。特にモード切替の閾値をKOL数で決めてるのは雑すぎる気がしていて、コンテキスト量で動的に判定すべきかもしれない。でもまあ、動いてるうちは触らないことにしてます。
