KOL向けの投稿指示書、PPTXで30枚くらいのやつを自動生成したいと思って始めたプロジェクトがあります。8人のKOLそれぞれにカスタマイズしたスライドを出す。調査して、素材作って、組み立てて、レビューして。

「エージェント分ければスケールするでしょ」で始めたんですけど、3ヶ月やって学んだのは、分け方より**エージェント間の関係性の設計**がはるかに難しいということでした。

## 1人に全部任せたら3人目で壊れた

最初は1つの大きなエージェントに全工程を任せました。コンテキスト途切れないし、受け渡しの設計不要だし、良いことづくめ。

3人目のKOLあたりから目に見えて品質が落ちました。キャプションの方向性が1人目のコピーになる。5人目のビジュアル指示が2人目と酷似する。LLMは長いコンテキストの中で「どの情報がどのKOLのものか」を区別できなくなって、全部がぼんやり平均化されていくんですよね。

で、分けるしかないと。ただ、マイクロサービスで同じ話があるように、サービスの分割は簡単で、サービス間のインターフェース設計こそが難しい。DDDの「境界づけられたコンテキスト」がそのまま当てはまりました。全エージェントが全情報を見るのは「全サービスが同じDBを見る」のと同じアンチパターンです。

ここまでは教科書通りの話です。

## ベースのスープを先に作れ

本当に困ったのはここからで。

8人のKOLに個別の素材を作りつつキャンペーン全体のトーンは統一したい。で、最初はKOL担当エージェントそれぞれに「キャンペーンの方向性」をテキストで渡して個別に作らせたんですけど、結果がバラバラ。あるKOLはポップな素材、別のKOLはシックな素材。同じキャンペーンに見えないんですよ。

ラーメンに例えると、ベースのスープを先に作ってからトッピングを変えるべきところを、8人に「豚骨っぽいやつで」って言葉だけ渡して各自にスープから作らせた。そりゃバラバラになります。

```
[素材取得] → [全体スタイル確定] → [個別KOL制作 ×N] → [統合チェック]
```

全体スタイルが確定する**前に**個別制作を始めちゃダメなんです。並列処理の誘惑に負けて全部同時に走らせたくなるんですけど、依存関係のあるタスクを並列実行するのは古典的なバグの温床で。KOL間は独立してるから並列OK、でも「全体スタイル → 個別KOL」は直列必須。

ここで気づいたんですけど、情報の境界って「誰が何を知るか」だけじゃなくて「どの順序で情報を確定させるか」の設計でもあるんですよね。上流の品質が下流を完全に規定する。工場の組み立てラインそのもので、工程間の検査ゲートが必要になる。

## ゲートは「中身」を見ない

で、ゲートで何を検査するかがまた厄介で。

調査エージェントが「このKOLは20代女性に人気」って出してきて、それが正しいかどうかなんて次のエージェントにはわかりません。中身の正しさを判定するのはゲートの仕事じゃない。

ゲートの仕事は**成果物の存在と完全性のチェック**です。8人分のリサーチ結果が全部あるか。必須項目が埋まっているか。3人分しかないのに次の工程に進もうとしていないか。

組織論で考えたら納得しました。マネージャーの仕事は部下の仕事の中身を判定することじゃなくて、「必要なものが全部揃ったか」と「次に進めていいか」の判断。CIパイプラインのゲートと同じ構造です。CIサーバー自体がコードの品質を判定するわけじゃない。テストが通ったかどうかを管理するだけ。

```
オーケストレーターの責務:
  ✅ 成果物の存在チェック
  ✅ 前提条件の検証
  ✅ 次工程への遷移判断
  ❌ 内容の品質判断
  ❌ 工程内の手順管理
```

あと、ゲートで「揃ってない」判定が出たときの再実行粒度も設計が要ります。8人中7人のリサーチが成功して1人だけ失敗したなら、その1人だけ再実行すればいい。ただしこれは各エージェントの出力が独立している場合だけで、「全体の方向性」に関わる工程だと1つの失敗が全体に波及する。再実行の粒度も情報の境界設計の一部なんですよね。

## 「これ前の案件のやつじゃん」

正直これが一番腹が立ちました。

案件を2回流したとき、前の案件の成果物が残っていて、次の案件のエージェントがそれを「最新の成果物」として使った。前の案件のカバー画像を新しい案件でそのまま採用。ファイルの場所も名前も同じ、中身だけが古い。

人間なら「あ、これ前のやつだ」ってわかる。エージェントにはわからない。

怒りが収まってから考えたら、エージェントは悪くなくて「どの実行に属する成果物か」の情報がどこにもないのが悪い。分散システムの「キャッシュの無効化」や「古いレプリカの読み込み」と本質的に同じ問題です。

解決策はデータベースのMVCCと同じ発想にしました。MVCCは各行がどのトランザクションで書かれたかを持っていて読み取り時にチェックする仕組みで、これを成果物にも適用する。各成果物に実行IDを刻印して、ゲートは存在チェックに加えてスコープの一致もチェックする。

「全部の成果物にスコープ付けるの大変じゃない？ 忘れたら終わりじゃん」と思いますよね。そう、だからゲートをフェイルセーフにする。スコープ情報がない成果物が見つかったら通過させるんじゃなくて止める。止める方向に倒す。

最初は「前の案件のファイルを消してから走らせればいいでしょ」って思ってました。消し忘れで2回やらかして学びました。人間の注意力に依存する設計は必ず破綻します。当たり前のことなんですけど、当たり前のことに気づくのに3案件かかった。

## 修正ループは「チーム」に閉じ込める

PPTXのレビュー工程は「ここ直して」→修正→再チェック…のループです。問題はこのループのルーティング。データの問題なのか見た目の問題なのかで、直すべきエージェントが違う。

```
[ビルド] → [ビジュアル適用] → [レビュー]
                                  ↓ REJECT
                           どこに戻す？
                         データの問題 → ビルドへ
                         見た目の問題 → ビジュアルへ
                                  ↓ APPROVE
                              完成
```

「これはオーケストレーターが中身を判断してない？」と一瞬思ったんですけど、違います。レビューエージェントが「データの問題です」「ビジュアルの問題です」という分類ラベルを付ける。オーケストレーターはラベルに従ってルーティングするだけ。郵便局の仕分け担当が封筒の中身を読まずに宛先だけ見て振り分けるのと同じ。

んで、このビルド→レビューの閉ループを丸ごと1つのチームに囲い込む設計にしました。オーケストレーターがステップごとに介入するんじゃなくて、チームに裁量を渡して内部ループはチームが自律的に回す。ループ上限だけ決めておいて、超えたら人間にエスカレーション。組織で言えばプロジェクトチームへの権限委譲ですね。

## 3人と30人で同じ設計にはならない

ふと気づいたんですけど、KOL3人のときと30人のときで同じ設計でいいのか。

3人なら正直、オーケストレーターが全部見ても収まります。30人だとリサーチ結果読むだけでコンテキストが溢れる。5人のスタートアップならCEOが全部見る。50人になったらチームリーダーを置いて権限委譲する。組織論でいうスパン・オブ・コントロールそのもの。

```
少数モード: オーケストレーターが処理の一部も直接担当
多数モード: オーケストレーターは管理に専念、サマリーだけ読む
```

どちらのモードでも成果物のインターフェースは統一する。モードが変わっても後工程に影響しない。ポリモーフィズムです。

正直、多数モードのほうが設計は綺麗なんですよね。少数モードはオーケストレーターに処理を持たせてるので美しくはない。でも3人のKOLのために大仰な委任構造を組むのもオーバーエンジニアリングで、この判断が正しかったかは今でもわかりません。

## 同じ原則が再帰する

設計してて笑ってしまったことがあります。

素材制作の工程が重すぎたので分割したら、分割した各ステップにもパイプライン全体とまったく同じ設計原則が必要でした。情報の境界、ゲート、スコープ刻印、全部。大きなパイプラインの中に小さなパイプラインがある。フラクタル。

```
パイプライン全体:
  [調査] →ゲート→ [計画] →ゲート→ [制作] →ゲート→ [レビュー]

「制作」の内部:
  [素材取得] →ゲート→ [全体トーン] →ゲート→ [KOL個別×N] →ゲート→ [統合]
```

最初は過剰設計じゃないかと思ったんですけど、内部にゲートを入れずに走らせたら途中で失敗したときにどこで壊れたか切り分けられない。最初からやり直しになって非効率の極み。「1人に全部任せたら壊れた」のと同じ問題が、粒度を変えて再発する。

で、これに気づいたとき「設計がスケールするってこういうことか」と思いました。同じ原則の適用先が増えるだけで、新しいルールが増えるわけじゃない。

## マルチエージェント設計は暗黙知の形式化

```
┌─ 情報の境界 ──── 誰が何を知るか
├─ 全体→個別 ──── グローバルを先に確定
├─ 成果物ゲート ── 完全性チェックで遷移判断
├─ スコープ刻印 ── どの実行に属するか
├─ 品質ループ ──── レビュー→修正→再検証
├─ 責務の絞込 ──── オーケストレーターは管理のみ
└─ モード切替 ──── 規模に応じた実行戦略
```

人間のチームだとこういうことを無意識にやっています。「あ、これ前の案件のだ」「先に全体の方向決めてから」「これ誰に聞けばいい？」。暗黙知として機能してることを、AIエージェントでは全部データ構造とプロトコルとして形式化しないといけない。

まだゲートの検査密度の最適解は見えていなくて、ゲートを増やすと安全だけど遅い、減らすと速いけど事故る。規模が変わるたびにチューニングが要る気がしていて、この辺の塩梅は正直まだ手探りです。
