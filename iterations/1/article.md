SNSインフルエンサーへの投稿指示書を自動生成するシステムを作りました。受注書とKOLリスト（Key Opinion Leader、インフルエンサーのリスト）を入力すれば、10体のClaudeエージェントが連携してKOL別のクリエイティブ指示スライドデッキを全自動で吐き出す仕組みです。

最初に1体のエージェントで全部やらせたとき、8人目のKOLのキャプションに薬機法アウトな表現が堂々と生成されて、しばらく画面の前から動けませんでした。

## 何の話か

インフルエンサーマーケティングの指示書作成は、地味に重い仕事です。KOL1人ごとに「スタイル調査 → クリエイティブ設計 → キャプション作成 → スライド組版 → レビュー」という工程をこなす。10人いれば丸1日かかります。これを`python orchestrator.py run`の1コマンドで完走させようとして、5ラウンドにわたって設計を壊しては作り直しました。

この記事はKOLマーケティングの話でも、PPTXを操るpython-pptxの話でもありません。**複数のエージェントを自律的に動かすとき、何が壊れて、何が助けになるか**という問いへの5つの答えです。複雑な業務ワークフローをマルチエージェント化しようとしている人なら、ここで出てきた問題のどれかに必ず引っかかるはずです。

記事全体を貫くテーゼは1つ——**エージェントは忘れる、ファイルは忘れない。**

---

## Round 1: 1体に全部やらせたら、8人目でKOL1の情報が消えた

最初の仮説は「大きな1つのエージェントに全部やらせる」でした。合理的に聞こえます。「10人分の指示書を作って」と言えばいい。

動かしてみると、8人目あたりから崩れてきました。KOL-Aのクリエイティブ方針として「ピンクトーン×ナチュラル系」を最初に設定したのに、後半になってKOL-Aのキャプション出力に「クールホワイト系スタジオ演出」という指示が混入してきた。コンテキストウィンドウの物理的制約の前では、エージェントの性能は関係ありません。後半で前半を忘れる。

解決策は「分割」でした。ただしナイーブな分割じゃない。

```
[入力] ─→ [調査Agent] ─→ [設計Agent] ─→ [生成Agent] ─→ [組版Agent]
               ↓               ↓               ↓               ↓
          research.json   plan.json       content.json    binding.json
```

**1エージェント1成果物**という原則です。各エージェントは1つのファイルを出力し、次のエージェントはそのファイルだけを読んで動く。コンテキストを引き継ぐのではなく、ファイルで受け渡す。調査エージェントは入力ドキュメントを読む。設計エージェントは調査結果だけを読む。後続エージェントが前工程の生ログを全部読む必要はありません。

この段階でコンテキスト爆発は解決しました。でも別の問題が顔を出してきます。

## Round 2: グローバルとローカルを同一エージェントに解かせた

分割すると次の壁にぶつかりました。キャンペーンとしての統一感は必要だけど、KOLごとの「らしさ」も出さないといけない。全員同じ指示書にしたら、KOLのファンには刺さらない。かといってKOLごとに完全独立で設計したら、同一キャンペーンに見えなくなる。

最初のアプローチはKOL3人分を1つのプランニングエージェントに一括設計させるものでした。トークン使用量が52,000を超えたあたりで、KOL-Aのキャプションに書いたはずのNG表現制約——「完全に焼けない」「100%防御」という薬機法アウトな表現を避けるよう明記したもの——が、後半の出力から跡形もなく消えていました。

「先にグローバルを固め、次にローカルを乗せる」という2段構造が答えでした。

```
Phase A: キャンペーン共通スタイル確定
   ↓ （ブランドカラー・トーン・NG要素を1ファイルに固定）

Phase B: KOL別カスタマイズ（並列可）
   KOL-A: 共通スタイル + KOL-A個性 → A専用設計
   KOL-B: 共通スタイル + KOL-B個性 → B専用設計
   KOL-C: 共通スタイル + KOL-C個性 → C専用設計
```

「スタイルアンカー」と呼んでいます。アンカーファイルに書かれた制約は全KOLの後続エージェントに引き継がれ、個別エージェントはその範囲内でKOLらしさを最大化する。グローバル一貫性とローカル個別化を同一フェーズで解こうとするから矛盾する。時間的に分離すれば両立できます。

## ファイルを「契約書」として設計する——これが全部の核心だった

Round 1の「1エージェント1成果物」でコンテキスト爆発を解決した。Round 2の「アンカー分離」でスケールの問題を解決した。4フェーズを超えたあたりから、また別の腐敗が始まりました。**情報の劣化です。**

具体的に言います。調査フェーズでKOL-Aの`caption_direction`フィールドに「NG表現: 医師が認めた・皮膚科推薦」と書きました。5フェーズ後のキャプション生成エージェントのアウトプットを見ると、それが3箇所出てきていた。調査エージェントはちゃんと記録したのに、5体を経由する間に情報が消えていました。

最初の対処は「プロンプトに全情報を詰め込む」でした。10体分の累積情報を全プロンプトに入れたら、今度はトークン爆発しました。詰め込む → 爆発、の繰り返し。

突破口は「どう渡すか」ではなく「**どう参照させるか**」の設計問題として捉え直したことでした。

```
❌ 積み上げ方式（情報劣化する）
   PhaseA の全出力 + PhaseB の全出力 + ... → PhaseN に全部入れる
   → トークン爆発、後半のAIが前半情報を無視し始める

✅ 引き出し方式（情報劣化しない）
   各エージェントが「必要なファイルだけを読む」設計にする
   → キャプション生成エージェントは per_kol_packages.json だけ読む
   → ビルドエージェントはアセットファイルだけ読む
```

重要なのは「このファイルのこのフィールドに、このエージェントが依存する」という依存関係を**設計の最初期に明示する**ことです。NGワードは調査エージェントが`per_kol_packages.json`の`caption_direction`に書く。キャプション生成エージェントは必ずそのファイルを読む。この「契約」が成立していれば、情報は5フェーズを経由しても消えません。エージェントは忘れる。ファイルは忘れない。

### 成果物ゲート: ファイルの「存在」が次フェーズへのパスポートになる

ファイルを契約として設計することと、もう1つセットで機能する仕組みが「成果物ゲート」です。

各フェーズは「必須成果物ファイルが存在するか」を条件に次フェーズへ進みます。ファイルがなければ当該フェーズを再実行する。存在すれば先に進む。

```
Phase 4 内部フロー（実際の設計）:

  4a: Asset Acquisition ──→ source_assets_manifest.json
        │ Gate: ファイルが存在するか？
        ▼ （存在する場合のみ）
  4b: Global Style Generator ──→ campaign_cover.png / global_assets_manifest.json
        │ Gate: 3画像すべて存在するか？
        ▼ （すべて存在する場合のみ）
  4c: KOL Creative Worker × KOL数 （並列実行可）
        │ Gate: 全KOLのassets_planが存在 + run_idが一致するか？
        ▼
  4d: Asset Resolver ──→ assets_resolved.json（status=ready）
        │ Gate: status=ready かつ campaign_cover が存在するか？
        ▼ 次フェーズへ
```

ゲートの条件は「ファイルが存在する」だけじゃない。**`run_id`の照合**も入っています。

`run_id`というのは実行サイクルごとに生成されるIDで、全フェーズの成果物に刻印されます。`run_state.json`の`run_id`と各フェーズ成果物の`run_id`が一致しないときは、そのフェーズを再実行する。これが何をしてくれるか。

前回の実行で生成された「古いファイル」が残っていたとき、エージェントがそれを掴んで「済んだ」と誤判断するのを防ぎます。前の実行で生成された`global_assets_manifest.json`が残っていても、`run_id`が違えばゲートで弾く。別の`run_id`を持つ成果物は「ステールアーティファクト（古い成果物）」として無効化されます。

正直に白状すると、この判断軸を入れる前はステール成果物による誤判定に2回やられました。「なんでレビュー済みのスライドが前回実行のデータで上書きされているんだ」という状況です。3時間くらい途方に暮れていました。`run_id`照合を実装した後は、このクラスの事故はゼロになっています。

### 成果物ゲートが解いた副次問題

成果物ゲートには当初想定していなかった副次効果がありました。**冪等性（べきとうせい）**の確保です。

途中のフェーズでエラーが起きたとき、最初からやり直す必要がなくなります。失敗したフェーズだけを再実行すれば、前フェーズの成果物をゲートが確認して「ここから続きだ」と判断して動き出す。10体が直列に動くシステムで、フェーズ7でエラーが起きたときにフェーズ1からやり直す必要がないのは、精神的にも計算コスト的にもかなり助かります。

あなたが今マルチエージェントのシステムを設計しているなら、フェーズの完了条件を「ロジックが成功したか」ではなく「成果物ファイルが存在するか」で定義することを強くお勧めします。ロジックの成功はエフェメラルで、ファイルの存在は永続的です。

## Round 5: 並列化できる箇所を感覚で決めたら、クライアントに怒られた

自動化率が95%に達したころ、「どこを並列にするか」の判断を感覚でやって失敗しました。

グローバルスタイル生成とKOL調査が並列にできると思って動かしたら、KOL-Bの画像だけブランドカラーが紺系（`#1A2B3C`）で生成されていた。正しいブランドカラーはオレンジ（`#F28C38`）です。そのままクライアントに送付して、30分後に「カラーが全然違う」と連絡が来ました。

嘘でしょ。

根本原因は単純でした。`global_assets_manifest.json`が存在しないまま起動したKOL-Bエージェントが、ブランドカラーをデフォルト値で補完してしまった。グローバルスタイルが確定する前にKOL個別エージェントを動かしたのが失敗でした。

「何が先に完了している必要があるか」を**情報依存グラフ**で可視化して初めて、並列化の境界が見えてきました。

```
情報依存グラフ（簡略版）:

[入力]
  ├─→ [KOL調査] ─────────────────────────┐
  └─→ [商品調査] ──→ [統合プランニング] ──→ [グローバルスタイル生成]
                                                ↓（完了後に解放）
                            ┌───────────────────┤
                            ├─── [KOL-A クリエイティブ] ─→ [A画像生成]
                            ├─── [KOL-B クリエイティブ] ─→ [B画像生成]
                            └─── [KOL-C クリエイティブ] ─→ [C画像生成]
                                                          ↓
                                                  [アセット統合]
                                                          ↓
                                          [スライド組版 → レビュー → 出力]
```

並列にできる箇所は2ヶ所だけです。①KOL調査と商品調査（どちらも入力のみに依存）、②KOL-A/B/Cのクリエイティブ作業（グローバルスタイル完了後に全員並列スタート）。これ以外は直列です。

グローバルアンカーは必ず先行完了させ、それを参照する個別ワーカーが並列に動く。このパターンは「1エージェント1成果物」の原則と相性が良くて、各KOLエージェントが別々のファイルを出力するので書き込み競合も同時に解決されます。KOL-A/B/Cが`assets_plan_kol-a.json`/`assets_plan_kol-b.json`/`assets_plan_kol-c.json`をそれぞれ独立して出力するなら、同一ファイルへの競合書き込みは構造上起きない。

この並列化で、直列なら20分かかるクリエイティブ設計フェーズが5分で終わるようになりました。KOL10名分の全体処理時間は手作業時の45分から約13分になっています。

## 「いつ止めるか」という最も地味な設計問題

レビューループの停止条件は、全設計の中で最も地味で、かつコストに直撃する問題です。

最初はスコアベースで止めようとしました。9点以上で合格。でも「9点と8.5点の違いは何か」が定義できなくて、ループが止まらなくなりました。

深夜、ログを眺めていて冷や汗をかきました。修正ループが10回回っている。9回目も10回目もスコアは7.8のまま。APIコストを確認したら1回の実行だけで4,200円飛んでいました。壊れた。ループが。

「絶対スコア」ではなく「変化の有無」で止める、というのが答えでした。

停止条件は3つを組み合わせています。スコアが9.0以上を2連続で達成すれば「良い水準で安定した」と判断してSTOP。スコアの変化が±0.5以内を3連続で続けば「改善が止まった」と判断してSTOP。どちらも発動しなければ最大10回でハードリミットがかかります。

重要なのは「スコアが何点か」より「上がっているか」という変化パターンの監視です。停滞したループを続けても品質は上がらない。あの夜の4,200円事故も、停滞検知が6回目で動いていれば1,700円で済んだ計算になります。

## 5ラウンドを経て残った問い

5ラウンドの格闘で発見した設計パターンをまとめると、ほとんどの問題は「誰が何を書き、誰が何を読むか」の設計が曖昧なことから生まれていました。1エージェント1成果物という原則が、コンテキスト爆発・情報劣化・並列化競合を一括で解いてくれた。ファイルを契約として設計し、成果物ゲートで次フェーズへの進行を制御し、`run_id`でステールアーティファクトを弾く。この組み合わせがあれば、エージェントが何体になっても情報の「消え方」が予測可能になります。

ただ、1つだけ設計で回避しただけで根本から解いていない問いが残っています。

**複数のエージェントが同一のファイルに書き込もうとしたとき、どうなるか。**

今回はKOL-A/B/Cがそれぞれ異なるファイルを出力することで競合を回避しています。でも「共有リソースへの競合書き込み」——複数エージェントが同一サマリーファイルに結果を追記するケース——は設計で禁止しているだけで、仕組みとして防げていません。ファイルロック、書き込みキューイング、マージ戦略のどれかが必要になるはずです。

個人的には「集約エージェントを1体立てて、書き込みを逐次化する」が最もシンプルな解だと思っています。ただまだ試せていない。あなたのシステムではこの問題をどう扱っていますか？
