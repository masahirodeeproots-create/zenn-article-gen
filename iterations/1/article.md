マルチエージェントシステムの設計で最初にぶつかる壁は、エージェントの能力ではなくエージェント間の「関係性」の設計です。

8体のAIエージェントが協調して1枚のPPTXスライドデッキを生成するシステムを作りました。KOL（インフルエンサー）ごとにカスタマイズされた投稿指示書を自動で出す仕組みで、調査、デザイン、ビルド、レビューを分担します。で、作ってわかったんですけど、個々のエージェントが賢いかどうかはあんまり関係なかった。壊れるのは常に「エージェントの間」でした。

この記事ではシステムの紹介ではなく、設計の過程で見つけた原則の話をします。マルチエージェントを組むなら、たぶんどのドメインでもぶつかる話だと思います。

## 「分ければスケールする」は半分だけ正しい

最初の設計仮説はシンプルでした。調査する人、デザインする人、組み立てる人、チェックする人。マイクロサービスと同じ発想で責務ごとにエージェントを分ければいいだろう、と。

これ、半分は正しいんですよね。実際、1つの巨大なエージェントに全部やらせたら3人目のKOLあたりからクオリティが露骨に落ちました。キャプションの方向性が1人目のコピーみたいになる。コンテキストウィンドウに全情報を詰め込むと、後半は前半の繰り返しになります。

だから分割は必要。でも分けた瞬間に「何を渡すか」の設計が必要になる。マイクロサービスではサービス間のAPI仕様を明示的に定義しますよね。エージェントも同じで、「なんとなく伝わる」は通用しません。LLMベースのエージェントは長いコンテキストの中で関係ない情報に引っ張られるので、渡す情報を絞る設計が本質になります。

DDDの「境界づけられたコンテキスト」がそのまま当てはまりました。全エージェントが同じ情報を見る（＝全サービスが同じDBを見る）のはアンチパターン。

## 情報の境界は「誰が何を知るか」だけじゃない

情報を分離する方針は固まったんですけど、次に困ったのが「全体方針って誰が決めるの」という問題でした。

8人のKOLに個別の素材を作りつつキャンペーン全体のトーンを統一したい。最初はKOL担当エージェントそれぞれに「キャンペーンの方向性」を渡して個別に作らせました。バラバラになりました。あるKOLはポップで、別のKOLはシック。同じキャンペーンに見えない。

これ料理で考えるとわかりやすくて、ベースのスープを先に作ってからトッピングを変えるべきなんですよね。スープなしにトッピングだけ変えても統一感は出ない。

```
[素材取得] → [全体スタイル確定] → [個別KOL素材生成 ×N] → [統合チェック]
```

で、この構造で本当に重要なのは、全体スタイルが確定する**前に**個別制作を始めてはいけないということです。並列処理の誘惑に負けて「全部同時に走らせちゃえ」ってやると不整合が起きる。並列にしていいのは互いに独立したタスクだけで、KOL間は独立してるから並列OK、でも「全体スタイル → 個別KOL」は直列でないといけない。

ここで気づいたのが、情報の境界設計は「誰が何を知るか」だけでなく「どの順序で情報を確定させるか」の設計でもあるということでした。上流の品質が下流の品質を規定する。工場の組み立てラインと同じで、工程間に検査ゲートが必要です。

## ゲートが検査すべきは「中身」ではなく「完全性」

ゲートを置くのはいい。でも何を検査するかがけっこう難しかった。

調査エージェントが「このKOLは20代女性に人気」って出してきたとして、それが正しいかどうかなんて次のエージェントにはわかりません。もっと現実的な問題として、8人分のKOLリサーチを並列で走らせたとき、全員分揃ったかどうか。地味ですけどこれがないと後工程が中途半端な入力で動き出します。

組織論で考えたら腹落ちしました。マネージャーの仕事は中身の正しさを判定することじゃなくて、「必要な成果物が全部揃ったか」と「次に進めていいか」の判断です。

オーケストレーターも同じにしました:

```
オーケストレーターの責務:
  ✅ 成果物の存在チェック（8人分あるか？）
  ✅ 前提条件の検証（必須フィールドが埋まっているか？）
  ✅ 次工程への遷移判断
  ❌ 内容の品質判断（それは専門エージェントの仕事）
  ❌ 工程内の具体的な手順（各エージェントに任せる）
```

CIパイプラインとまったく同じ構造ですね。CIサーバーがコードの品質を判定するんじゃなくて、テストが通ったかどうかのゲートを管理する。テストの中身はテストコードの責務。

ゲートで「揃ってない」って判定されたときの再実行粒度も考えないといけません。8人中7人のリサーチが成功して1人だけ失敗したなら、その1人だけ再実行すればいい。ただし各エージェントの出力が独立している場合に限ります。「全体の方向性」に関わる工程だったら1つの失敗が全体に波及するので、部分再実行ではなく全体再実行が要る。再実行の粒度も情報の境界設計の一部です。

## 成果物にスコープを刻印する — 一番腹が立った設計ミス

ここからが正直一番腹が立った話なんですけど。

案件を2回流したとき、前の案件の成果物が残っていて、次の案件のエージェントがそれを「最新の成果物」だと思って使いました。前の案件で作ったカバー画像を新しい案件でもそのまま使っちゃう。ファイルの場所は同じ、名前も同じ、でも中身は前の案件のもの。

人間だったら「これ古いやつだ」ってわかります。でもエージェントには判断できない。怒りが収まってからよく考えたら、エージェントは悪くなくて、「どの実行に属する成果物か」の情報がどこにもないのが悪いんですよね。

分散システムで「キャッシュの無効化」とか「古いレプリカを読む」問題と本質的に同じです。解決策はデータベースのMVCC（Multi-Version Concurrency Control — 各行がどのトランザクションで書かれたかを持っていて読み取り時にチェックする仕組み）と同じ発想にしました。各成果物に実行IDを刻印して、ゲートは存在チェックだけでなくスコープの一致もチェックする。

で、全ての成果物にスコープの刻印を付けるのは大変じゃない？ 忘れたら終わりだし。だからゲートをフェイルセーフにしました。「スコープ情報がない成果物」が見つかったらゲートが止める。通過させるのではなく止める方向に倒す。

最初は「前の案件のファイルを消してから走らせればいい」と思ってました。消し忘れで2回やらかして、人間の注意力に依存する設計は必ず破綻するという教訓を得ました。まあ当たり前の話なんですけどね。当たり前のことに気づくのに3案件かかった。

## 品質の閉ループ — レビューと修正のルーティング

ここまでの話は「前工程 → ゲート → 次工程」の一方通行でした。でも最終成果物のクオリティ保証にはフィードバックループが要ります。

PPTXができあがった後のレビューは「ここ直して」→ 修正 → またチェック、の繰り返しです。問題はこのループをどう閉じるか。無限ループになりかねないし、「どこを誰が直すか」のルーティングも必要です。データの問題なのか見た目の問題なのかで直すべきエージェントが違う。

```
[ビルド] → [ビジュアル適用] → [レビュー]
                                  ↓ REJECT
                           どこに戻す？
                         データの問題 → ビルドへ
                         見た目の問題 → ビジュアルへ
                                  ↓ APPROVE
                              出力へ
```

ここで面白い設計判断がありました。ルーティングのために「データの問題か、ビジュアルの問題か」を判断しないといけないんですけど、それは「オーケストレーターは中身を判断しない」という原則と矛盾しないのか。

矛盾しません。郵便局の仕分け担当と同じです。中身は読まないけど宛先を見て振り分ける。レビューエージェントが「データの問題です」「ビジュアルの問題です」という分類ラベルを付けてくれれば、オーケストレーターはラベルに従ってルーティングするだけ。

もう1つ、ビルドからレビューまでを1つのチームとして囲い込む設計も入れました。オーケストレーターがステップごとに介入するんじゃなくて、チームに裁量を渡して内部のループはチームが自律的に回す。ループの上限だけ決めておいて、超えたら人間にエスカレーション。組織で言えばプロジェクトチームに裁量を渡す感じですね。

## スケール時のモード切替

KOLが3人のときと30人のときで同じ設計でいいのか。3人だったらオーケストレーターが全部見ても大丈夫です。30人になったらコンテキストが溢れる。会社と同じで、5人のスタートアップならCEOが全部見るけど50人になったらチームリーダーを置いて権限委譲する。組織論でいう「スパン・オブ・コントロール」の問題です。

入力の規模に応じて実行モードを切り替えるようにしました:

```
少数モード: オーケストレーターが直接処理の一部も担当
  → コンテキストに収まるのでシンプルに

多数モード: オーケストレーターは管理に専念、処理は全部委任
  → コンテキスト溢れを防ぐため、成果物のサマリーだけ読む
```

重要なのはどちらのモードでも成果物の形式を統一すること。モードが変わっても後工程に影響しない。ポリモーフィズムですね。

正直、多数モードのほうが設計は綺麗だし少数モードはオーケストレーターに処理を持たせてるので美しくはないです。でも3人のKOLのために大仰な委任構造を組むのもオーバーエンジニアリングだと思っていて、この判断が正しかったかは今でもよくわかりません。

## フラクタル構造 — 設計原則は再帰する

設計してて笑ってしまったことがあります。

素材生成の工程が重すぎたから分割したんですけど、分割した小さなステップにもパイプライン全体とまったく同じ設計原則が必要でした。素材集め → 全体トーン確定 → KOL個別 → 統合チェック。これ自体が1つの小さなパイプラインで、情報の境界もゲートもスコープの刻印も全部要る。

```
パイプライン全体:
  [調査] →ゲート→ [計画] →ゲート→ [制作] →ゲート→ [レビュー]

「制作」の内部:
  [素材取得] →ゲート→ [全体トーン] →ゲート→ [KOL個別 ×N] →ゲート→ [統合]
```

大きなパイプラインの中に小さなパイプラインがある。フラクタルです。最初は「各工程の内部まで同じ設計にするのは過剰設計では」と思ったんですけど、内部にゲートを入れずに走らせたら工程の途中で失敗したときにどこで壊れたか切り分けられない。結局最初からやり直すしかなくて非効率の極み。Round 2で1つの巨大エージェントの限界にぶつかった話が、スケールを変えて再発したわけです。

## オーケストレーターは何を「しないか」の設計

最後に、ゲート管理、モード選択、ルーティングと責務を足していくとオーケストレーター自体がモノリスになります。マイクロマネジメントの反パターンです。

やること: 実行モードの選択。工程の順序管理。成果物のゲート判定。
やらないこと: 各工程の具体的手順。成果物の内容評価。情報の加工。

工程固有の手順は各エージェントに閉じ込めて、オーケストレーターが知るのは「何が入って何が出るか」だけ。ブラックボックスとして扱う。

```
┌─ 情報の境界 ──────── 誰が何を知るか
├─ 全体→個別の順序 ── グローバルを先に確定
├─ 成果物ゲート ────── 完全性チェックで遷移判断
├─ スコープの刻印 ─── どの実行に属するか
├─ 品質の閉ループ ─── レビュー→修正→再検証
├─ 責務の限定 ──────── オーケストレーターは管理のみ
└─ モード切替 ──────── 規模に応じた実行戦略
```

最初に「エージェント分ければスケールする」って言ったとき、こんな複雑な設計論になるとは思ってませんでした。でもこの複雑さは偶発的な複雑さじゃなくて本質的な複雑さで、人間のチームワークでは暗黙知として機能していることを全部明示的にプロトコルとして形式化する必要がある。そこが大変だけど、そこにしか面白さはないという気がしています。モード切替の閾値をKOL数で決めてるのは雑すぎるかもしれなくて、コンテキスト量で動的に判定すべきかもしれない。でもまあ動いてるうちは触らないことにしてます。
