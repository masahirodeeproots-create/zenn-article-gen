3人目のKOLへの指示書が、1人目のコピーになっていました。

キャプションの方向性も、ビジュアルの雰囲気も、ほぼ同じ。KOLごとにフォロワー層もプラットフォームも違うのに、後半に行くほど指示がワンパターンになっていく。AIに全部任せたのに、人間が疲れたときと同じ劣化パターンが出てるんですよね。

これは、インフルエンサー向けの投稿指示書をAIエージェントで自動生成しようとしたときの話です。KOL（Key Opinion Leader、SNSで影響力を持つインフルエンサー）に「こう投稿してください」と渡すスライド資料を、商品調査からデザイン、組み立てまで全部AIにやらせたかった。KOL3人なら手作業で半日ですが、8人で丸1日、20人の大型案件だと2-3日。人間がスケールしない仕事の典型です。

で、最初の設計は単純でした。「エージェント分ければスケールするでしょ」と。調査担当、デザイン担当、組み立て担当、レビュー担当。マイクロサービスと同じ発想です。

**甘かった。**

## 「分ける」は出発点にすぎない

エージェントを分けること自体は間違っていません。問題は「分けた後」に始まります。

1つの大きなエージェントに全工程を任せてみました。コンテキスト（AIが参照する情報の範囲）が途切れないからシンプルだろうと。結果は冒頭の通りで、3人目から品質が崩れます。LLMは入力が長くなると後半を軽視する傾向があって、8人分のKOLリサーチと素材生成を1つのコンテキストに詰め込むと、後半のKOLが前半のコピーになる。

じゃあ分けるしかない。でも分けた瞬間に新しい問題が出ます。**「誰が何を知るべきか」の設計**です。

マイクロサービスだと、全サービスが同じデータベースを見るのはアンチパターンですよね。同じことがAIエージェントにも当てはまります。全エージェントにプロジェクト全体の情報を渡すと、エージェントは何が重要か判断できなくなる。関係ない情報に引っ張られて、的外れなアウトプットを出す。逆に情報を絞りすぎると、必要な文脈がなくて的外れになる。

ドメイン駆動設計で言う「境界づけられたコンテキスト」と同じ構造です。で、これに気づくまでに結構時間がかかりました。

答えとして落ち着いたのは、「全体方針」と「個別情報」の分離です。

```
全体方針（全員が知るべきこと）
  ├── ブランドのトーン
  ├── キャンペーンの方向性
  └── 品質基準

個別情報（担当者だけが知ればいいこと）
  ├── 特定のKOLのリサーチ結果
  ├── 特定のプラットフォーム要件
  └── 特定のビジュアル素材
```

でもこれだけじゃ足りなかった。「全体方針って誰が決めるの？」って話になるわけです。全体方針自体がエージェントの仕事で、最初のエージェントが出した方針がイマイチだったら、後工程の全員がイマイチな方向に進む。**上流の品質が下流を決める。** だから工程間に品質チェックのゲート — 「関所」みたいなもの — を置く必要がある。

ここまでは教科書的な話です。面白いのはこの先。

## ゲートは何を検査すべきか — 「司令塔」の設計で一番ハマった話

オーケストレーター（各エージェントの実行順序や成果物の受け渡しを管理する司令塔プログラム）に何をさせるか。ここが一番議論になりました。

最初は「オーケストレーターがリサーチの中身もチェックすればいいじゃん」と思ったんですよ。存在チェックだけだと「ファイルはあるけど中身がスカスカ」を見逃すから。フォロワー分析が500文字以上あるか、商品との相性スコアが含まれているか、みたいな基準をオーケストレーターに持たせようと。

これ、マイクロマネジメントでした。

KOLが20人になったら、20人分のリサーチ内容を全部読んで品質判断するの？ オーケストレーター自体のコンテキストが溢れて、まさに最初の「1つのエージェントに全部やらせる」問題に戻ります。組織で考えると当たり前の話で、マネージャーの仕事は「中身の正しさの判定」じゃなくて「必要な成果物が全部揃ったか」と「次に進めていいか」の判断なんですよね。

```
オーケストレーターの責務:
  ✅ 成果物の存在チェック
  ✅ 前提条件の検証
  ✅ 次工程への遷移判断
  ❌ 内容の品質判断（専門エージェントの仕事）
  ❌ 工程内の具体的な手順（各エージェントの仕事）
```

CIパイプラインと同じ構造です。CIサーバーがコードの品質を判定するんじゃなくて、テストが通ったかどうかのゲートだけを管理する。テスト自体の責務はテストコードにある。

ゲートで「揃ってない」と判定されたとき、部分的に再実行できるかどうかも重要です。8人中7人のリサーチが成功して1人だけ失敗したなら、その1人だけ再実行すればいい。ただし、各エージェントの出力が独立していることが前提。1人のリサーチ結果が他に影響する工程だったら、全部やり直しになりかねない。

再実行の粒度を設計するのも、情報の境界設計の一部だったんですよね。ここに気づくのに時間がかかりました。

## 「ベースのスープ」問題 — 並列処理の罠

ビジュアル素材で完全にやらかしたのがこれです。

8人のKOLに個別の素材を作るとして、キャンペーン全体のトーンは統一したい。ロゴの使い方、色味、雰囲気。で、それぞれのKOL担当エージェントに「キャンペーンの方向性」を渡して個別に作らせたんですけど、バラバラになるんですよ。あるKOLはポップで、別のKOLはシックで。

料理で言うとベースのスープを先に作ってから個別のトッピングを変えるべきなのに、各自が勝手にスープから作り始めちゃった。

```
[素材取得] → [全体スタイル確定] → [個別KOL素材生成 ×N] → [統合・整合性チェック]
```

全体スタイルが確定する前に個別制作を始めてはいけない。これはゲート設計の話と直結しています。「全体スタイルの成果物が存在する」ことが、個別制作開始の前提条件になる。

並列にしていいのは「互いに独立したタスク」だけです。個別KOL間は独立してるから並列OK。でも「全体スタイル → 個別KOL」は直列でないといけない。「全部同時に走らせちゃえ」の誘惑に負けると事故ります。

## 前の案件の亡霊

実運用で一番腹が立ったバグがこれ。

案件を2回流したとき、前の案件のカバー画像が残ってて、次の案件のエージェントがそれを「最新の成果物」だと思って使ってしまった。ファイルの場所は同じ、名前も同じ。でも中身は前の案件のもの。人間だったら「あ、これ古いやつだ」ってわかるけど、エージェントには判断できない。

よく考えたらエージェントは悪くなくて、「どの実行に属する成果物か」の情報がどこにもないのが悪い。

解決策は単純で、各成果物に実行IDを埋め込みます。成果物を読むときに「今回の実行で作ったものか？」をチェックする。古い実行のIDが付いてたら前の案件の残骸だとわかる。

「でもすべての成果物に刻印するの大変じゃない？ 忘れたら終わりじゃん」

だからゲートの検査項目に「スコープの一致チェック」を含めます。刻印を忘れたとしても、ゲートが「この成果物にスコープ情報がない」と検知して止める。忘れた場合に通過させるのではなく、**止める方向に倒す**。フェイルセーフです。

:::details 全体のアーキテクチャ
```
┌─ 情報の境界 ──────── 誰が何を知るか
├─ 全体→個別の順序 ── グローバルを先に確定
├─ 成果物ゲート ────── 完全性チェックで遷移判断
├─ スコープの刻印 ─── どの実行に属するか
├─ 品質の閉ループ ─── レビュー→修正→再検証
├─ 責務の絞り込み ─── オーケストレーターは管理のみ
└─ モード切替 ──────── 規模に応じた実行戦略
```
:::

## レビューループをどう閉じるか

PPTXができあがった後のレビュー。「ここ直して」ってフィードバックを返して、修正して、またチェックして...のループです。人間のレビュープロセスと同じなんですけど、このループをどう閉じるかが意外に難しい。

無限ループになりかねないし、「どこを誰が直すか」のルーティングも必要です。データの問題とビジュアルの問題では、直すべきエージェントが違う。

で、ここで前に決めた「オーケストレーターは中身の判断をしない」と矛盾するように見えるんですけど、実は矛盾してなくて。オーケストレーターはレビュー結果の「分類ラベル」を見てルーティングするだけ。郵便局の仕分けと同じで、中身は読まないけど宛先を見て振り分ける。レビューエージェントが「データの問題です」「ビジュアルの問題です」というラベルを付けてくれれば、オーケストレーターはそのラベルに従うだけ。

ビルド→レビュー→修正のループは、1つのチームとして囲い込む設計にしました。オーケストレーターがステップごとに管理するんじゃなくて、「ビルドからレビューまで一括で任せる」ユニットを作って、内部のループはそのユニットが自律的に回す。ループの上限は10回。超えたら人間にエスカレーション。

## 15人案件でタイムアウトした話

3人のKOLで動いてた設計をそのまま15人案件に流したら、オーケストレーターがタイムアウトしました。15人分のリサーチ結果を全部読もうとして、コンテキストに収まらなくなって。

「KOLが3人のときと30人のときで同じ設計でいいのか？」

いいわけがなくて、規模に応じて実行モードを切り替える必要があります。

少数モード（3人以下）ならオーケストレーターが直接処理の一部も担当。シンプル。多数モード（4人以上）ならオーケストレーターは管理に専念して、処理は全部サブエージェントに委任。成果物のサマリーだけ読む。

重要なのは、どちらのモードでも成果物の形式は統一すること。外から見た形は同じだけど中の動き方が違う。後工程のエージェントはモードの違いを意識しなくていい。

正直、こんなに設計が複雑になるとは思ってませんでした。「エージェント分ければOK」からここまで来た。で、もう1つ嫌な発見がありまして。

素材を作る工程が重すぎて、全部1つのエージェントにやらせたら、途中で失敗したときどこで失敗したかわからない。しかもコンテキストが長すぎて後半の品質がまた落ちる。

...最初の問題と同じじゃん。

大きなパイプラインの中に小さなパイプラインがある。入れ子構造。で、どっちの層にも同じ原則が必要 — 情報の境界、ゲート、スコープの刻印、全部。パイプライン全体の設計原則が、各工程の内部にもそのまま適用される。

## まだわかっていないこと

マルチエージェントシステムの設計は「エージェントの能力」の設計じゃなくて「エージェント間の関係性」の設計です。個々のエージェントが何をできるかより、エージェント同士がどう情報を渡し、どこでチェックし、どう修正サイクルを回すかのほうがはるかに重要でした。

人間のチームワークでは暗黙知として機能していること — 情報の境界、品質チェック、責任の委譲 — を、AIエージェントでは全部明示的に設計しないといけない。「あうんの呼吸」はAIには通じません。

ただ、この設計で50人のKOL案件が来たら本当にスケールするのかは、正直まだ確信がありません。ゲートが多すぎて全体の処理時間が爆発しないか。エージェント間の受け渡しでデータが劣化していかないか。やってみないとわからない部分がまだ山ほどあります。
