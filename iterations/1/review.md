# Review - Iteration 1

## 感想

冒頭は良い。「3人目のKOLへの指示書が、1人目のコピーになっていました」で引き込まれる。「甘かった。」の極短文も効いている。前半はテンポよく読める。

ただ、中盤から後半にかけて「概念の羅列」感が出てくる。情報の境界→ゲート→ベースのスープ→スコープ→レビューループ→モード切替→入れ子構造...と7つの概念が順番に並ぶ構成になっていて、どれが「一番面白い話」なのかわからない。全セクションが似たような深さ（15-25行）で並んでいて、カタログ的。

「前の案件の亡霊」セクションは面白いのに短すぎる。もっと掘れるはず。逆に「レビューループをどう閉じるか」は既に述べた原則の応用であり、あっさり済ませてよかった。

文体は概ねです/ます調でカジュアルが混ざっていて自然。ただ「ここに気づくのに時間がかかりました」が2回出てくるのが気になる。

専門用語の説明は丁寧。初見の読者でも追える。

## ベンチマークとの差分

ベンチマーク「ハーネスエンジニアリング」:
  "2ヶ月で21体のAIエージェントによる自律開発パイプラインを構築した。GitHub Issueに要求を書いて寝れば、朝にはマージレディのPRが上がっている。"
  → 冒頭1文で「何を作ったか」「何が嬉しいか」が圧縮されている。読者の好奇心を一発で掴む

生成記事:
  "3人目のKOLへの指示書が、1人目のコピーになっていました。"
  → 失敗から入るのは良いが、3文目まで読まないと「何の話か」がわからない。冒頭の引き込みは強いが文脈の提供が遅い

---

ベンチマーク「イルシル CRA→Vite移行」:
  記事全体の40%以上をCRA→Vite移行の泥臭い作業に費やし、残りのメモリリーク調査等は従属的な扱い
  → 1つの核心テーマに集中。読者は「この記事は○○の話だ」と早期に把握できる

生成記事:
  7つの概念（情報境界、ゲート、ベースのスープ、スコープ、レビューループ、モード切替、入れ子構造）がそれぞれ15-25行で均等に並ぶ
  → 看板料理がどれかわからない。カタログ的

---

ベンチマーク「Claude Code DDD」:
  "「AIにアプリを作らせようとしたが、仕様を勝手に変えられた」「文脈を忘れてバグを埋め込まれた」そんな経験はありませんか？"
  → 読者への直接的な問いかけ。共感ポイントが明確

生成記事:
  読者への問いかけが記事全体で0回。終始「自分の体験」を語る一方通行
  → 読者との対話感がない

## 次に直すべき1つ

**概念を2-3個に絞り、核心1つに記事全体の40%以上を割く。** 現状7つの概念が均等に並んでいるのが最大の問題。「オーケストレーターの責務設計」か「前の案件の亡霊（スコープ管理）」のどちらかを核心として深掘りし、残りは圧縮するか:::detailsに移す。
