# Review - Iteration 1

## 感想

全体としてはちゃんと読める記事になっている。マルチエージェントの設計課題を順番に紹介していく構成で、各セクションで何を学んだかは伝わる。

ただ、正直に言うと「面白い」とは思わなかった。読み終わって「なるほど、勉強になった」とは思うけど「自分もやってみたい」とは思わない。理由は明確で、**体験記ではなく設計解説になっている**。設計課題→解決策が整然と並びすぎていて、試行錯誤の生々しさがない。

冒頭は良い。「5人目のコピペ問題」から入って、「分ければスケールする」は半分嘘だったという展開。ここは引き込まれた。でもそこから先は「知識の境界の話」「ゲートの話」「ステールの話」「画像分割の話」「チーム化の話」「モード切り替えの話」が均等に並んでいて、カタログ感がある。

最後まで読んだけど、どの概念が一番面白かったのか、著者が一番苦労したのはどこなのか、メリハリがわからない。

## ベンチマークとの差分

ベンチマーク「Claude Code に全部賭けて - 5万円吹っ飛んだ実録」:
  "全部で 5 万円くらい気づいたら吹っ飛んだよ！"
  → 冒頭1文で衝撃。具体的な金額。読者が「え？」となる

生成記事:
  "インフルエンサー向けの投稿指示書を自動生成するシステムを作りました。"
  → 説明的。丁寧だけど衝撃がない。context_brief的な情報提供が長い

---

ベンチマーク「確定申告を自動化する Agent Skill "shinkoku"」:
  "確定申告、めんどくさくないですか？ ...この苦行をもう繰り返したくない。"
  → 読者の共感を直撃する問いかけ → 動機が1段落で完結。そこから「できること」に素早く入る

生成記事:
  各セクションが20-30行で均等。「手作業のつらさ」→「最初の設計」→「知識の境界」→...と7セクションが同じ深さで並ぶ
  → 全部同じ深さなので、どこが核心かわからない。style_guideの「核心1つに40%以上」に反している

---

ベンチマーク「チームのCLAUDE.mdが勝手に育つ」:
  "スラッシュコマンドを使う方法は、この仕組みのことをよく知っている人が使う分には便利です。でも、チーム全体で確実に運用するとなると課題があります。"
  → 問題を1文で圧縮。すぐに具体的な課題を箇条書きで3つ。テンポが速い

生成記事:
  「ゲートは2層必要だった」のセクション: 問題提起→解決策が教科書的に整理されすぎ。驚きや苦労の痕跡がない
  → 「2層構成にしました」の前に、1層だけのときに何が起きたかの具体例がほしい

## 次に直すべき1つ

**概念の深さに濃淡をつける。** 現在7つの概念（知識の境界、パイプライン、ゲート、ステール、画像分割、チーム化、モード切り替え）が均等に20-30行ずつ並んでいる。「知識の境界」を核心として記事の40%以上を使い、残りは圧縮するか:::detailsに移すべき。タイトルが「誰が何を知るべきかの設計」なのだから、そこに全力を注ぐ。
