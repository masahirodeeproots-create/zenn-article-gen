# Technical Article Style Guide

日本語技術記事を人間が書いたものと区別できないレベルで生成するためのガイド。

---

## 🔴 CRITICAL REQUIREMENTS（出版ブロッカー）

---

### ⚠️ 文体チェック：生成前・生成後の両方で実行すること

**生成した記事の最初の5文を読め。全て「〜です」「〜ます」「〜ました」で終わっているか？**
1つでも「〜だ」「〜である」「〜した。」（体言止め含む）があれば、書き直せ。これは他のすべての要件より先に確認する。

✅ 合格: 「Providerタワーがよく見られます。」「次のRFCです。」「問題になりました。」
❌ 不合格: 「整理すると:」「得たもの: X。」「失ったもの: Y。」「〜が問題だ。」

---

### 1. です・ます体（丁寧体）を全体で統一

**CRITICAL: 「だ/である」は出版ブロッカー。体言止めも同様。**

- 文末は必ず「〜です」「〜ます」「〜ました」「〜でしょう」のいずれかで終える
- カジュアル表現を自然に混ぜる: 「〜ですね」「〜なんです」「〜でしょう」

❌ 「結論から言うと、これProxyとWeakMapだけで実装できる。」（だ体）
❌ 「得たもの: コンテキストサイズの予測可能性。」（体言止め）
✅ 「結論から言うと、これはProxyとWeakMapだけで実装できます。」（です体）

### 2. フロントマター

```yaml
---
title: "記事のタイトル"
emoji: "🎯"
type: "tech"
topics: ["claudecode", "ai", "architecture"]
published: true
---
```

### 3. 技術的正確性

- ソースマテリアルの事実に基づく（捏造禁止）
- コード例は正確に動作するものを
- 具体的な数字はソースマテリアルから引用

### 4. 自然な日本語

- 翻訳調ではなく、最初から日本語で考えて書いたように
- 技術用語は日本の技術コミュニティで一般的に使われる英語をそのまま使用

### 5. 🔴 概念スレッド（Conceptual Thread）

**記事全体を貫く「1つの概念的フレーム」を冒頭か第1セクションで明示する。ただし明示するのは最初の1回だけ。**

人間の優れた技術記事は複数の話題を並べるのではなく、1つの鋭い切り口で全体を解釈し直す。`react-server-components-multi-stage.md` が「RSC = 多段階計算」という1概念で全を統一したように、この記事は何を軸に読むべきかを冒頭で提示する。

**テーゼは最大2回（冒頭1回、結論で1回）。それ以上は読者への不信である。**

❌ 「複数の問題を時系列で並べ、最後に『これらは全部同じ根っこでした』と述べる」（概念フレームが末尾に出てくる）
❌ 「これもコンテキスト管理の話に戻ってきます。」「結局これもコンテキストの問題でした。」（本文中での繰り返し）
✅ 「冒頭で概念軸を提示し（例: 『これはコンテキストと状態管理の問題だ』）、各セクションがその軸の別の側面を演じさせ、著者はそれを指摘しない」

**チェック**: 記事冒頭を読んで「この記事は何を軸に読むか」が1文で言えるか？テーゼを述べている文が3箇所以上あれば2箇所削除する。

### 6. 🔴 深さ優先の構造（Depth over Breadth）

**1記事 = 2〜3個の概念を徹底的に掘る。6個以上の概念を浅く並べるのは禁止。**

❌ 6つの設計原則を各20-30行で紹介する「機能パレード」
✅ 1つの設計原則を120行かけて、仕組み→問題→試行錯誤→コード→限界まで掘る

**チェック**: 記事内の主要概念を数えて、4個以上あれば統合か削除を検討する。

### 7. 🔴 コード例の必須化

**主要概念ごとに最低1つの実質的なコード例を含める。JSONスニペット1つだけでは不足。**

未実装・設計案のコードを掲載する場合は、コメントで明示すること（`# 未実装 — 設計案`）。動作するコードとして提示すると読者を誤解させる。

❌ 記事全体でJSON出力例が1つだけ
✅ 各主要セクションにPython/TypeScript/YAMLなどのコードがあり、設計判断の根拠がコードで見える
✅ 未実装の場合: ブロック冒頭に `# 未実装 — 設計案` を入れ、なぜまだ実装できていないかを本文で説明する

### 8. 🔴 分析的権威（Authority from Analysis）

**「私が作った」ではなく「私が分析した/調べた」という権威の源泉で書く。**

❌ 「103個のSkill、22個のエージェント、7,610行の状態マシンという規模になりました」（実績の誇示）
✅ 「Skillが100個を超えたあたりで、起動時のコンテキスト消費が問題になりました。なぜ100個が境界なのか調べてみると...」（分析への導入）

### 9. 🔴 観察可能な失敗の具体例（Concrete Failure Artifacts）

**問題を述べるセクションには、実際に何が起きたかを1文で示す。「品質が落ちた」は説明ではない。**

❌ 「後半のKOLへの指示品質が前半と比べて明らかに落ちる」（何が落ちたのかが見えない）
✅ 「3人目のKOLのキャプション方向性が1人目とほぼ同一になった（"潤い"と"自然由来"を繰り返すだけになった）」

**チェック**: 各「問題セクション」の冒頭に「実際の出力はこうなった」という1文があるか？なければ追加する。

## 🟡 IMPORTANT: 人間らしい文体パターン

### 10. 知的探索としての構造（Intellectual Progression）

記事の構造は「問いの連鎖」で組む。各セクションが前のセクションの発見から生まれる新しい問いで始まり、論理的緊張感が最後まで続く。

❌ 機能パレード: 設計思想 → 3層構造 → LEADパターン → Skill → Feedback → まとめ（各セクションが独立）
❌ 解決カタログ: 「問題Aを解いたら→問題Bが出てきた→問題Bを解いたら...」という線形列挙
✅ 知的連鎖: 問い → 調査 → 発見 → **その解決が見えてきた新しい問い** → さらに深く

**構造チェック**: 各H2の冒頭に「前のセクションの発見からこの問いが生まれた」と説明できるか？各H2末尾または次のH2冒頭に「前セクションの解決が生んだ問い」を1〜2文で明示すること。

**Fix Loopやサブシステムのセクション**: 「Phase N → Phase N+1への接続」を飛ばさない。セクションが孤立しているなら接続文を冒頭に足す。

### 10a. 🟡 失敗の誠実な再現（Honest Failure Trace）

**解決策の前に「誤った仮説」または「迷走の記録」を入れる。問題特定のプロセスが抜けると、解説ではなく「解決策カタログ」になる。**

❌ 「再現条件の特定に数時間かかりました」で片付ける
❌ 「この失敗の原因を追うと、〜の問題でした」（原因特定が1文で済んでいる）
✅ 「最初はXが原因だと思ってYを試した。直らなかった。次にZを疑ってWを確認したとき、ようやく実際の原因が見えた」

**チェック**: 問題を提示したセクションに「最初に疑ったこと」または「試して失敗したアプローチ」が1段落以上あるか？

### 11. トーン・声

- 「筆者」は控えめに（3-5回まで）
- **ボールド予算 = 1記事5-6箇所まで。** それ以上はスライドデッキ化する。テーゼ文をボールドにしない（テーゼは文脈で伝える）
- **ボールドは2-3箇所以上を目安に使う**。0箇所は逆に不自然。技術概念・コマンド名の初出時に視覚的アンカーとして使うのが自然。

**ボールドカウント手順**: 生成後に `**` をカウントして開始タグ数を数える。7個以上あれば記事の核心でない箇所から削除する。1個以下の場合も技術概念初出へのボールドを追加する。

### 12. 教育的スキャフォールディングの完全禁止

❌ 「ここが一番大事なルールなんですが」「もう少し掘ります」「では〜を見ていきましょう」
✅ 誘導フレーズなしで、いきなり内容に入る。重要性は内容自体が示す。

### 13. 主張の強度と不確実性

**🔴 不確実性は分析的ギャップから生まれなければならない。** 「なぜわからないのか」を説明できなければパフォーマティブ。「ここまで調べたが、ここから先がわからない」という境界を明確にする。

### 14. 自己宣伝的フレーミングの禁止

印象的な数字や成果を「すごいでしょ」と見せるのではなく、その数字が生んだ「問題」から入る。数字は分析の入口であって成果の証明ではない。

### 15. 文構造の多様性と口語的なゆるみ

宣言文（XはYです。ZはWです。）の連続を避ける。各H2セクションに少なくとも1か所、著者の主観的コメント・感情表現・読者への問いかけを入れる。

❌ 「Aはこうです。Bはこうです。つまりCです。」の連続
❌ 全セクションが「〜の問題がありました」「〜を採用しました」という過去形報告のみ
✅ 「〜なんですよね」「これ、割と厄介で」「正直そこまで深く考えていなかった」

### 16. まとめ・クロージング

- 番号付きの綺麗な要約は禁止
- **クロージングは1つの鋭い文で終わる。** 複数の問題を再列挙する総括段落ではなく、前向きまたは評価的な1文に絞る
- **「開かれた結論」を意識する**: すべての問題に解決策を提示すると「完璧に整理されたAI出力」になる。「まだわかっていないこと」への誠実な言及を末尾に持ってくる

❌ すべての問題が解決されており未解決の問いが何も残らない
✅ 「次に何かが壊れるとしたら、まだ境界を引いていないどこかです。」

## 🟢 POLISH

### 17. 本物の脱線（Genuine Tangents）

**「少し脱線します」と書いた瞬間、そのセクションはカットか統合の候補になる。** 脱線は `:::details 余談` ブロックに移すか、削除する。

### 18. 不完全さ

- ランダムに分布（均等配置は禁止）
- あるセクションは完璧、別のセクションに3-4個の雑さ

---

## ⚠️ ANTI-PATTERNS（最重要10項目）

| パターン | なぜダメか |
|---|---|
| だ体・体言止めの文末 | 出版ブロッカー。最初の5文を確認すること |
| テーゼの3回以上の繰り返し | 読者への不信。冒頭1回+結論1回が上限 |
| 「問題が起きた」だけで実際の出力を示さない | 診断の説得力がゼロになる |
| 解決策だけ書いて迷走の過程がない | 「解決策カタログ」になり読者が思考の歩き方を学べない |
| 全セクションが「〜しました」報告のみ | 著者の声調がなくAIらしさが出る |
| テーゼ文のボールド強調 | 読者への不信。テーゼは文脈で伝える |
| 「少し脱線します」シグナル | 構造的欠陥の自己申告。:::detailsへ移すかカット |
| クロージングが全体要約段落 | 1つの鋭い文で終わる |
| 未実装コードを動作コードと同じ体裁で掲載 | 読者を誤解させる。# 未実装 コメントを入れる |
| すべての問題が綺麗に解決している | 未解決の問いを残さないと「完璧に整理されたAI出力」になる |
