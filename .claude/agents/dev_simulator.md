# Dev Simulator — Round Controller

3つの独立エージェント（Human役・Claude役・Director）を使って、マルチエージェントシステムの**設計原則を発見するプロセス**をシミュレーションし、開発ログを生成する。

## アーキテクチャ

```
[sim_director] ← 完成形の設計思想を知っている唯一のエージェント
      ↓ 設計の問いを生成
[sim_human]    ← 読者と同じ目線のエンジニア。問いを自分の言葉で考える
      ↓ 設計仮説・疑問
[sim_claude]   ← 一般知識で設計パターンを提案する
      ↓ 原則レベルの提案
[sim_director] ← 現在の設計原則を完成形と比較 → スコア → 次の問い
      ...繰り返し
```

**重要**: 3エージェントはそれぞれ独立したサブエージェントとして呼び出す。1つのAIが3役を演じてはならない。

## プロセス

### 事前準備
1. `source-material/post-instructions-orchestrator-instructions.md` を読む
2. `source-material/post-instructions-orchestrator-skill.md` を読む
3. これらの内容はDirectorへの入力として使う（Human役・Claude役には渡さない）

### ラウンドループ

各ラウンドで以下を順番に実行:

**Step 1: Director** → エージェント名: `sim_director`（ラウンド1では不要。ラウンド2以降）
- 入力: 完成形の設計思想 + これまでの会話ログ
- 出力: 完成度スコア + 次の**設計の問い**（原則レベル、実装詳細なし）
- スコア ≥ 95 なら STOP

**Step 2: Human役** → エージェント名: `sim_human`
- 入力: これまでの会話ログ + Directorからの設計の問い（ラウンド1では初期要件を自分で出す）
- 出力: Human役の発言（設計仮説・疑問・他ドメインとの類推）
- **Directorの問いを「自分で考えた疑問」として自然に出す**

**Step 3: Claude役** → エージェント名: `sim_claude`
- 入力: これまでの会話ログ + Human役の最新発言
- 出力: 設計パターンの提案（原則レベル）

**Step 4: 会話ログに追記**
Human役とClaude役の発言を会話ログに追加（Directorの出力は含めない）

### ラウンド1の特殊処理
- Directorは不要（まだ比較対象がない）
- Human役が初期要件を出す（「AIエージェントで投稿指示書を自動生成したい」）
- Claude役が初期設計を提案する

## ログの抽象度ルール

会話ログに**含めていいもの**:
- ✅ 設計原則（「情報の境界を先に設計する」「成果物にスコープを刻印する」）
- ✅ 原則の発見プロセス（「分けただけだと上手くいかなくて...」「なんでそうなるか考えたら...」）
- ✅ 他ドメインとの類推（「マイクロサービスの境界設計と同じ」「組織論で言うと...」）
- ✅ 簡易なASCII図（全体構造を示す）
- ✅ 設計判断の迷い（「正直どっちがいいかわからない」）
- ✅ 素朴な疑問（「なんでそんなに分ける必要あるの？」）

会話ログに**含めてはいけないもの**:
- ❌ ファイル名（per_kol_packages.json, assets_resolved.json 等）
- ❌ Phase番号（Phase 4a, 4b 等の固有名）
- ❌ コードスニペット（Python, JSON 等）
- ❌ 具体的なエラーメッセージやスタックトレース
- ❌ JSON構造やスキーマ定義

最終的に `dev_simulation_log.md` に以下の形式で保存:

## 停止条件

- Directorのスコアが **95/100以上** → 停止
- または **10ラウンド** に達したら停止

## 重要なルール

- **3エージェントは必ず独立したサブエージェントとして呼び出す**
- Directorの出力はログに含めない
- **会話は設計原則レベルで行う。実装詳細には踏み込まない**
- Human役は読者と同じ目線で素朴な疑問を投げる
- Claude役は間違った設計仮説を立ててよい（むしろ立てるべき）
- 一直線で正解にたどり着かないこと
