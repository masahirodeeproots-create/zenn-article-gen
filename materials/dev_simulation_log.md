# Development Log

> **この記録について**: 10体のAIエージェントを束ねて投稿指示書を全自動生成するシステムを設計した5つのラウンドの記録。中心にあった問いはシンプルだった——**「後段のエージェントが、前段で決めたことを知らない」という問題をどう解くか**。エージェントを増やすほど情報が劣化し、コストが爆発し、並列化が競合する。この記録はその格闘の跡だ。

---

## なぜこの問題はあなたにも起きるのか

Ambient Agent、claude -p によるサブエージェント呼び出し、Agent Teamsがトレンドになっている今、「エージェントを複数体動かしてみた」という体験は珍しくない。だが、こんな失敗をしたことはないだろうか。

- フェーズ1で決めたNGワードを、フェーズ5のエージェントが平気で使う
- 8体目が1体目の方向性を完全に忘れ、正反対の出力をする
- レビューループが止まらず、気づいたらAPIコストが¥4,200飛んでいた

これはKOLマーケティングに限った話ではない。**エージェント数が増えるほど、情報はコンテキスト上で劣化する**。この記録で扱うのは、その普遍的な問題への5つのアプローチだ。

---

## Round 1: 初期の設計仮説
**自動化率: 約10%（入力受け取りのみ自動、あとは全て手作業）**

👤 Human:
KOLへの投稿指示書を自動生成したい。今は手作業で、KOL1人ごとに「スタイル調査 → クリエイティブ方向設計 → キャプション作成 → スライド組版 → レビュー」という工程を踏んでいる。KOLが10人いると全員分で丸1日かかる。この多段階プロセスを一括で自動化したい。

最初の仮説は「大きな1つのAIに全部やらせる」こと。入力を渡して「スライドを作って」と言えばいいのでは？でも実際にやってみたら、一個のエージェントが全工程を抱えるとコンテキストが爆発した。KOL10人分を一括で投げたとき、8人目あたりでKOL1の「ピンクトーン×ナチュラル系」という方向性を完全に忘れ、KOL1のキャプションに「クールホワイト系スタジオ演出」という指示が混入してきた。出力を見たとき背筋が凍った。KOL1のファン層は「お姉さん系」ではなく「等身大ナチュラル系」なのに、正反対のディレクションになっていた。「伝言ゲーム問題」の内側バージョン、とでも言おうか。

これはKOLの文脈でたまたま起きた問題ではない。**1体のエージェントに10個の仕事をさせれば、後半で前半を忘れる**。コンテキストウィンドウの物理的な制約の前では、エージェントの性能は関係ない。

🤖 Claude:
「1体に全部やらせる」から「フェーズごとに専門家を立てる」への転換が自然な次の一手です。

設計原則として「1エージェント1成果物」を提案します。各エージェントは一つのファイルを出力し、次のエージェントはそのファイルを読んで動く。エージェント間の情報伝達はコンテキスト継承ではなくファイル渡しで行う。

```
[入力] ──→ [調査Agent] ──→ [設計Agent] ──→ [生成Agent] ──→ [組版Agent] ──→ [出力]
              ↓               ↓               ↓               ↓
          research.json   plan.json       content.json    binding.json
```

コンテキスト爆発を防ぐ理由は「各エージェントが自分の仕事に必要な情報だけを読む」から。調査エージェントは入力ドキュメントを読む。設計エージェントは調査結果だけを読む。後続エージェントが前工程の生ログを全部読む必要はない。「ジャストインタイム取得」の原則がここに効いてくる。

✅ Design Insight: コンテキスト爆発は「1体に全部持たせること」から生まれる。情報はファイルに書き出し、各エージェントは自分の責務に必要なファイルだけを読む設計が基本形。

## Round 2: グローバル一貫性 vs KOL個別化の矛盾
**自動化率: 約30%（調査フェーズが自動化。設計・生成・組版はまだ手作業）**

👤 Human:
エージェント分割の方向性は腑に落ちた。でも次の壁にぶつかった。キャンペーンとして統一感は必要だけど、KOLごとの「らしさ」も出さないといけない。全部同じ指示書にしたら、KOLのファンには刺さらない。かといってKOLごとに完全独立で設計したら、同一キャンペーンに見えなくなる。

最初のアプローチ：KOLを全員まとめて1つのプランニングエージェントに渡した。KOL3人分を一括設計させたらトークン使用量が約52,000に達し、プランナーの出力が5,000字を超えたあたりから前半のKOL-Aの詳細が消えていた。具体的には、KOL-Aの `caption_direction` に書いたはずの「NG表現: 完全に焼けない・100%防御」という制約が、後半の出力では跡形もなくなっていた。10人にスケールしたら、もっと惨事になることは明らかだった。

🤖 Claude:
「先に共有の型を作り、後から個別の色を乗せる」という2段構造が有効です。

```
Phase A: キャンペーン共通スタイル確定
   ↓（ブランドカラー・トーン・NG要素などを1ファイルに固定）

Phase B: KOL別カスタマイズ（並列可）
   KOL-A: 共通スタイル + KOL-A個性 → A専用設計
   KOL-B: 共通スタイル + KOL-B個性 → B専用設計
   KOL-C: 共通スタイル + KOL-C個性 → C専用設計
```

「スタイルアンカー」という考え方です。キャンペーン全体のビジュアルアイデンティティを先に1ファイルに固定し、それをすべての後続エージェントが「制約」として参照する。KOL個別エージェントは「このアンカーの範囲内でKOLらしさを最大化せよ」という命令になる。

これにより「統一感は共有ファイルで保証」「個性はエージェントの裁量に委任」という分離が生まれる。

もう一つの発見：KOL調査（何者か）とKOLクリエイティブ設計（どう表現するか）を分けると良い。前者は事実収集、後者は解釈。混ぜると品質が落ちる。実装すると、`per_kol_packages.json` の構造は自然とこうなった：

```json
// per_kol_packages.json（1 KOL分の抜粋）
{
  "kol_slug": "sakura_m",
  "selected_platform": "instagram_reel",
  "render_medium": "semi_real_anime",
  "visual_direction": {
    "color_tone": "ナチュラルウォーム",
    "lighting": "自然光",
    "mood": "等身大感",
    "avoid": "スタジオ撮影風・クール系"
  },
  "caption_direction": "【必須訴求内容】\n日差しが強い季節でも素肌感を守りながらUVケア。\n\n【必須ワード】\nUV、スキンケア感覚、もっちりテクスチャー\n\n【NG表現】\n「完全に焼けない」「100%防御」等の過大表現",
  "caution_points": ["景品表示法・薬機法遵守", "効果効能の過大表現禁止", "#PRステマ防止"]
}
```

グローバルスタイル（`visual_direction`）は全KOL共通のアンカーから派生しつつ、KOLの個性（`render_medium`）だけが個別エージェントの判断で上書きされる。

✅ Design Insight: グローバル一貫性とローカル個別化は同一フェーズで解こうとするから矛盾する。「先にアンカーを作り、アンカーを制約として個別化する」という時間的分離で両立できる。

## Round 3: 「いつ止めるか」問題 — ループコストの現実
**自動化率: 約55%（設計フェーズまで自動化。生成・レビューループが未制御）**

👤 Human:
設計の方向性はクリアになってきた。でも別の問題が出た。「レビュー→改善→レビュー」のループをどこで終わらせるか分からない。

実際に回してみると：
- 止めなさすぎ → 同じ品質の成果物が何度も生成されてコスト爆発。最悪のケースでは1実行で¥4,200（API料金換算）飛んだ。10回以上ループして9回目も10回目もスコアは7.8のまま動かなかった
- 止めすぎ → 低品質のまま完了して成果物として使えない。スコア5.0で打ち切ったら、キャプションに必須ワードが半分しか含まれていなかった

スコアリングを入れようとしたが「どのスコアなら合格か」も分からない。9点と8点の違いが定義できない。

🤖 Claude:
「絶対スコア」ではなく「変化の有無」でループを制御する発想が効きます。

```
停止条件の組み合わせ:

① 品質閾値: スコア ≥ 9.0 が 2連続 → 「良い水準で安定した」と判断してSTOP
② 停滞検知: スコア変化 ±0.5 以内が 3連続 → 「改善が止まった」と判断してSTOP
③ ハードリミット: max 10回 → 「どんな場合もここで打ち切り」

①②③ のどれかが先に発火したら止める。
```

直感に反するが、重要なのは「何点取ったか」より「上がっているか」。停滞したままループを続けても品質は上がらない。停滞検知は「無駄回しを早期終了する」コスト最適化装置として機能する。前述の10回ループも、停滞検知が3連続±0.5以内を検出した時点（6回目）で止めていれば、コストは約¥1,700で済んだ計算になる。

もう一つ：素材の品質とアウトプットの品質は別サイクルにする価値がある。素材（調査・設計）の品質が低いまま生成ループに入ると、どれだけ生成を改善しても上限が低い。2段階のPDCA（素材ループ → 生成ループ）が合理的。

失敗体験として：最初は生成ループだけに注力して品質が頭打ちになった。素材フェーズに戻って調査品質を上げたら、生成品質が一気に跳ね上がった。「原因は下流ではなく上流にある」。

✅ Design Insight: ループ停止は「良い状態が続いた（閾値×連続）」か「変化が止まった（停滞検知）」の二軸で判断する。絶対スコアの定義より「変化パターンの検出」の方が実装も品質判断も安定する。

## Round 4: 情報劣化との戦い — ファイルを「契約」にする
**自動化率: 約70%（生成まで自動化。ただし後段で情報漏れが頻発）**

👤 Human:
エージェント10体が順番に動くと「前フェーズで決めたことを後フェーズが知らない」問題が繰り返し出た。調査フェーズでKOLのNGワードを記録したのに、キャプション生成フェーズでNGワードを平気で使う。

具体的には、KOL-Aの調査でNGワードとして記録した「医師が認めた」「皮膚科推薦」という薬機法アウトな表現が、5フェーズ後のキャプション生成で普通に出てきた。実際のキャプション本文に3箇所も混入していた。エージェントが5体を超えたあたりから急激に情報劣化が起き始めた。

最初は「プロンプトに全情報を詰め込む」で対処しようとした。でも10体分の累積情報を全部のプロンプトに入れるとトークン爆発した。

🤖 Claude:
「すべてを伝えようとする」から「必要なものだけを取りに行かせる」への転換が鍵です。

情報伝達の問題は「何を渡すか」ではなく「どう参照させるか」の設計問題です。2つのアプローチを比較してみます：

```
❌ 積み上げ方式（情報劣化しやすい）
   PhaseA の全出力 + PhaseB の全出力 + ... → PhaseN のプロンプトに全部入れる
   → トークン爆発 + 後半のAIが前半情報を「無視」し始める

✅ 引き出し方式（情報劣化しにくい）
   各フェーズが「必要なファイルだけを読む」という設計にする
   → Captionエージェントは per_kol_packages.json（NGワード含む）だけを読む
   → BuildエージェントはAssetsファイルだけを読む
```

ファイルを「情報の置き場所」ではなく「エージェント間の契約書」として設計することが重要です。「このファイルのこのフィールドに、このエージェントが依存する」という依存関係を明示的に設計する。NGワードは調査エージェントが `per_kol_packages.json` の `caption_direction` に記録し、キャプションエージェントが必ず読む契約になっていれば、情報は劣化しない。

この「契約」が機能しているかを確認するため、`assets_resolved.json` で最終アセット統合が正しく行われているかをゲートとして使った：

```json
// assets_resolved.json（status=ready でなければ次フェーズに進まない）
{
  "status": "ready",
  "global_assets": {
    "campaign_cover": { "asset_id": "campaign_cover", "path": "cache/images/campaign_cover.png", "status": "ready", "aspect_ratio": "16:9" },
    "title_cover":   { "asset_id": "title_cover",   "path": "cache/images/title_cover.png",   "status": "ready", "aspect_ratio": "21:9" },
    "product_main":  { "asset_id": "product_main",  "path": "cache/images/product_main.png",  "status": "ready", "aspect_ratio": "9:16" }
  },
  "per_kol_assets": {
    "sakura_m": {
      "kol_scene0": { "asset_id": "kol_scene0_sakura_m", "path": "cache/images/scenes/kol_scene0_sakura_m.png", "status": "ready" },
      "kol_scene1": { "asset_id": "kol_scene1_sakura_m", "path": "cache/images/scenes/kol_scene1_sakura_m.png", "status": "ready" }
    }
  },
  "warnings": []
}
```

`status: partial` や `warnings` に何かあれば次フェーズに進まない。このゲートを設けてから「NGワードが後段で使われる」事故がゼロになった。

迷い：フェーズが増えるほど「何が正とするか」が曖昧になる。後から「このフィールドは誰が書くの？」という問いが頻発した。フィールドの所有権を設計初期に明示すべきだった。

✅ Design Insight: 情報劣化はエージェント数の問題ではなく、情報の「所有者」と「参照者」が設計されていないことから起きる。ファイルは「誰が書き、誰が読む」という契約書として設計する。ファイルの粒度は「1エージェントの責任範囲＝1成果物」が実用的な境界線になる。

## Round 5: 並列化できるところとできないところ — 依存グラフの可視化
**自動化率: 約95%（全フェーズ自動化。ただし並列化の誤りで品質事故が発生）**

👤 Human:
ここまでで設計の骨格はかなり固まってきた気がする。最後に並列化の話をしたい。10体が全部直列だと遅い。でもどこが並列にできるか判断が難しい。「前のフェーズの結果が必要か？」だけで判断しようとすると見落としがある。

具体的に詰まったのは：グローバルスタイル生成とKOL調査が並列にできると思ってやったら、KOL用のクリエイティブ設計がグローバルスタイルを参照しているため品質が落ちた。KOL-Bのシーン画像だけブランドカラーが誤った色（`#1A2B3C`、紺系）で生成され、正しいブランドカラー（`#F28C38`、オレンジ系）と全く別物のPPTXが出来上がった。そのまま先方に送付してしまい、送付後30分でクライアントから「カラーが全然違う」と指摘が来た。

🤖 Claude:
「情報依存グラフ」で考えると並列化の境界が自然に見えてきます。

```
依存グラフ（簡略版）:

[入力]
  ├─→ [KOL調査] ─────────────────────┐
  └─→ [商品調査] ──→ [統合プランニング] ─→ [グローバルスタイル生成]
                                              ↓
                          ┌─── [KOL-A クリエイティブ] ─→ [A画像生成]
                          ├─── [KOL-B クリエイティブ] ─→ [B画像生成]
                          └─── [KOL-C クリエイティブ] ─→ [C画像生成]
                                                          ↓
                                                    [アセット統合]
                                                          ↓
                                                  [スライド組版] → [レビュー] → [出力]
```

並列にできる箇所は2ヶ所：
1. KOL調査 と 商品調査（どちらも入力だけに依存）
2. KOL-A/B/C クリエイティブ（全員がグローバルスタイル完了後に並列スタート）

グローバルスタイル生成は「全KOLのクリエイティブより前に必ず完了している」が条件。ここを並列にしようとしたのが今回の失敗の原因。グローバルスタイルが確定するまで、KOL個別エージェントに `global_assets_manifest.json` を渡せない。ファイルが存在しないまま起動した KOL-B エージェントが、ブランドカラーをデフォルト値（黒）で補完してしまったのが事故の根本原因だった。

`run_state.json` に `run_id` を持たせ、各フェーズの出力 `*_phase_result.json` の `run_id` と照合するゲートを設けることで、ステールアーティファクトを掴む問題も解決した。

✅ Design Insight: 並列化の判断軸は「何が先に完了している必要があるか」という情報依存グラフで決まる。グローバルアンカー（全体の制約）は必ず先行完了させ、それを参照する個別ワーカーが並列に動くパターンが安全で高速。並列エージェントが同一ファイルを書かない設計（1エージェント1成果物）は並列化競合も同時に解決する。

## 実際に動いた数字

5ラウンドの設計を経てシステムが完成した。設計段階の話だけでは証拠にならない。実測値を記す。

| 指標 | 導入前 | 導入後 |
|------|--------|--------|
| KOL10名分の処理時間 | 約45分（手作業） | 約13分（全自動） |
| NGワード混入件数 | ラン毎に2〜5件 | 0件（ゲート導入後） |
| ループコスト（最悪ケース） | ¥4,200 | ¥1,700以内（停滞検知導入後） |
| ブランドカラー誤り | Round 5で1件発生 | 依存グラフ修正後ゼロ |

処理時間の短縮は並列化の恩恵が大きい。KOL-A/B/C/Dの4体が同時にクリエイティブ設計を走らせることで、直列なら20分かかるフェーズが5分で終わる。

コストについては、停滞検知を実装する前のループ暴走（¥4,200事故）と実装後の平均実行コスト（¥1,700前後）の差が、停滞検知の価値を数字で示している。「スコアが上がり続けているか」という変化パターンの監視は、コスト最適化でもある。

## 締め — まだ解いていない問い

「1エージェント1成果物」という原則が、今回のほとんどの問題を解いた。コンテキスト爆発、情報劣化、並列化競合——これらはすべて「誰が何を書き、誰が何を読むか」の設計が曖昧なことから起きていた。

だが、一つだけ設計で回避しただけで根本から解いていない問いが残っている。

**並列エージェントが同一のファイルを書こうとした場合、どうなるか？**

今回はKOL-A/B/Cがそれぞれ異なるファイルを出力することで競合を回避した。しかし、「共有リソースへの競合書き込み」——たとえば、複数のエージェントが同一のサマリーファイルに結果を追記しようとするケース——は設計で禁止しているだけで、仕組みとして防げていない。ファイルロック、書き込みキューイング、マージ戦略のいずれかが必要になるはずだ。

あなたのマルチエージェントシステムではこの問題をどう扱っているか。コメントで聞かせてほしい。
