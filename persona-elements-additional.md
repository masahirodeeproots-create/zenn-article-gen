## 記事A: Claude Codeで「AI部下10人」を作ったら、勝手にバグ直して「違反は切腹」ルールを追加してきて、オレは適当にしゃべるだけになった

この記事を面白くしている要素:

- **「神回」エピソード集という構成**: 技術解説ではなく「神回その1」「神回その2」…とエピソードを連番で並べ、テレビ番組のようなエンタメ構成で読ませる。読者は「次は何が起きるんだ」と先を読みたくなる。
- **AIが自主的にルールを追加した驚きの実録**: 「🔴自分のタスクのみ実行せよ（超重要・違反は切腹）」というルールを人間が何も指示していないのにAI将軍が自分で追加した、という「AIの自律性」の衝撃的な実例。
- **音声入力の誤認識を笑いに変える**: 「Notion」→「農村」、「さーGitHub」→「サーギットハブ」、「いいね、やっといて」→「オイネ。やっといて。」という音声認識の面白い誤変換を隠さずネタにする。技術的な不完全さをユーモアとして消費する姿勢。
- **API費用の生々しい計算と追記での修正**: 「Max ×5で大丈夫だった」→追記で「不足。Max ×20に変更」と正直に訂正。運用の現実感と正直さ。
- **ポーリング方式の計算で「API破産」を可視化**: 「9体 × 5秒ポーリング = 108回/分 → 8時間で51,840回」という具体的な算数でイベント駆動への移行の必然性を示す。
- **send-keysの2回分割という泥臭いハマりポイント**: tmuxのsend-keysでEnterを別コマンドにしないと動かない、発見に3時間かかったという、ググっても出てこない実践知。
- **Claude Code vs 素のClaude APIの比較表**: ツール有無・環境認識・永続性の3軸で差異を整理。「Claude Codeはすでにエージェント化されたOpus」という一行の結論。
- **人間の役割変化を工程別のBefore/After表で整理**: 設計・実装・テスト計画・バグ発見・修正・レビューの6工程で「従来: 人間 → claude-shogun: 将軍が○○」と対比。最後の「レビュー」だけ「人間が承認」と残る。
- **「仕事してる感ゼロ。Netflix見てる感覚」という締め**: 技術的な到達点を脱力感で表現する。読者の「そんな世界があるのか」という驚きと羨望を同時に喚起。
- **戦国口調での報告というエンタメ要素**: 「むっ...！これは問題でござる！」「全9名、出陣準備完了でござる」など、AI出力がそのまま面白いコンテンツになっている。武家メタファーが「読んで楽しい」を実現。（※既存要素35と重複するが、本記事はV1.0版でありエピソードが異なる）
- **GitHubリポジトリの公開**: 記事で語られたツールが実際にOSSとして公開されている。読者が検証・追体験できる。
- **「経営者ムーブ」という表現**: 音声入力で指示を出すだけの状態を「経営者ムーブ」と名付けることで、エンジニアが経営者の視座を体験するという新しい価値を言語化。
- **デビルズアドボケイト・テストの実施**: 意図的にシステムを壊すテストをAI自身にやらせ、脆弱性を発見させるという品質保証の実践。

## 記事B: Claude Codeの「すぐルール忘れる問題」を解決する超効果的な方法を見つけた気がする

この記事を面白くしている要素:

- **「すぐルール忘れる問題」という万人が感じる痛みをタイトルで言語化**: Claude Codeユーザーなら誰もが経験する「3ラリーくらいするとすっかり忘れてどんどん適当に動き出す」という課題をズバリ言い当てる。共感からの入り。
- **「語尾ルールは守れるのに挙動ルールは忘れる」という観察の鋭さ**: 「語尾を◯◯にして」はずっと守るのに「事前に確認を取って」は忘れるという、多くの人が感じていたが言語化できていなかった非対称性を指摘。
- **「なぜ守れるものは守れるのか」から逆算して解法を導出**: 問題の直接的な解決（「もっと強く命令する」等）ではなく、うまくいっているケースの分析から解法を導くという思考プロセス自体が記事の核心。
- **ハム太郎の例で原理を直感的に説明**: 「自分が何者かは忘れたけど語尾に～なのだをつけている」というコンテキスト維持の仕組みを、ハム太郎のロールプレイで説明する親しみやすさ。
- **再帰的ルール構造という発明**: 第5原則「AIは全てのチャットの冒頭にこの5原則を逐語的に必ず画面出力してから対応する」→5原則を出力すると第5原則も出力される→次も第5原則を守る必要がある、という自己参照的な構造。プログラミングの再帰と同じ原理。
- **失敗パターンと成功パターンの対話形式での比較**: 通常の指示が「3ラリーで忘れる」流れと、AI運用5原則が「何回やり取りしても忘れない」流れを、チャットログ風の具体例で対比。読者が違いを体感できる。
- **コピペで即使えるCLAUDE.md全文の掲載**: XMLフォーマットのCLAUDE.md設定をそのまま掲載。「上の内容をCLAUDE.mdにそのままコピペすれば動きます」という即座の実用性。
- **XML/JSON/YAML/Markdownの比較検証結果**: 「何種類か試したのですが一番AIが守りやすいのがxmlでした」という実験に基づくフォーマット選定。
- **「考えてみればすごくシンプルなのですがかなり効果的に機能します」という自己評価**: 複雑な解法ではなくシンプルな原理に基づくことを強調。読者に「なぜ今まで気づかなかったんだ」と思わせる。
- **「20チャットしても30チャットしても事前確認を行ってくれる」という定量的効果**: 抽象的な「効く」ではなく、具体的なチャット数で効果を示す。
- **「たまに忘れますが大体次のチャットで思い出してくれます」という正直な限界開示**: 100%ではないことを正直に認めつつ、実用上十分であることを伝える。
- **追記でCLAUDE.mdの設計意図を補足**: 「シンプルに見えて、かなり構造を試行錯誤した成果」であり、「一度全ての設定ファイルを消してこのCLAUDE.mdだけを設置して試してみるのをおすすめ」という実践的アドバイス。
- **コメント欄が効果を実証**: 「すさまじく安定しました！」「これは、、、神ですね」「自己再帰ルールを作ればいいと思って検索したらこの記事に辿り着きました」等、ユーザーの実体験が記事の有効性を裏付ける。
- **いいね1042・ブックマーク488という高エンゲージメント**: 記事の実用性を数値が証明している。
- **hooksへの発展的活用のコメント**: 読者が「userpromptsubmitでclaudeに自動的に表示するように変更」と応用例を共有。記事が起点となってコミュニティの知恵が広がる。
- **5原則の中身は自由に変えてよいが再帰部分は残せ、という設計のポイント明示**: カスタマイズ可能な部分と不変の核心を明確に分離して伝える。
