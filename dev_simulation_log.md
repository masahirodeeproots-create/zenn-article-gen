# Development Log: マルチエージェントPPTX生成システムの設計

## Round 1: 初期の設計仮説

👤 Human:
KOL向けの投稿指示書を自動で作りたいんですよね。8人分のKOLそれぞれにカスタマイズされたPPTXスライドを出す。で、最初に考えたのが「エージェント分ければスケールするんじゃない？」ってこと。調査する人、デザインする人、組み立てる人、チェックする人...みたいに。マイクロサービスと同じ発想で。

🤖 Claude:
まあ、マイクロサービスで言うところの「単一責任の原則」をエージェントに当てはめるイメージですよね。

ただ、1つ気になるのは、マイクロサービスだとサービス間の通信はAPIで明確に定義されますよね。エージェント間の「受け渡し」はどうします？ 人間のチームだと「なんとなく伝わる」でいけますけど、AIエージェントはコンテキストが共有されないので、渡す情報を明示的に設計しないといけない。

最初のアプローチとしては、1つの大きなエージェントに全部やらせるほうがシンプルかもしれません。コンテキストが途切れないので。ただ、それだと5人分、8人分とKOLが増えたときに品質が落ちる可能性はありますね。試してみましょうか。

✅ Design Insight: エージェント分割の動機は「責務の分離」だが、分けた瞬間に「情報の受け渡し」の設計が必要になる。分けること自体ではなく、何を渡すかの設計が本質。

## Round 2: コンテキスト汚染と情報の境界

👤 Human:
実際に1つの大きなエージェントに全部やらせてみたんですけど、やっぱりダメで。3人目くらいからクオリティが露骨に下がるんですよね。キャプションの方向性が1人目のコピーみたいになったり。コンテキストウィンドウが足りないのか、後半が前半の繰り返しになる。

で、分けるしかないなと。でも分けるとして、「誰が何を知るべきか」の線引きが全然わからない。全員にプロジェクト全体の情報を渡す？ それとも最小限？

マイクロサービスだと「境界づけられたコンテキスト」ってやつがありますよね。ドメイン駆動設計の。あれと同じ話な気がするんですけど...。

🤖 Claude:
すごくいい疑問です。DDDの境界づけられたコンテキストとまさに同じ構造ですね。

マイクロサービスで「全サービスが同じDBを見る」のがアンチパターンなのと同じで、全エージェントに全情報を渡すと問題が起きます。情報が多すぎると、エージェントは何が重要かを判断できなくなる。特にLLMベースのエージェントは、長いコンテキストの中で「関係ない情報に引っ張られる」問題がある。

逆に情報を絞りすぎると、エージェントが必要な文脈を持てなくて、的外れなアウトプットを出す。

提案としては、「全体方針」と「個別情報」を分離するのはどうでしょう：

```
全体方針（全員が知るべきこと）
  ├── ブランドのトーン
  ├── キャンペーンの方向性
  └── 品質基準

個別情報（担当者だけが知ればいいこと）
  ├── 特定のKOLのリサーチ結果
  ├── 特定のプラットフォーム要件
  └── 特定のビジュアル素材
```

ただ、正直これが正解かはわかりません。実際にやってみないと。

👤 Human:
なるほど...。でもさ、「全体方針」って誰が決めるの？ それ自体がエージェントの仕事じゃない？ 調査して、分析して、方針を決めて、それから個別に作る...って順番になるとしたら、最初のエージェントが出した「全体方針」がイマイチだったら、後工程の全員がイマイチな方向に進むよね。

🤖 Claude:
ああ、それは鋭いですね。情報の境界の話が、今度は「依存関係の順序」の話になってきました。

工場の組み立てラインと似ていて、上流の品質が下流の品質を決定する。だから「全体方針を出す」工程と「個別制作をする」工程の間に、品質チェックのゲートを置くべきかもしれません。

製造業でいう「工程間検査」ですね。前の工程の出力が基準を満たしているか確認してから、次の工程に進む。

✅ Design Insight: 情報の境界は「誰が何を知るか」だけじゃない。「どの順序で確定させるか」も含む。上流の品質が下流を決める。

## Round 3: ゲートは何を検査すべきか — オーケストレーターの責務

👤 Human:
ゲートを置くのはわかった。でも「何を検査するか」が難しい。たとえば調査エージェントが「このKOLは20代女性に人気」って出してきたとして、それが本当かどうかなんて次のエージェントにはわからないですよね。

あと、もっと現実的な問題として、調査が終わったかどうかの判定。8人分のKOLリサーチを並列で走らせたとして、全員分揃ったかどうか、欠けてるKOLがいないかのチェック。地味だけどこれがないと後工程が中途半端な入力で動き出しちゃう。

🤖 Claude:
であれば、オーケストレーターがリサーチの中身もチェックしたほうがいいかもしれません。存在チェックだけだと「ファイルはあるけど中身がスカスカ」を見逃しますよね。たとえば「フォロワー分析が500文字以上あるか」とか、「商品との相性スコアが含まれているか」とか、品質の基準をオーケストレーターが持つのはどうでしょう。

👤 Human:
うーん、それって結局オーケストレーターが全部の中身を読むってことでしょ？ KOLが20人になったら、20人分のリサーチ内容を読んで品質判断するの？ それ、マイクロマネジメントじゃない？

組織論的に言うと、マネージャーの仕事って中身の正しさを判定することじゃなくて、「必要な成果物が全部揃ったか」と「次に進めていいか」の判断なんですよね。中身の品質はそれぞれの専門家に任せるべきで。

🤖 Claude:
あ、すみません、確かにそうですね。中身の判断までオーケストレーターに持たせると破綻しますね。

整理すると：

- 「8人分のリサーチ結果が全部あるか」を確認する
- 「必須フィールドが埋まっているか」を確認する
- 「リサーチ内容が正しいか」は判定しない（それは別の専門エージェントの仕事）

CIパイプラインの考え方と似ています。CIサーバー自体がコードの品質を判定するんじゃなくて、テストが通ったかどうかのゲートだけを管理する。テスト自体の責務はテストコードにある。

```
オーケストレーターの責務:
  ✅ 成果物の存在チェック
  ✅ 前提条件の検証
  ✅ 次工程への遷移判断
  ❌ 内容の品質判断（専門エージェントの仕事）
  ❌ 工程内の具体的な手順（各エージェントの仕事）
```

👤 Human:
それでいいと思う。で、そうすると気になるのが、ゲートで「揃ってない」って判定されたときどうするか。もう一回やり直す？ 部分的に再実行？ 全部最初から？

🤖 Claude:
理想は部分的な再実行です。8人中7人のリサーチが成功して1人だけ失敗したなら、その1人だけ再実行すればいい。

ただ、これには前提があって、各エージェントの出力が独立している必要があります。1人のKOLのリサーチ結果が他のKOLに影響しないなら部分再実行できる。でも「全体の方向性」に関わる工程だったら、1つの失敗が全体に波及する可能性がある。

再実行の粒度を設計するのも、情報の境界設計の一部ですね。

✅ Design Insight: オーケストレーターの責務は「成果物の完全性チェック」と「遷移判断」に絞るべき。中身の品質判断まで持たせるとマイクロマネジメントになり破綻する。CIパイプラインと同じで、ゲートは「通ったかどうか」だけを見る。また、失敗時の再実行粒度は情報の独立性に依存する。

## Round 4: 全体トーンと個別制作の分離 — 「ベースのスープ」問題

👤 Human:
ちょっと別の角度から考えたいんですけど。ビジュアル素材の話。8人のKOLに個別の素材を作るとして、でもキャンペーン全体のトーンは統一したいわけですよ。ロゴの使い方とか、色味とか、全体の雰囲気。

で、最初はそれぞれのKOL担当エージェントに「キャンペーンの方向性」を渡して個別に作らせてたんですけど、バラバラになるんですよ。あるKOLはポップで、別のKOLはシックで...みたいな。

これ、料理で言うとベースのスープを先に作ってから個別のトッピングを変える、みたいな順番が要るんですかね。

🤖 Claude:
まさにそういう構造だと思います。

「グローバルなスタイル」を先に確定させてから、それを制約として各個別エージェントに渡す。こうすれば全体の統一感を保ちながら個別のカスタマイズができる。

```
[素材取得] → [全体スタイル確定] → [個別KOL素材生成 ×N] → [統合・整合性チェック]
```

重要なのは、全体スタイルが確定する前に個別制作を始めてはいけないということ。これは先ほどのゲートの話と直接つながります。「全体スタイルの成果物が存在する」ことが、個別制作開始の前提条件になる。

👤 Human:
あー、なるほど。でも待って。この「全体スタイル確定 → 個別制作」の依存関係、ちゃんと強制しないと事故りますよね。並列処理の誘惑に負けて「全部同時に走らせちゃえ」ってやると...。

🤖 Claude:
おっしゃる通りで、これは並行処理の古典的な問題です。「依存関係のあるタスクを並列実行して不整合が起きる」パターン。

並列にしていいのは「互いに独立したタスク」だけです。個別KOL間は独立してるから並列OK。でも「全体スタイル → 個別KOL」は直列でないといけない。

だから、ゲートの設計で重要なのは「何が並列可能で何が直列必須か」を明示的に宣言することですね。依存関係グラフとして。

✅ Design Insight: 「全体方針を先に確定してから個別制作」は情報の境界設計の帰結。全体スタイルが確定する前に個別制作を開始してはいけない。並列可能な工程と直列必須な工程を明示的に区別する必要がある。

## Round 5: 成果物のスコープ — 「これは誰のものか」問題

👤 Human:
実際に運用してて困った問題があって。案件を2回流したとき、前の案件の成果物が残ってて、次の案件のエージェントがそれを「最新の成果物」だと思って使っちゃったんですよ。たとえば前の案件で作ったカバー画像を、新しい案件でもそのまま使っちゃう。

ファイルの場所は同じ、名前も同じ。でも中身は前の案件のもの。人間だったら「あ、これ前ので古いやつだ」ってわかるけど、エージェントにはそれが判断できない。

これマジで腹が立ったんですけど、よく考えたらエージェントは悪くなくて、「どの実行に属する成果物か」の情報がどこにもないのが悪いんですよね。

🤖 Claude:
あ、これは分散システムでもよく見る問題ですね。「キャッシュの無効化」や「古いレプリカを読んでしまう」問題と本質的に同じです。

解決策としていくつか考えられます：

1. **毎回クリーンな環境で実行する**（コンテナの使い捨てと同じ発想）
2. **成果物にバージョンやスコープを紐付ける**（「この成果物はどの実行に属するか」を明示）
3. **タイムスタンプで鮮度チェック**

2番目のアプローチが一番堅実かもしれません。各成果物に「この実行のものである」という刻印を付けて、読む側はその刻印が現在の実行と一致するか確認する。

要は実行IDみたいなものを成果物に埋め込んでおいて、「今回の実行で作ったものか？」をチェックする仕組みです。古い実行のIDが付いてたら、そのファイルは前の案件の残骸だとわかる。

👤 Human:
でもこれ、すべての成果物にスコープの刻印を付けるって、けっこう大変じゃない？ 忘れたら終わりだし。

🤖 Claude:
忘れるリスクは確かにあります。だから、ゲートの検査項目に「この成果物のスコープが現在の実行と一致するか」を含めるのがいいかもしれません。成果物の存在チェックだけでなく、スコープの一致チェックも。

そうすれば、たとえスコープの刻印を忘れたとしても、ゲートが「この成果物にスコープ情報がない」と検知して止められる。

フェイルセーフですね。忘れた場合に通過させるのではなく、止める方向に倒す。

✅ Design Insight: 成果物には「どの実行に属するか」のスコープを刻印し、ゲートはスコープの一致を検証する。実行IDの照合という単純な仕組みだが、これがないと古い成果物の混入を防げない。刻印漏れはゲートがフェイルセーフで検知する。

## Round 6: 品質の閉ループ — レビューと修正のサイクル

👤 Human:
ここまでの設計で「前工程のチェック → 次工程へ」の流れはだいぶ見えてきたんですけど、最終成果物のクオリティチェックの話がまだできてない。

PPTXができあがった後のレビューって、「ここ直して」ってフィードバックを返して、修正して、またチェックして...っていうループになりますよね。人間のレビュープロセスと同じで。

でもこのループをどう閉じるかが難しい。無限ループになりかねないし、「どこを誰が直すか」のルーティングも必要。データの問題なのか、ビジュアルの問題なのかで、直すべきエージェントが違う。

🤖 Claude:
レビューループの設計は、まさにソフトウェア開発のCI/CDパイプラインと同じ課題ですね。

```
[ビルド] → [テスト] → [FAIL] → [修正] → [ビルド] → [テスト] → ...
```

ポイントは3つあると思います：

1. **ループの上限を決める** — 無限ループを防ぐ。N回やってダメなら人間にエスカレーション
2. **修正のルーティング** — レビュー結果に基づいて、どの工程に戻すかを判定。データの問題ならデータ生成エージェントへ、見た目の問題ならビジュアルエージェントへ
3. **修正後の再検証** — 修正したら必ずもう一度レビューを通す。修正が別の問題を引き起こすこともある

特に2番目が重要で、レビューエージェントは「何が問題か」だけでなく「どこに問題があるか」を出力すべきです。そうすれば、オーケストレーターが適切なエージェントにルーティングできる。

👤 Human:
で、これって前に話した「オーケストレーターの責務」と矛盾しない？ オーケストレーターは中身の判断をしないって言ったけど、ルーティングするためには「これはデータの問題か、ビジュアルの問題か」を判断しないといけないよね。

🤖 Claude:
いい指摘です。でも矛盾してないと思います。オーケストレーターはレビュー結果の「分類ラベル」を見てルーティングするだけで、自分で問題を分析してるわけではない。

郵便局の仕分け担当と同じです。中身は読まないけど、宛先を見て振り分ける。レビューエージェントが「データの問題です」「ビジュアルの問題です」というラベルを付けてくれれば、オーケストレーターはそのラベルに従ってルーティングするだけ。

あと、このビルド→テスト→修正のループは1つのチームとして囲い込んでしまう手もあります。オーケストレーターがステップごとに管理するんじゃなくて、「ビルドからレビューまで一括で任せる」ユニットを作る。内部のループはそのユニットが自律的に回す。組織で言えば、プロジェクトチームに裁量を渡す感じ。

✅ Design Insight: レビュー→修正→再検証の閉ループが必要。レビューは「何が問題か」だけでなく「どこに問題があるか」を出力し、ルーティングの判断材料を提供する。ループには上限を設け、超えたらエスカレーション。ビルド→レビューの閉ループは1つのチームとして囲い込む設計も有効。

## Round 7: スケール時の設計モード切替

👤 Human:
あのさ、実際に困ったことがあって。3人のKOLで動いてた設計をそのまま15人案件に流したら、オーケストレーターがタイムアウトしたんですよ。15人分のリサーチ結果を全部読もうとして、コンテキストに収まらなくなって。

3人だったら全部見ても大丈夫だけど、15人は無理。「KOLが3人のときと30人のときで同じ設計でいいのか」って話で、これは結構根本的な問題だなと。

🤖 Claude:
これは組織論で言う「スパン・オブ・コントロール」の問題ですね。管理する対象が増えすぎると、管理者自体がボトルネックになる。

入力の規模に応じて実行モードを切り替えるのが現実的だと思います：

```
少数モード: オーケストレーターが直接処理の一部も担当
  → コンテキストに収まるのでシンプルに

多数モード: オーケストレーターは管理に専念、処理は全部委任
  → コンテキスト溢れを防ぐため、成果物のサマリーだけ読む
```

重要なのは、どちらのモードでも成果物の形式は統一すること。モードが変わっても後工程が受け取るデータの形が変わらなければ、後工程のエージェントは何も変更しなくていい。

👤 Human:
なるほど、外から見た形は同じだけど中の動き方が違う、って感じか。

正直、こんなに設計が複雑になるとは思わなかった。最初は「エージェント分ければOK」くらいに思ってたのに。情報の境界、ゲート、スコープ管理、品質ループ、モード切替...。全部つながってるのが怖い。

🤖 Claude:
そうなんですよね。これ、1つ抜けると連鎖的に壊れるんです。ゲートなしでスコープ管理しても意味がないし、情報の境界がなければモード切替の効果も出ない。各設計原則が互いに支え合っている構造です。

✅ Design Insight: スケール時には実行モードを切り替えるが、成果物の形式は統一する。外から見た形は同じで中の動き方が違う。各設計原則は有機的に連携しているので、どれか1つ抜けると全体が壊れる。

---

## Round 8: 大きな工程の内部分割 — 同じ問題が入れ子で再発する

👤 Human:
あのさ、もう1つ困ったことがあって。素材を作る工程が重すぎるんですよ。素材集めて、全体のトーン決めて、KOL個別の素材作って、最後に全部揃ってるか確認して...。

最初これ全部1つのエージェントにやらせてたんだけど、途中で失敗するとどこで失敗したかわからない。最初からやり直すしかなくて、めちゃくちゃ非効率。しかもコンテキストが長すぎて後半の品質がまた落ちる。嘘でしょ。Round 2と同じ問題が工程の内部で再発してる。

🤖 Claude:
あー、なるほど。「分割すべき理由」が、パイプライン全体と同じなんですね。モノリスが大きくなりすぎて部分障害の切り分けができない。

大きな工程を小さな独立したステップに分割して、各ステップに明確な入出力を定義する。そうすれば：

- どこで失敗したかがわかる
- 失敗したステップだけ再実行できる
- 各ステップのコンテキストが小さくなり品質が上がる

```
大きすぎる工程
  → ステップに分割
    → 各ステップに明確な入力・出力を定義
      → ステップ間にゲートを配置
        → 各ステップは独立して再実行可能
```

👤 Human:
で、さっきの「全体スタイル → 個別制作」の話もここに効いてくるんだよね。素材系の工程だと、入力素材の取得→全体トーン確定→KOL個別→統合チェック、みたいな流れになるけど、これって全部1つの大きな工程の中のサブステップで、それぞれにゲートが要る。

大きなパイプラインの中に小さなパイプラインがある。入れ子構造。で、どっちの層にも同じ原則が必要 — 情報の境界、ゲート、スコープの刻印、全部。

🤖 Claude:
そうですね。設計の原則がスケールをまたいで同じように適用される。パイプライン全体と、各工程の内部で、同じ問題が同じ形で出てくる。だから同じ解決策が使える。

✅ Design Insight: 大きすぎる工程は独立したステップに分割し、各ステップ間にもゲートを置く。パイプライン全体の設計原則が、各工程の内部設計にも同じように適用される。情報の境界もゲートもスコープの刻印も、どの階層でも必要。

---

## Round 9: オーケストレーターは何を「しないか」の設計

👤 Human:
ここまで来て思ったんだけど、オーケストレーターにどんどん責務が増えてる気がするんですよね。ゲート管理、モード選択、成果物チェック、ルーティング...。これ以上責務を増やすと、オーケストレーター自体がモノリスになりません？

前に「中身の品質判断はしない」って決めたけど、他にも「オーケストレーターがやるべきでないこと」を明確にしたほうがいいのかな。

🤖 Claude:
これは組織論でいう「マネージャーの反パターン」と同じですね。マネージャーが部下の仕事のやり方まで口を出し始めると、マイクロマネジメントになってチーム全体の生産性が落ちる。

オーケストレーターの責務を厳密に定義するなら：

**やること:**
- 実行モードの選択（規模に応じた切替）
- 工程の順序管理（次にどの工程を実行するか）
- 成果物のゲート判定（存在・完全性・スコープ一致）

**やらないこと:**
- 各工程内部の具体的手順（各エージェントに任せる）
- 成果物の内容の品質判断（レビューエージェントの仕事）
- 情報の加工や変換（パススルーのみ）

肥大化を防ぐために、工程固有の手順は各エージェントの中に閉じ込めて、オーケストレーターからは参照しないようにする。オーケストレーターが知るのは「何が入って何が出るか」だけ。ブラックボックスとして扱う。

👤 Human:
あ、それってまさに「入出力だけ決めて、中で何やるかは知らない」ってことだよね。オーケストレーターから見たら各工程はブラックボックスで、出てきた成果物だけ見る。

正直、最初に「エージェント分ければスケールする」って言ったとき、こんな複雑な設計論になるとは想像もしてなかった。でも考えてみれば、人間のチームだって同じことを無意識にやってるんですよね。情報の境界、品質チェック、責任の委譲...。

AIエージェントの場合は、人間が暗黙にやってることを全部明示的に設計しないといけない。「あうんの呼吸」はAIには通じない。だから大変だけど、逆に言うと設計が全部見えるのが面白い。

🤖 Claude:
まさにその通りだと思います。人間のチームワークでは暗黙知として機能していることを、全てデータ構造とプロトコルとして形式化する必要がある。

結局、マルチエージェントシステムの設計って「エージェントの能力」の設計じゃなくて「エージェント間の関係性」の設計なんですよね。個々のエージェントが何をできるかより、エージェント同士がどう情報を渡し、どこでチェックし、どう修正サイクルを回すかのほうがはるかに重要。

```
┌─ 情報の境界 ──────── 誰が何を知るか
├─ 全体→個別の順序 ── グローバルを先に確定
├─ 成果物ゲート ────── 完全性チェックで遷移判断
├─ スコープの刻印 ─── どの実行に属するか
├─ 品質の閉ループ ─── レビュー→修正→再検証
├─ 責務の集中 ──────── オーケストレーターは管理のみ
└─ モード切替 ──────── 規模に応じた実行戦略
```

👤 Human:
...とは言ったものの、正直まだ確信はないんですよね。この設計で50人のKOL案件が来たら本当にスケールするのか？ ゲートが多すぎて全体の処理時間が爆発しないか？ エージェント間の受け渡しでデータが劣化していかないか？ 

やってみないとわからない部分がまだ山ほどある。

✅ Design Insight: マルチエージェント設計の本質は、エージェント間の関係性の設計にある。
